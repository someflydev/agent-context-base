# Seed Data Contract

## Source
Fixture files are generated by examples/canonical-analytics/domain/generate_fixtures.py.
That script dogfoods examples/canonical-faker/ — see AGENT.md for the dogfooding principle.

## Determinism Rules
  - All fixtures use seed=42.
  - The same seed must produce byte-identical fixture files on any machine.
  - Do not use system time, random.uuid4() without a seed, or os.urandom() in fixture generation.
  - Re-running generate_fixtures.py must be idempotent: same output, no side effects.

## Fixture Profiles
(derive row counts from examples/canonical-faker/domain/profiles.md)

| Profile  | Seed | Use                                           |
|----------|------|-----------------------------------------------|
| smoke    | 42   | CI / smoke tests — minimal row counts         |
| small    | 42   | Unit test data — representative but fast      |
| medium   | 42   | Integration tests and local development       |
| large    | 42   | Load testing and realistic scenario validation |

## Required Coverage in Smoke Profile
The smoke fixture must include at minimum:
  - at least 3 distinct services
  - at least 2 distinct environments (production + staging)
  - at least 3 incident severity levels
  - at least 4 funnel stages
  - at least 30 days of event data (to allow date-range filter testing)
  - at least 3 data points for each time-series chart (to verify line rendering)
  - at least 1 anomaly or outlier value in latency_samples (to verify histogram shape)

## Usage in Implementations
All four implementations must read from these fixture files:
  examples/canonical-analytics/domain/fixtures/smoke.json
  examples/canonical-analytics/domain/fixtures/small.json

Fixture path resolution must work from the implementation directory or accept
a FIXTURE_PATH environment variable pointing to the canonical-analytics domain directory.
