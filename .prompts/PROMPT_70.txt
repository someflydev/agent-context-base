You are working inside the agent-context-base repository.

This prompt establishes tool-invocation discipline across all derived repos. The problem it solves:
when assistants start a task in a derived repo, they often try `python`, fail, try `python3`,
fail, try to invoke `pytest` globally, fail, and repeat variations of this loop. The same pattern
recurs with templ in Go repos, with playwright installed globally instead of locally, with
`go install` causing version drift, and with other language-specific tool invocations.

The fix is not to retry smarter — it is to have deterministic, pre-established invocation paths
baked into the repo's boot context from the first session.

This prompt creates that discipline: a doctrine doc that covers every first-class stack, template
updates that surface the right invocation paths in every derived repo's CLAUDE.md and AGENT.md,
gitignore updates so the tool environments are excluded from version control, and stack doc
additions so assistants see the right tool commands on the same page as the framework guidance.

Also addressed: whether canonical Playwright examples are needed for non-TypeScript backends
(Go with templ/htmx, Python with htmx), and how Playwright installs should work per stack.

2F. The core tool invocation rules to document
  Based on what you know about each stack, write down the correct invocation pattern for each.
  Use this as the ground truth before writing any docs:

  Python (FastAPI / uv / Ruff / Polars — derived repos):
    Runtime: managed by `uv` via `pyproject.toml`. Never use system `python` or `python3`.
    Test tools setup: `uv venv --python 3.12 .venv_tools` (use the version from pyproject.toml
      requires-python, default 3.12 if unspecified)
    Install test tools: `uv pip install --python .venv_tools/bin/python pytest httpx pytest-asyncio`
    Add playwright: `uv pip install --python .venv_tools/bin/python pytest-playwright playwright`
                    then `.venv_tools/bin/playwright install chromium`
    Run tests: `.venv_tools/bin/pytest tests/unit/`
               `.venv_tools/bin/pytest tests/integration/`
               `.venv_tools/bin/pytest tests/e2e/` (if playwright)
    NEVER: `python -m pytest`, `python3 pytest`, `pip install pytest`, global pytest
    NEVER: `source .venv_tools/bin/activate` — always use the explicit path
    Gitignore: `.venv_tools/`

  TypeScript / Hono / Bun:
    Runtime: Bun. Never use npm to run a Bun project.
    Install: `bun install`
    Test tools: `bun add -d @playwright/test` (dev dependency in package.json)
    Run Playwright: `bunx playwright install chromium` (once, after bun install)
                    `bunx playwright test` or `bun run test:e2e`
    Run unit/integration tests: use Bun's built-in test runner or vitest — `bun test`
    Gitignore: `node_modules/` (bun creates this for compatibility)

  TypeScript / Node (non-Bun stacks):
    Runtime: nvm-managed Node. Confirm node is available: `which node` or `node --version`.
    Install: `npm install`
    Playwright: `npm install --save-dev @playwright/test`
                then `npx playwright install chromium` (or `./node_modules/.bin/playwright install`)
    Run tests: `./node_modules/.bin/playwright test` (prefer explicit path over global npx)
    Gitignore: `node_modules/`

  Go / Echo / templ:
    Runtime: Go toolchain. Minimum version: 1.24.
    Tool installs (project-scoped, not global):
      `go get -tool github.com/a-h/templ/cmd/templ@latest`
      The `-tool` flag records it in go.mod as a project tool dep.
    Run templ: `go tool templ generate`
    Run tests: `go test ./...`
    Playwright (Go): `go get github.com/playwright-community/playwright-go`
                     `go run github.com/playwright-community/playwright-go/cmd/playwright install --with-deps chromium`
                     Tests in `tests/e2e/` as `*_test.go` files, run with `go test ./tests/e2e/...`
    NEVER: `go install github.com/a-h/templ/cmd/templ` (global install, version drift)
    NEVER: use a Go version before 1.24 for the -tool approach

  Rust / Axum:
    Runtime: Rust toolchain (cargo is always present).
    Run tests: `cargo test`
    Run app: `cargo run`
    No special tool setup needed — cargo handles everything.

  Elixir / Phoenix:
    Runtime: Elixir + Mix (via asdf or system install).
    Install deps: `mix deps.get`
    Run tests: `mix test`
    Run app: `mix phx.server`
    No separate tool environment needed.

  Scala / Tapir / http4s / ZIO:
    Runtime: sbt.
    Run tests: `sbt test`
    Run app: `sbt run`
    No separate tool environment needed.

  Kotlin / http4k / Exposed:
    Runtime: Gradle wrapper.
    Run tests: `./gradlew test`
    Run app: `./gradlew run`
    The wrapper script (gradlew) is committed to the repo.

  Clojure / Kit / next.jdbc:
    Runtime: Clojure CLI tools.
    Run tests: `clojure -M:test` (alias defined in deps.edn)
    Run app: `clojure -M:run`
    No separate tool environment needed.

  Ruby / Hanami:
    Runtime: Ruby via bundler.
    Install: `bundle install`
    Run tests: `bundle exec rspec` (or `bundle exec rake test` depending on setup)
    NEVER: `gem install rspec` globally
    Gitignore: `.bundle/` (if bundle path is local), `vendor/bundle/`

  OCaml / Dream / Caqti / Tyxml:
    Runtime: opam environment + dune.
    Build: `dune build`
    Run tests: `dune test`
    The opam switch defines the environment — rely on it being set up.

  Crystal / Kemal / Avram:
    Runtime: Crystal + Shards.
    Install: `shards install`
    Run tests: `crystal spec`

  Dart / Dartfrog:
    Runtime: Dart SDK.
    Install: `dart pub get`
    Run tests: `dart test`

  Nim / Jester / HappyX:
    Runtime: Nim + Nimble.
    Install: `nimble install`
    Run tests: `nimble test`

  Zig / Zap / Jetzig:
    Runtime: Zig toolchain.
    Build: `zig build`
    Run tests: `zig build test`
