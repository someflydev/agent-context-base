================================================================================
BACKGROUND
================================================================================

RUBY: Ruby has a strong fixture-generation culture rooted in FactoryBot (Rails).
      This arc focuses on raw output generation, not ORM fixtures, but the README
      must honestly note the FactoryBot path for Rails teams. The primary teaching
      surface in Ruby is showing that faker's output becomes a relational dataset
      only when you build the graph layer explicitly in Ruby code.
      ffaker is the secondary library: faster (no locale overhead), simpler API,
      useful as a performance alternative or legacy context.

PHP: PHP is the most interesting language in this arc because of Alice.
     FakerPHP/Faker does atomic field generation in the standard imperative style.
     Alice is a YAML/PHP fixture definition language that lets you write relational
     graph definitions declaratively. The key teaching point is showing BOTH:
     imperative generation (FakerPHP alone) and declarative generation (Alice),
     and explaining when each approach is appropriate.

A-3. lib/distributions.rb

Ruby weighted random without gems:

  PLAN_WEIGHTS = { "free" => 50, "pro" => 35, "enterprise" => 15 }.freeze

  def weighted_choice(weights_hash, rng)
    total = weights_hash.values.sum
    target = rng.rand(total)
    cumulative = 0
    weights_hash.each do |value, weight|
      cumulative += weight
      return value if target < cumulative
    end
  end

Note: Ruby's Random class supports seeding: `rng = Random.new(seed)`.
All generation calls must use this rng instance, not Kernel#rand.

Note in the code: Ruby's faker gem uses Faker::Config.random globally,

B-4. alice/organizations.yaml

Alice fixture definition. This is the key PHP teaching surface.

Alice uses a YAML format that:
  - generates multiple records with ranges
  - assigns fake values via <method()> syntax
  - allows cross-references via @name

Example:
  AgentContextBase\Faker\Domain\Organization:
    organization_{1..3}:
      id: '<uuid()>'
      name: '<company()>'
      slug: '<slug()>'
      plan: '<randomElement(["free","pro","enterprise"])>'
      region: '<randomElement(["us-east","us-west","eu-west","ap-southeast"])>'
      created_at: '<dateTimeBetween("-3 years", "now")->format("c")>'
      owner_email: '<safeEmail()>'

alice/users.yaml:
  AgentContextBase\Faker\Domain\User:
    user_{1..10}:
      id: '<uuid()>'
      email: '<unique().safeEmail()>'
      full_name: '<name()>'
      locale: '<randomElement(["en-US","en-GB","de-DE","fr-FR"])>'
      # ...

alice/memberships.yaml:
  AgentContextBase\Faker\Domain\Membership:
    membership_{1..18}:
      id: '<uuid()>'
      org_id: '@organization_<numberBetween(1,3)>->id'
      user_id: '@user_<numberBetween(1,10)>->id'
      role: '<randomElement(["owner","admin","member","viewer"])>'
      # ...

Add alice/README.md explaining:
  - Alice uses @reference syntax for FK relationships
  - Alice handles simple ranges but NOT weighted distributions natively
  - For weighted distributions, you need custom Faker providers in PHP
  - Alice does NOT enforce joined_at >= org.created_at — temporal rules
    require either a custom provider or post-generation validation
  - Alice is excellent for: readable fixture files, team-readable graph
    definitions, simple testing scenarios
  - Alice is NOT ideal for: weighted distributions, complex temporal rules,
    large volumes (>10K rows), programmatic distribution shaping

  This is the honest contrast with imperative generation:
    - FakerPHP imperative: full control, handles all 7 doctrine rules
    - Alice declarative: readable, team-friendly, simpler but less flexible

  class SmokeTest extends TestCase
  {
      public function testGeneratesCorrectOrgCount(): void
      {
          $pipeline = new TenantCorePipeline(Profile::SMOKE);
          $dataset = $pipeline->generate();
          $this->assertCount(3, $dataset->organizations);
      }

      {
          $pipeline = new TenantCorePipeline(Profile::SMOKE);
          $dataset = $pipeline->generate();
          $report = Validator::check($dataset);
          $this->assertTrue($report->ok, implode("\n", $report->violations));

      {
          $d1 = (new TenantCorePipeline(Profile::SMOKE))->generate();

      }
  }
