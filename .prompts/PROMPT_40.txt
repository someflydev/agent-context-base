SECTION 1 — Overview

Explain briefly why the repository needs these two clusters:

Parquet + MinIO:
- The existing `duckdb-trino-polars` stack implicitly assumes analytical reads from

SECTION 3 — Stack pack content requirements

--- parquet.md ---

Frame Parquet as a file-format discipline document, not a database.

Cover:
- When to choose Parquet over JSONL, CSV, or Avro:
  - Parquet when the reader is columnar (DuckDB, Polars, Spark, Trino)
  - JSONL when the consumer is a log processor or document store
  - Avro when schema evolution with a schema registry is required
- Schema definition and evolution:
  - pin the Arrow/PyArrow schema explicitly; do not infer from the first batch
  - add fields with nullable defaults; never remove or reorder existing fields
  - increment a `schema_version` column when the shape changes materially
- Partitioning strategy:
  - partition by the most common filter dimension first (e.g. tenant_id, then date)
  - avoid high-cardinality partition keys (e.g. do not partition by UUID or timestamp directly)
  - name partition directories using the Hive convention: `tenant_id=acme/date=2026-03-10/`
- Compression codec selection:
  - prefer Snappy for latency-sensitive reads (fast decode, moderate ratio)
  - prefer Zstandard for cold storage (better ratio, acceptable decode speed)
  - avoid uncompressed Parquet for anything that will be stored longer than a day
- Row group sizing:
  - target 128 MB to 512 MB per row group for large analytical files
  - smaller row groups for streaming or incremental writes
- Common mistakes:
  - writing one file per row or one file per minute (creates a small-files problem)
  - re-inferring schema on every write (causes silent type coercion)
  - mixing compressed and uncompressed parts in the same dataset

--- minio.md ---

Frame MinIO as an S3-compatible object store for local dev, CI, and self-hosted workloads.
Do not present it as a managed AWS S3 replacement for production; that is a deployment
decision outside this repository's scope.

Cover:
- When MinIO is the right choice:
  - local dev and CI where S3 credentials are unavailable or undesirable
  - self-hosted environments where data must not leave a network boundary
  - integration tests that need a real object store without mocking
- Bucket naming discipline:
  - use a consistent prefix per environment: `dev-`, `test-`, `prod-`
  - keep bucket names lowercase, hyphen-separated, no underscores
  - one bucket per major data domain, not one bucket per tenant
- Object key (path) conventions:
  - use a deterministic, hierarchical key schema:
    `<domain>/<partition-key>/<date>/<uuid-or-sequence>.parquet`
  - avoid keys that sort into a hot partition (e.g. timestamp prefix without sharding)
- Credentials and access:
  - use environment variables: `MINIO_ACCESS_KEY`, `MINIO_SECRET_KEY`, `MINIO_ENDPOINT`
  - never hardcode credentials in application code
  - for integration tests, use a fixed test credential pair defined in docker-compose.test.yml
- Docker dev setup:
  - the standard MinIO image is `minio/minio`
  - the console port is 9001; the API port is 9000
  - for CI use: `command: server /data --console-address ":9001"`
  - bucket creation on startup: use the `minio/mc` (MinIO Client) image or a startup script
- Presigned URLs:
  - generate presigned GET URLs for short-lived file access
  - do not expose bucket credentials to frontend clients
- Common mistakes:
  - treating MinIO as an append-log (it is an object store; keys overwrite silently)
  - not creating buckets before writing (the SDK will fail with a bucket-not-found error)
  - using path-style vs virtual-hosted-style incorrectly; default to path-style for MinIO

--- parquet-minio.md ---

Frame this as the canonical data-lake write path for repos that use DuckDB, Polars, or
Trino as the read layer.

Cover:
- When to load this combo pack:
  - the repo writes Parquet files and stores them in an S3-compatible bucket
  - the read layer is DuckDB, Polars, or Trino (load the relevant read-layer pack alongside)
  - the design question is how to partition, compress, and name files so the reader performs well
- Write path discipline:
  - buffer writes into row groups of target size before flushing to MinIO
  - use a staging prefix (`_staging/`) during multi-part writes; rename to final path on commit
  - write a `_SUCCESS` marker or equivalent after completing a partition write if downstream
    consumers poll for completeness
- Read path expectations:
  - DuckDB reads MinIO Parquet via the S3 extension with httpfs: `SET s3_endpoint='...'`
  - Polars reads via `pl.scan_parquet("s3://bucket/prefix/**/*.parquet")` with S3 credentials
  - Trino reads via the Hive or Iceberg connector pointed at the MinIO catalog
- Retention and deletion:
  - delete old partition prefixes rather than individual files
  - use a lifecycle policy on the bucket if the deployment supports it
- Interaction with existing stacks:
  - `parquet-minio` is the write side; `duckdb-trino-polars` is the read side
  - when both are present, load both packs
  - `parquet-minio` does not replace `duckdb-trino-polars`; it extends it

--- nats-jetstream.md ---

Frame NATS JetStream as a durable, subject-routed message streaming layer for
lightweight event pipelines. Distinguish it clearly from Kafka.

Cover:
- When NATS JetStream is the right choice:
  - lightweight durable message bus without the operational overhead of Kafka
  - subject-based routing where consumers filter by subject pattern
  - pipelines where at-least-once delivery with consumer ack is sufficient
  - small-to-medium message volumes where horizontal Kafka partitioning is unnecessary
- When NATS JetStream is NOT the right choice:
  - very high throughput requiring horizontal partition scaling (use Kafka)
  - when a schema registry is needed for consumer compatibility (use Kafka + Avro/Protobuf)
  - when ordering guarantees must span multiple subjects simultaneously
- Core concepts to cover:
  - Streams: durable storage for messages matching a subject pattern
  - Subjects: dot-separated routing keys, e.g. `reports.requests.acme.daily-signups`
  - Consumers: durable pull consumers with ack policies (explicit ack, ack wait, max deliver)
  - Subject naming convention: `<domain>.<event_type>.<tenant_id>.<entity_id>`
    - use dots as separators, not slashes or underscores
    - keep the prefix stable; the terminal segments can be wildcarded with `*` or `>`
  - Publish: fire and forget with optional reply subject
  - Pull consumer: fetch N messages, process, ack or nack individually
  - Dead letter: nacked messages that exceed max_deliver go to an advisory subject;
    route those to a DLQ stream for inspection and replay
- Message payload shape:
  - always include a `payload_version` field so consumers can reject or adapt to schema changes
  - always include `published_at` as an ISO 8601 UTC timestamp set by the producer
  - cap message size; do not put raw HTTP bodies or unbounded arrays in the payload
  - include `tenant_id` and a correlation identifier so the consumer can deduplicate
- Testing expectations:
  - use the official `nats-server` Docker image for integration tests
  - enable JetStream in the test server: `command: "-js"`
  - use a dedicated test stream with a short retention (e.g. 1 hour or 1000 messages)
  - always purge or delete the test stream in teardown
- Common mistakes:
  - treating NATS subjects like REST routes (they are routing keys, not URLs)
  - not setting `max_deliver` on the consumer (messages can loop forever on failure)
  - storing raw large payloads in NATS instead of storing a reference key and retrieving
    the payload from object storage

--- nats-jetstream-mongo.md ---

Frame this as the canonical capture-enrich-persist pipeline:

  Producer → NATS JetStream → Consumer/Enricher → MongoDB

This pack is for repos where:
- one process (the producer) captures request/response events and publishes them as
  bounded, versioned messages to a NATS JetStream stream
- a separate process (the consumer/enricher) reads from NATS, validates, cleans, and
  enhances each message, then inserts the enriched document into MongoDB
- MongoDB holds the enriched documents for reporting queries

Cover:

Producer responsibilities:
- construct a bounded payload from the raw request/response (never store raw bodies)
- set `payload_version`, `tenant_id`, `correlation_id`, `published_at`
- publish to subject: `reports.requests.<tenant_id>.<report_id>`
- do not wait for consumer confirmation; the stream provides durability

Consumer/enricher responsibilities:
- read from a durable pull consumer on the stream
- validate: reject messages where `payload_version` is unrecognised
- clean: strip any residual PII fields that should not be in the reporting layer
- enhance: compute derived fields that the producer cannot know:
  - `response_time_class`: `fast` (< 200ms), `slow` (200–2000ms), `timeout` (> 2000ms)
  - `error_category`: `none`, `client_error` (4xx), `server_error` (5xx)
  - `enriched_at`: ISO 8601 UTC timestamp of enrichment
- insert the enriched document into the target MongoDB collection using the weekly
  bucketing pattern from `context/stacks/mongo.md`
- ack the NATS message only after successful MongoDB insertion
- on MongoDB insertion failure: nack with no immediate redelivery; let `max_deliver`
  handle retries before the message routes to the DLQ stream
- on DLQ: log the failed message with the error reason for manual inspection

MongoDB target document shape:
- collection naming: weekly-bucketed as per `mongo.md` (e.g. `request_logs_2026_w12`)
- document must include all producer fields plus all enrichment fields
- `payload_version` must be preserved in the stored document
- do not store the raw NATS message metadata; extract only the payload fields

Stream and consumer configuration:
- stream name: `REPORT_REQUESTS`
- subject filter: `reports.requests.>`
- retention: `WorkQueuePolicy` (messages deleted after ack) or `LimitsPolicy` with a
  retention window appropriate to the expected consumer lag
- consumer: durable pull consumer named `report-enricher`
- ack policy: `AckExplicit`
- ack wait: 30 seconds
- max deliver: 5 (after 5 failures the message routes to the advisory DLQ)
- DLQ stream: `REPORT_REQUESTS_DLQ` on subject `$JS.EVENT.ADVISORY.CONSUMER.MAX_DELIVERIES.>`

Testing expectations:
- run `nats-server -js` and `mongo:7` in `docker-compose.test.yml` for integration tests
- use a short-lived test stream (purge in teardown)
- verify the full pipeline: publish one message → consumer processes it → assert MongoDB
  document exists with expected enrichment fields
- verify the DLQ path: publish a message that will fail enrichment → assert it lands in
  the DLQ stream after max_deliver attempts

Common mistakes:
- acking before the MongoDB write confirms (creates data loss on crash)
- putting the producer and consumer in the same function (breaks independent scaling)
- not capping message payload size (NATS has a default max payload of 1 MB)
- using publish-subscribe (core NATS) instead of JetStream (no durability guarantee)
