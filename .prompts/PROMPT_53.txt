BROKER EXAMPLE: Go + Elixir via NATS JetStream

Directory: examples/canonical-multi-backend/duos/go-elixir-nats/

go-side.go
  - Uses nats.go (github.com/nats-io/nats.go)
  - Connects to NATS from NATS_URL env var
  - Ensures a stream exists: name "DOMAIN_EVENTS", subject "events.>"
  - Publishes a single typed event: {"type":"user.created","user_id":"u123","ts":"..."}
  - Includes a health check HTTP handler at /healthz that returns 200 if NATS is connected
  - Uses a struct for the event payload; marshals to JSON before publish
  - Handles publish errors and logs them

elixir-side.ex
  - Uses Gnat (hex package gnat)
  - Connects to NATS from NATS_URL env var
  - Subscribes to "events.>" subject with a durable consumer
  - Decodes the JSON payload, logs the event type and user_id
  - Implements a GenServer that holds the NATS connection and subscription
  - Handles reconnection on connection loss
  - Phoenix router exposing /healthz returning JSON {status: "ok"}

docker-compose.yml
  - nats service: nats:2.10-alpine with -js flag and /healthz check
  - go-service: built from a Dockerfile.go or using the go image
  - elixir-service: built from a Dockerfile.ex or using the elixir image
  - go-service depends_on: nats (service_healthy)
  - elixir-service depends_on: nats (service_healthy)
  - Environment: NATS_URL=nats://nats:4222 in both services
  - Ports: nats:4222, go-service:8080, elixir-service:4000

seam.md
  Purpose, seam type, why Go publishes and Elixir consumes (not reversed),
  the event schema, how to run locally, what to observe, related docs.


REST EXAMPLE: Go + Python via HTTP/JSON

Directory: examples/canonical-multi-backend/duos/go-python-rest/

python-side.py
  - FastAPI application
  - POST /score endpoint accepting {"features": [float, ...], "model": str}
  - Returns {"score": float, "label": str, "latency_ms": float}
  - Uses a stub model (deterministic function, not real ML) for the example
  - GET /healthz returning {"status": "ok"}
  - Uses uvicorn with PORT env var (default 8001)
  - Includes request validation (Pydantic model)

go-side.go
  - Uses net/http from stdlib
  - Calls the Python scoring service at PYTHON_SCORE_URL env var + "/score"
  - Constructs the JSON request body using a Go struct + json.Marshal
  - Reads the response, decodes it into a Go struct
  - Handles HTTP errors and service unavailability gracefully
  - Exposes its own /healthz at :8080 that calls the Python /healthz as a
    downstream health probe (returns degraded if Python is unreachable)
  - Has a main() that boots a minimal HTTP server with one route at /run-inference

docker-compose.yml
  - python-service: python:3.12-slim with uvicorn command
  - go-service: golang:1.23-alpine or CGO_ENABLED=0 binary
  - go-service depends_on: python-service (service_healthy)
  - Environment: PYTHON_SCORE_URL=http://python-service:8001 in go-service
  - Ports: python-service:8001, go-service:8080

seam.md
  Purpose, why Go is the gateway and Python owns scoring, the JSON schema,
  the downstream health probe pattern, how to run locally, related docs.


gRPC EXAMPLE: Kotlin + Rust via Protocol Buffers

Directory: examples/canonical-multi-backend/duos/kotlin-rust-grpc/

service.proto
  - syntax = "proto3"
  - package compute.v1
  - Message ComputeRequest: int32 n, repeated double values
  - Message ComputeResponse: double result, string method, int64 duration_ns
  - Service ComputeService: rpc Compute(ComputeRequest) returns (ComputeResponse)
  - Service health: rpc Check(HealthCheckRequest) returns (HealthCheckResponse)
    using google.health.v1 standard, OR a simple custom health message

rust-server.rs
  - Uses tonic (tonic = "0.12" or current) and prost
  - Implements ComputeServiceServer trait
  - Compute RPC: takes request.values, computes the mean, returns result
  - Handles empty input gracefully (returns 0.0 with a note)
  - Boots on GRPC_PORT env var (default 50051)
  - Logs each request with duration_ns in the response

kotlin-caller.kt
  - Uses grpc-kotlin (io.grpc:grpc-kotlin-stub) and protobuf-kotlin
  - Reads RUST_GRPC_URL env var (default localhost:50051)
  - Builds a ManagedChannel to the Rust server
  - Creates a ComputeServiceCoroutineStub
  - Sends a ComputeRequest with 5 sample values
  - Prints the response: result, method, duration_ns
  - Exposes a /healthz HTTP endpoint (using Ktor) that calls
    the Rust gRPC health check and returns 200 or 503

docker-compose.yml
  - rust-service: built from rust:1.82-slim, exposes port 50051
  - kotlin-service: built from openjdk:21-slim, exposes port 8080
  - kotlin-service depends_on: rust-service (service_healthy via grpc_health_probe
    or a simple TCP check at port 50051)
  - Comment noting grpc_health_probe installation pattern for health checks

seam.md
  Purpose, why proto is the canonical contract, how to regenerate stubs
  (protoc commands for each language), how to run locally, related docs.

DUO 1 — Rust + Python (gRPC seam)
  Rust owns: compute-intensive processing, hot-path ML inference serving,
             codec/crypto operations, systems-level tasks
  Python owns: model training, ML experimentation, data science, batch pipelines,
               orchestration and scripting
  Primary use case: AI platform where Python trains and Rust serves at inference time.
  gRPC proto snippet: InferenceService.Predict(PredictRequest) → PredictResponse
  Note: tonic (Rust) + grpcio or grpc-python (Python client)

DUO 2 — Go + Elixir (broker seam via NATS)
  Go owns: external-facing HTTP/gRPC API, data ingestion, fast I/O, ops tooling
  Elixir owns: distributed state, real-time pub/sub, WebSocket fan-out,
               fault-tolerant supervision, presence tracking
  Primary use case: high-throughput data ingestion (Go) driving real-time
                    dashboards and client notifications (Elixir/Phoenix).
  Note: Go publishes, Elixir subscribes and fans out via Phoenix Channels.

DUO 3 — Go + Python (REST seam)
  Go owns: API gateway, routing, auth, rate limiting, service mesh
  Python owns: ML inference, data science, analytics, complex business logic
               that benefits from numpy/pandas/scikit ecosystem
  Primary use case: production ML system where Go is the request-handling layer
                    and Python owns the scoring/recommendation engine.
  Note: the most operationally common duo; REST is sufficient and avoids proto ceremony.

DUO 4 — Elixir + Rust (NIF/Port seam)
  Elixir owns: process supervision, distributed coordination, API surface,
               fault tolerance, retry and recovery logic
  Rust owns: CPU-bound computation — audio/video transcoding, crypto, parsing,
             compression — called synchronously from Elixir as a library
  Primary use case: media processing platform where Elixir orchestrates jobs
                    and Rust does the heavy lifting inline.
  Note: this is NOT a two-service system. Rust is a compiled library loaded into
        the BEAM. Use Rustler for safe NIFs; use Port for potentially panicking code.
        The docker-compose snippet shows only one service (Elixir+Rust compiled together).

DUO 5 — Clojure + Go (broker seam via Kafka)
  Clojure owns: event processing, business rule engines, Kafka Streams topology,
                data transformation, rich domain logic
  Go owns: high-throughput external API, lightweight microservices,
            fast consumer workers, operational tooling
  Primary use case: event-driven fintech/insurtech platform where Clojure
                    owns the business logic and Go owns the execution tier.
  Note: Kafka (not NATS) here because Clojure's Kafka Streams and confluent ecosystem
        are a natural fit; Go consumes with franz-go or confluent-kafka-go.
        Include Kafka in docker-compose (bitnami/kafka or confluentinc/cp-kafka).

DUO 6 — Node + Go (REST seam)
  Node/TypeScript owns: BFF layer, GraphQL schema, WebSocket connections,
                        SSE, client-specific data shaping, OAuth flows
  Go owns: internal microservices, domain APIs, fast background workers,
            storage-backed CRUD, auth verification
  Primary use case: consumer product where Node owns the frontend contract
                    and Go owns the backend domain services.
  Note: Node calls Go over HTTP/REST. Go services may not need to call Node at all.

DUO 7 — Elixir + Clojure (broker seam via NATS or Kafka)
  Elixir owns: real-time coordination, distributed state, fault-tolerant supervision,
               presence, pub/sub to clients, OTP process trees
  Clojure owns: complex data transformation, business rule engines,
                stream processing (Kafka Streams DSL), rich map-based domain logic
  Primary use case: event-driven platform where Clojure processes and enriches
                    events, then Elixir delivers them fault-tolerantly in real time.
  Note: both languages are immutable-first, functional, and LISP-family (Clojure) or
        actor-based functional (Elixir). Data shapes translate cleanly.
        Prefer NATS for simplicity; Kafka if Clojure is already using Kafka Streams.

DUO 8 — Kotlin + Rust (gRPC seam)
  Kotlin owns: JVM ecosystem, Android platform backend, Spring/Ktor services,
               enterprise auth, billing, user/account management
  Rust owns: performance-critical compute — media encoding, search indexing,
             cryptographic operations, native library integration
  Primary use case: mobile platform backend where Kotlin handles user concerns
                    and Rust handles media or search at scale.
  Note: grpc-kotlin (Kotlin) + tonic (Rust). Consider whether the Rust service
        could be replaced by a Rust NIF in a future consolidation.

DUO 9 — Scala + Rust (gRPC seam)
  Scala owns: distributed streaming (Akka Streams / Spark), actor model,
              batch analytics, stream topology orchestration
  Rust owns: low-latency compute kernels — transforms too hot for JVM GC pauses,
             SIMD-accelerated data processing, memory-safe systems work
  Primary use case: financial data platform or observability pipeline where Scala
                    orchestrates streams and Rust handles the critical compute path.
  Note: Akka Streams hands off to Rust via gRPC when a stage exceeds JVM latency
        budget. Scala uses scalapb for proto; Rust uses tonic.

DUO 10 — Node + Elixir (broker seam via NATS)
  Node/TypeScript owns: client connection management, WebSocket/SSE sessions,
                         custom protocol handling, client-specific auth tokens,
                         BFF GraphQL subscriptions
  Elixir owns: distributed state, multi-node presence, fan-out coordination,
               channel lifecycle, crash recovery, durable subscription state
  Primary use case: collaborative apps, multiplayer games, live dashboards —
                    Node owns the client connection, Elixir owns the distributed truth.
  Note: Node connects clients; Elixir maintains the authoritative shared state.
        NATS is the seam: Node publishes client actions, Elixir processes and
        rebroadcasts state updates back through NATS to Node.
