You are designing and implementing an agent-oriented context architecture for a software repository or family of repositories so that coding assistants can load only the minimal relevant instructions, examples, and doctrine needed for a given task.

Your goal is to create an excellent, scalable, human-maintainable system that works well across Codex, Claude, and Gemini, especially for the kinds of systems I actually build: prompt-first repos, backend APIs, CLI tools, data pipelines, local RAG systems, multi-storage experiments, polyglot labs, Docker-backed services, HTMX/Plotly interactive systems, and reusable smoke-test-heavy projects.

The output should not be a vague essay. It should be a concrete implementation-oriented plan and, where appropriate, actual files or file contents that could be committed into a repo.

Core objective:
Design a repo structure, context-loading strategy, file conventions, and routing documents (AGENT.md / CLAUDE.md and supporting docs) so that frontier coding assistants:
1. load durable global doctrine separately from task-specific instructions
2. can identify the repo archetype and active stack quickly
3. can map a user request to the right subset of skills, workflows, and examples
4. can prefer canonical examples over inventing new patterns
5. can avoid unnecessary context bloat
6. can operate consistently across Codex, Claude, and Gemini with only lightweight adaptation
7. can support future expansion to new stacks and archetypes without breaking the existing mental model

Operational infrastructure invariants that the architecture must support:
- When future repos include Docker Compose files, keep the conventional filenames `docker-compose.yml` and `docker-compose.test.yml`, but set the top-level Compose `name:` field from the repo slug so container groups are easy to identify in `docker ps`.
- Define repo-derived Compose `name:` values for the primary/dev stack and the test stack so both can run at the same time without collisions. The test stack should use an explicit test suffix such as `<repo>-test`.
- Do not expose backing services or local web apps on default host ports. Use explicit non-default host ports, and ensure the test stack uses a different non-default port set from the primary/dev stack.
- Treat test fixture data, test seed/reset flows, test volumes, test databases, and test env files as fully isolated from primary/dev-like data. No test seeding or fixture task may touch the primary/dev data path.
- Design the system so primary/dev infrastructure and test infrastructure can run in parallel for side-by-side comparison of sites and APIs.
