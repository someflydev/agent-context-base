================================================================================
BACKGROUND: WHY PYTHON AND TYPESCRIPT TOGETHER
================================================================================

Python and TypeScript are chosen first because they are the clearest exemplars
of the hybrid (Lane C) approach — and because their library ecosystems require
the most careful comparison work.

For Python:
  Pydantic is not just a runtime validator. The model class IS the type, IS the
  validator, IS the JSON Schema source. When you define a BaseModel subclass,
  you get all three for free. marshmallow does something fundamentally different:
  it asks you to define a Schema object (separate from your data class) and then
  use that schema to serialize/deserialize/validate. Both are valid; they have
  different ergonomic tradeoffs that the examples must make concrete.

For TypeScript:
  Zod and io-ts are both TS libraries that give you both static type inference
  and runtime validation — but they do so with very different designs:
    - Zod: you define a schema, infer a type from it. Ergonomic, mainstream.
    - io-ts: you define a Codec, which is a value that encodes BOTH a runtime
      decoder AND a static type. FP-oriented, more compositional, more explicit.
  These are not interchangeable. Do not flatten them.
  Valibot is the lightweight alternative to Zod: same general hybrid approach,
  modular imports for tree-shaking, simpler API surface.
  TypeBox + Ajv represents a different lane entirely: TypeBox schemas ARE JSON
  Schema natively, so the "contract" and the "validator" are the same object.
  This is the contract-first lane in TypeScript.

--- README.md ---

  # TypeScript — io-ts (Lane A: Codec-Based FP Approach)

  ## Lane
  Lane A: runtime decoding with explicit error type. NOT a simpler Zod.

  ## The Codec Model
  4–6 sentences. Codec = runtime value encoding type + decode + encode.
  decode() returns Either<Errors, T>. The Either type forces callers to handle
  errors explicitly — no uncaught ValidationError. This is the FP-style
  boundary: instead of try/catch, you pattern-match on Left/Right.

  ## When to Choose io-ts
  3–4 sentences: FP codebases using fp-ts; when you need composable codec algebra;
  when explicit error typing (not throw) is a design requirement.

  ## What io-ts is NOT
  It is not a lighter Zod. It is a different design philosophy. Do not pick it
  for ergonomic schema definition — pick it for compositional codec construction.

  # TypeScript — TypeBox + Ajv (Lane B: Contract-First, JSON Schema Native)

  ## Lane
  Lane B: TypeBox schemas ARE JSON Schema natively. No transformation needed.
  This is the contract-first lane: define once in TypeBox, validate with Ajv,
  export as JSON Schema for cross-language interoperability.

  ## Key Distinction from Zod
  Zod derives JSON Schema via a transformation. TypeBox schemas ARE JSON Schema —
  the Static<> type is extracted from the schema at compile time, not the other
  way around. Different mental model: schema-first vs type-first.

  ## Limitations
