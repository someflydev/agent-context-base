BACKGROUND: THE SCHEMA VALIDATION AND CONTRACT GENERATION ARC
================================================================================

agent-context-base has canonical examples for API services, CLI tools, TUI
applications, data pipelines, and observability patterns. It does not yet have a
canonical area focused on a concern that cuts across all those surfaces: runtime
validation, contract generation, and type-driven hybrid validation.

These concerns matter because:
  - Every service boundary needs to validate incoming data at runtime.
  - Cross-language teams need interoperability artifacts (JSON Schema, OpenAPI specs).
  - Library choices in this space vary enormously in philosophy and ergonomics.
  - The right tool depends on whether you need runtime checking, schema export,
    static-type integration, or all three.

The arc adds a canonical-schema-validation area with examples in 7 languages
across 3 distinct conceptual lanes:

  Lane A — Runtime validation
    Validating objects/values at runtime: boundary checks, nested structure,
    cross-field rules, collection constraints, optional vs nullable handling,
    coercion and parsing behavior.

  Lane B — Contract generation / interoperability
    Producing machine-readable artifacts: JSON Schema, OpenAPI snippets. Used
    for cross-language interoperability, contract-first workflows, and drift
    detection between runtime behavior and exported contracts.

  Lane C — Hybrid type-driven flows
    Libraries that unify static typing + runtime validation + schema behavior.
    The static type definition IS the validation schema IS the contract source.
    Strong in TypeScript (Zod, io-ts) and Python (Pydantic). Weaker in Go.

These lanes are NOT the same thing. An assistant should be able to answer:
  "Is Pydantic a runtime validator or a contract generator?"
  "Why is serde + schemars a contract lane, not a validator?"
  "When would I use marshmallow instead of Pydantic?"
  "What is io-ts doing that Zod does not?"

The doctrine, archetype, stacks, and skills you create here encode these
distinctions clearly for future assistant sessions.

Required sections:

  # Schema Validation and Contract Generation

  ## Purpose
  (2–3 sentences) Why this doctrine exists. What problem it solves. What
  happens when these distinctions are not maintained.

  ## The Three Lanes

  Define each lane with its name and 2–4 sentences of explanation:

  ### Lane A: Runtime Validation
  Concerned with checking values against constraints at execution time.
  Operates on live data. Errors are surfaced as validation exceptions or
  result types. Does not inherently produce a machine-readable schema.

  ### Lane B: Contract Generation and Interoperability
  Concerned with producing a machine-readable description of expected structure.
  Outputs JSON Schema, OpenAPI fragments, or equivalent artifacts. May or may
  not be derived from the same code that performs runtime checking. Critical for
  cross-language workflows and drift detection.

  ### C: Hybrid Type-Driven Flows
  Concerned with unifying the model definition, runtime validation, and
  schema generation in a single construct. The type IS the validator IS the
  contract source. Strongest in TypeScript and Python ecosystems. Go and Elixir
  favor more separated approaches.

  ## Rules

  Number each rule. Include at minimum:

  Rule 1 — Do not conflate lanes.
    Explain why a contract-generation library (e.g., schemars) is not the same
    as a validation library (e.g., garde), even when they are used together in
    the same codebase. Each has a distinct job.

  Rule 2 — Treat serde + schemars as a contract lane, not just a serialization add-on.
    In Rust, serde defines the data's serialization shape. schemars derives JSON
    Schema aligned with serde's JSON representation. This pairing is Rust's primary
    path for contract generation and interoperability — treat it with the same
    weight as Pydantic's schema_model() or Zod's z.toJsonSchema().

  Rule 3 — Explicitly compare Konform and Hibernate Validator in Kotlin contexts.
    Do not reduce Kotlin validation to "JVM annotation-based." Konform is a
    Kotlin-first DSL that represents a distinct design philosophy. Contrasting
    the two is part of a complete Kotlin validation picture.

  Rule 4 — Distinguish marshmallow from Pydantic in Python contexts.
    Pydantic is a hybrid type-driven library: types are the model, validator,
    and schema source. marshmallow is an explicit schema/serialization library:
    you define schema objects separately from your data classes. Both are valid;
    neither is simply "a Python validator."

  Rule 5 — Preserve io-ts in TypeScript comparisons.
    io-ts takes a codec-based approach: types are defined as runtime values
    that encode both static type inference and decode/encode behavior. It is
    not a lightweight version of Zod — it is a different design point. Omit
    it from "advanced TS validation" only when explicitly out of scope.

  Rule 6 — Maintain a shared fixture corpus across language examples.
    Canonical schema-validation examples must use the same logical domain and
    the same set of valid/invalid/edge fixture values. This enables direct
    cross-language comparison and drift detection.

  Rule 7 — Distinguish drift detection from validation.
    Validating a payload at runtime and detecting drift between runtime behavior
    and an exported contract are different operations. A codebase can pass all
    validation tests while its exported JSON Schema drifts from its actual
    behavior. Drift detection is a separate verification concern.

  Rule 8 — Smoke tests must cover at least one valid and one invalid fixture
    per example. Asserting on a valid-only path does not verify the validation
    logic; it only verifies that the code runs.

  ## What This Doctrine Does NOT Cover
  (3–5 bullet points of explicit exclusions)
    - Input sanitization and XSS/injection prevention (security doctrine)
    - API authentication and authorization checks (security doctrine)
    - Database-level constraints and migrations (storage doctrine)
    - ORM-level validation (storage doctrine)
    - Exhaustive library API documentation (belongs in example comments)

Target length: 80–110 lines. Voice: direct, rule-numbered, lowercase headings
below the title. Do not use filler phrases like "it is important to note."

         If in scope: load examples/canonical-schema-validation/PARITY.md for
         cross-language parity rules.

    4. For drift detection, recommend a round-trip check:
         - Serialize a known-valid object
         - Validate the serialized form against the exported schema
         - Assert both the runtime validator and the schema validator accept it
         - Do the same with a known-invalid object: both should reject it

    5. Do not present a schema export as drift-free without running step 4.

  - "generate OpenAPI from my types"
  - "does my schema match my runtime validation"
  - "drift between schema and validator"
  - "contract-first workflow in Go"
  - "schemars for OpenAPI in Rust"
