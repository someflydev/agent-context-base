This prompt adds two distinct things:

   A concept document explaining when the same functionality should be exposed

   This verifies that the domain behavior is consistent across languages —

PART 1: TERMINAL ↔ WEB PARITY CONCEPT DOC
================================================================================

Create: docs/terminal-web-parity.md

Content structure:

  # Terminal ↔ Web Parity

  ## Purpose
  Helps future assistants and operators decide when a capability should be
  exposed as a terminal tool, a web UI, or both.

  ## When Terminal is the Right Choice

  ### Automation and scripting pipelines
  Terminal tools compose naturally with unix pipes, cron, CI, and shell scripts.
  Web UIs do not. If the primary consumer is another program or script, prefer terminal.

  ### Operator-local inspection
  When an operator needs to inspect local state (files, processes, queues) without
  deploying a web service, terminal is lower friction. No port, no browser, no auth.

  ### Latency-sensitive feedback loops
  Terminal tools start faster and require no network round trip for display.
  For fast interactive inspection, terminal is better.

  ### Secure/isolated environments
  In air-gapped or locked-down environments, a terminal tool works where a
  browser-accessible web UI cannot.

  ## When Web is the Right Choice

  ### Multi-user collaboration
  Web UIs support concurrent users, sessions, and access control naturally.
  Terminals are inherently single-user.

  ### Rich data visualization
  Charts, graphs, and large tabular data with sorting/filtering UI are better
  served by a browser with full CSS/JS capabilities.

  ### Public-facing interfaces
  Public access without SSH or tool installation favors web.

  ### Mobile/non-terminal clients
  Web works on mobile; terminal tools do not.

  ## When Both Are Right (Parity Pattern)

  Build both when:
  - The domain has both operator-local and collaborative consumption needs
  - Automation (terminal) and exploration (web) are both first-class use cases
  - The team has strong competence in both terminal and web tooling

  Implementation pattern:
    - shared domain core (no I/O, no rendering — pure logic)
    - terminal surface: CLI + TUI (this arc)
    - web surface: REST API + frontend (canonical-api examples)
    Both surfaces import from the shared core. Neither reimplements domain logic.

  ## Parity Example: TaskFlow Monitor

  The TaskFlow Monitor domain (job queue inspector) is a good parity candidate:

  | Aspect | Terminal | Web |
  |--------|----------|-----|
  | Primary users | On-call engineers, CI automation | Managers, distributed teams |
  | Deployment | CLI binary, no server | Deployed web service |
  | Data source | Fixture files or local API | Remote API, database |
  | Auth | OS-level (SSH, local user) | Web auth (cookie, JWT) |
  | Real-time | Polling refresh or event replay | WebSockets or SSE |

  The terminal examples in this arc implement the terminal surface.
  A web surface for the same domain would use a backend-api archetype
  (FastAPI, Go Echo, etc.) exposing the same job data via REST.

  ## Connecting Terminal and Web

  When building both, consider:
  - A shared local HTTP API (see examples/canonical-terminal/docker/) that
    the terminal tool connects to via --api-url and the web UI connects to.
  - The terminal `--output json` flag makes terminal tool output consumable
    by web pipelines (webhook, curl-to-API patterns).
  - PTY-based terminal output can be embedded in web UIs via xterm.js or
    similar (advanced pattern; not in scope for this arc).

  ## See Also
  - context/archetypes/terminal-dual-mode.md
  - context/archetypes/backend-api-service.md
  - examples/canonical-terminal/DECISION_GUIDE.md
  - examples/canonical-api/ (web parity examples for REST APIs)

Create: verification/terminal/comparison.py

This harness runs the same CLI operation across all available language
implementations and compares their JSON output for structural consistency.

Goal: verify that the TaskFlow domain is implemented consistently across 7 languages.
If `taskflow list --output json` returns 20 items in Python, it should return
20 items in Rust, Go, TypeScript, Java, Ruby, and Elixir — same fixtures, same count.

  from dataclasses import dataclass
  from typing import Any
  import json, subprocess
  from pathlib import Path

  @dataclass
  class ComparisonResult:
    example_name: str
    command: list[str]
    exit_code: int
    output: str
    parsed: Any               # JSON-parsed output, or None on error
    parse_error: str | None

  def run_comparison_op(examples: list, command_template: list[str],
                         fixtures_env: dict) -> list[ComparisonResult]:
    """
    Run command_template against each available example.
    command_template: e.g., ["taskflow", "list", "--output", "json"]
    Returns one ComparisonResult per example.
    """

  def compare_list_outputs(results: list[ComparisonResult]) -> dict:
    """
    Compare list --output json results across examples.
    Returns:
      {
        "expected_count": 20,
        "results": {
          "python-typer-textual": {"count": 20, "match": True},
          "rust-clap-ratatui":    {"count": 20, "match": True},
          ...
        },
        "all_match": True,
        "mismatches": []
      }
    """

  def compare_stats_outputs(results: list[ComparisonResult]) -> dict:
    """
    Compare stats --output json results across examples.
    Assert: total == 20 across all implementations.
    """

  def compare_inspect_outputs(results: list[ComparisonResult], job_id: str) -> dict:
    """
    Compare inspect <job_id> --output json results across examples.
    Assert: id == job_id in all results.
    """

Create: verification/terminal/test_comparison.py

  class TestCrossLanguageComparison(unittest.TestCase):

    def test_list_json_count_consistent(self):
      """
      All available examples should return 20 jobs for list --output json.
      Skip examples that are not installed/available.
      Assert all available examples agree on count.
      """

    def test_stats_total_consistent(self):
      """
      All available examples should return total=20 for stats --output json.
      """

    def test_inspect_job001_id_consistent(self):
      """
      All available examples should return id="job-001" for inspect job-001.
      """

    def test_filter_done_all_status_done(self):
      """
      All available examples: list --status done --output json
      → all returned items have status "done"
      """

  Mark individual example sub-checks with @unittest.skipUnless so that
  unavailable examples are skipped rather than failed.
