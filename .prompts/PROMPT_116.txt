BACKGROUND: GO AND RUST CONCEPTUAL FRAMING
================================================================================

GO — The Externalized Contract Problem

Go does not have a library that unifies type definition, runtime validation,
and schema export in the way Pydantic or Zod do. This is a real architectural
difference, not a gap to paper over.

  go-playground/validator: Struct tags drive validation rules. The struct
  definition carries tags like `validate:"required,min=3,max=100"`. The
  validator reads them at runtime via reflection. It is mature and widely used.
  Limitation: cross-field rules and discriminated union validation are awkward
  with tags alone.

  ozzo-validation: Explicit code drives validation. You call
  validation.ValidateStruct(&obj, validation.Field(&obj.Name, is.Required, ...)).
  More readable cross-field rules. Better for complex business logic. More
  verbose for simple cases.

  Contract generation: Go structs can generate JSON Schema or OpenAPI specs via
  external tools (swaggo annotations, kin-openapi, openapi-generator, or hand-
  crafting). There is no single standard. Show the swaggo comment annotation
  approach because it is the most common in Go HTTP services. Acknowledge the
  ergonomic gap — do not pretend Go has Pydantic-style inline schema derivation.

RUST — Separation of Concerns as a Feature

Rust's approach is the clearest example of separation of concerns in this arc:

  serde: Defines how a type is serialized and deserialized. Attributes like
  #[serde(rename_all = "camelCase")] and #[serde(skip_serializing_if = "Option::is_none")]
  control the wire representation. serde does NOT validate.

  validator / garde: Validate struct field values against rules. Attributes like
  #[validate(length(min = 3))] or #[garde(length(min = 3))] check constraints.
  These libraries do NOT control serialization.

  schemars: Derives a JSON Schema from a Rust struct, using serde's representation
  as the reference. The generated schema reflects the serde wire format, not the
  Rust type system representation. This is the CONTRACT lane.

This separation is a design strength, not a limitation. Make it explicit:
  1. deserialize from JSON (serde)
  2. validate the deserialized value (validator or garde)
  3. export a JSON Schema reflecting the wire format (schemars)

These are three distinct operations, each with its own library and derive macro.
Never conflate step 3 (schemars) with step 2 (validator/garde) in code or docs.
