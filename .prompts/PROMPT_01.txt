You are designing and implementing the best possible foundational base repository for future software projects that will be used with frontier coding assistants such as Codex, Claude, and Gemini.

This repository is not a finished application. It is a reusable agent-oriented base repo that future projects will start from. Its purpose is to provide an excellent context architecture, routing system, doctrine layer, workflow layer, stack packs, archetype packs, canonical example strategy, manifest/profile loading system, and bootstrap-ready structure so that future repos can be created cleanly and used effectively with coding assistants.

The output should be implementation-oriented and as concrete as possible. Do not produce vague advice. Produce directory structures, file contents, contracts, naming conventions, manifest schemas, routing logic, examples, and strong documentation drafts.

Treat `.prompts/PROMPT_00_s.txt` as the baseline architectural brief for this repo. Refine and operationalize that design for the reusable base repo; do not restart from a competing architecture unless a small correction is required for consistency or implementation quality.

The base repo should reflect how I actually build and think. It should strongly support these real patterns and preferences:

- prompt-first repo workflows
- reusable AGENT.md / CLAUDE.md context systems
- doctrine separated from workflows and examples
- deterministic task routing
- canonical examples
- smoke-test-heavy development
- bootstrap-friendly architecture
- strong repo-level guidance for coding assistants
- polyglot extensibility
- Docker-backed local development where appropriate
- Dokku-oriented deployment thinking instead of Traefik-first deployment thinking
- modern Python stack preferences: FastAPI, uv, ruff, orjson, polars
- modern Rust stack preferences: Axum and modern Rust tooling
- Go web stack preference: Echo
- Elixir web stack preference: Phoenix
- support for Redis/KeyDB, MongoDB, DuckDB, Trino, NATS Jetstream, Meilisearch, TimescaleDB, Elasticsearch, Qdrant, local RAG, CLI tools, data pipelines, and prompt meta-runner style repos
- support for future VPS deployment patterns, especially Dokku-oriented app packaging and conventions
- strong anti-sprawl design so future projects load only the smallest relevant context

The base repo should be designed to help pioneer uniquely effective usage patterns for all three frontier assistants:
- Codex
- Claude
- Gemini

The system must reduce the burden on the human operator. The human should not need to memorize skill names, workflow names, or exact manifest names in order to use the repo effectively. The repo itself should help the assistant infer what to load based on normal task descriptions and repository signals.

Operational infrastructure invariants for future repos derived from this base:
- When Docker Compose files are created, they should keep the conventional filenames `docker-compose.yml` and `docker-compose.test.yml`, while the top-level Compose `name:` field should be repo-derived so `docker ps` output is easy to read.
- Compose `name:` values should be repo-derived so the primary/dev stack and test stack can run in parallel without collisions, with the test stack carrying an explicit test suffix such as `<repo>-test`.
- Host ports for local web servers and backing services should be explicit non-default values, and the test stack must use a different non-default port band from the primary/dev stack.
- Test fixture data, test seed/reset flows, test env files, test databases, and test volumes must be fully isolated from primary/dev data.
- The system should make it easy for operators to run primary/dev and test infrastructures side by side and compare their behavior.

Treat this as a serious systems-design and implementation task.
