================================================================================
BACKGROUND: STARTUP VISIBILITY PROBLEM
================================================================================

After PROMPT_91-93, the system has strong foundations: 3-tier memory, operator console,
behavioral contract. But a subtle gap remains: there is no explicit record of what context
a session loaded, and no single place that tells a fresh assistant "here is exactly what
you should load and what the current state is" in a standardized, readable format.

The existing `work.py resume` output is good but assistant-focused. It shows git state,
runtime file health, next-step hints, and checkpoint reminders. What it lacks is a
clear, structured "Session Context Briefing" that answers these questions upfront:

  - What is the current repo state (cleanliness, branch, HEAD)?
  - What context files are present and loaded?
  - What is the complexity budget / memory posture?
  - What is the operator queue state (next prompt, quota readiness)?
  - What memory summaries exist that are relevant to the current work?
  - What is the single recommended next action?

This briefing should be:
  - Printed by `work.py resume` (and optionally stored as a startup log artifact)
  - Inspectable after the fact
  - Not a hallucination — it reflects actual file presence, not assumed loading

Implementation notes:

- `print_session_context_briefing` receives the RepoInspection object (already available)
  plus the repo_root path. Detect memory/ files by checking the filesystem.

- For any local planning-status section, prefer signals the repo actually has today:
  active `tmp/*.md` files, runtime markdown presence, and git/history clues. Do not
  invent a `work/` state model if the live repo does not have one.

- For "Complexity Budget": sum the line_count of present runtime files. The thresholds
  (lean/moderate/heavy) are heuristics, not hard limits. Show the number, not just a label.

- For "Memory Base / Relevant summary": check memory/summaries/ for files matching
  PROMPT_XX_resume.md or PROMPT_XX_completion.md where XX is the most recent prompt
  prefix found in git history.

- Keep the function deterministic and fast. Do not shell out to additional git commands
  beyond what inspect_repo() already runs.

- The briefing is printed as a clearly delimited block so it is easy to find in a long
  terminal session.

This file documents the startup visibility model for the repo. Cover:

1. The problem it solves
   Without explicit startup logging, sessions are opaque: it is unclear what was loaded,
   what the complexity posture was, or what the operator queue said at start.

   Level 1 — Session Context Briefing (printed by `work.py resume`)

     Optional; persists to logs/startup/ for after-the-fact review.

   - Always run `work.py resume` first in a fresh session
   - Read the briefing before loading task-specific context
   - If complexity budget is heavy, prune before starting
   - If quota is not ready, wait or use a different assistant
   - If a relevant memory summary exists, load it

   Brief table or bullet list: what lean/moderate/heavy means and what to do

5. Integration with memory/ and work/
   The briefing reads from both. It is the synthesis point for the session start.
