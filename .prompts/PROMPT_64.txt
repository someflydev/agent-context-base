BEHAVIORAL SPECIFICATION

Internalize the following rules before producing any deliverable. Every file produced
must enforce them without exception.


RULE 4 — TEXT SEARCH COUNT SEMANTICS

Text search is a "base filter" applied before any facet logic runs. When a query
string Q is present (non-empty, after trimming):

  a) filter_rows(state): applies text search to narrow the row set first, then applies
     all dimension filters (team_in/out, status_in/out, region_in/out) to the narrowed set.
     Search is case-insensitive substring match against the report_id field (at minimum).
     Implementations may also match against other string fields (region, team, status);
     if they do, they must do so consistently and document it.

  b) facet_counts(state, dimension): the relaxation of same-dimension includes applies
     only to facet filters, never to the text search. Text search is always applied.
     Concretely: apply Q first, then apply all other dimensions' filters fully,
     then relax same-dimension includes. Count rows.

  c) exclude_impact_counts(state, dimension): apply Q first, then apply all other
     dimensions' filters fully (without this dimension's includes OR excludes),
     then count rows per value in this dimension.

  d) The result-count badge always reflects rows passing both Q and all facet filters.

  e) When Q is empty, behaviour is identical to the existing implementation (no change).

  f) The count shown next to each facet option changes when Q is active. This is
     correct and expected — the filter panel is showing counts scoped to what is
     currently visible given the search. Do not suppress or cache pre-search counts.

  g) The search input must carry: name="query", data-role="search-input",
     data-search-query="{Q}" (the current value). It must be inside the HTMX form so
     that typing triggers partial updates. Use hx-trigger="keyup changed delay:300ms"
     to debounce.


RULE 5 — SORT ORDER SEMANTICS

Sort affects the result display order only. It does NOT affect counts of any kind.

  a) Three options are required (exact values used as query parameter values):
       events_desc  —  "Events: high → low"   (default when sort param is absent)
       events_asc   —  "Events: low → high"
       name_asc     —  "Name: A → Z"           (alphabetical by report_id)

  b) The sort value is part of query state. It must be preserved across HTMX partial
     swaps: when the results fragment is re-rendered, the sort dropdown reflects the
     current sort value.

  c) The sort select element must carry:
       name="sort"
       data-role="sort-select"
       data-sort-order="{current sort value}"
     It must be inside the HTMX form so that changing the select triggers a partial update.

  d) The sort select is rendered in the results section header, above the result list
     but below the result-count badge.

  e) The query parameter name is "sort". Default (when absent): "events_desc".


RULE 6 — INDEPENDENT SCROLL LAYOUT

The page layout must allow the filter panel and the results section to scroll
independently. This is achieved through CSS only — no JavaScript required.

  a) The outermost wrapper uses a horizontal flex layout with a fixed viewport height:
       class="flex h-screen overflow-hidden"
       id="reports-layout"

  b) The filter panel (aside) is a fixed-width column with independent vertical scroll:
       id="filter-panel"
       class="w-72 flex-shrink-0 overflow-y-auto border-r p-4"

  c) The results section container takes remaining width with independent vertical scroll:
       id="report-results-container"
       class="flex-1 overflow-y-auto p-4"

  d) The actual results list (the HTMX swap target for partials) lives inside the
     results container. The results container is NOT itself the swap target — the
     existing id="report-results" element remains the swap target inside it.

  e) The layout wrapper is rendered once on full page load. HTMX partial swaps replace
     only id="report-results" (and OOB result-count badge). They do not re-render the
     layout wrapper. This ensures scroll position of the filter panel is preserved across
     result updates.

  f) Implementations must carry data-role="reports-layout" on the outer wrapper
     for Playwright assertions to locate and verify the layout structure.


UPDATED QUERY STATE MODEL

All existing six fields remain unchanged. Add two new fields:

  query   — string, the text search term (empty string when no search is active)
             Query parameter name: "query"
             Default: "" (empty string)

  sort    — string, one of: "events_desc", "events_asc", "name_asc"
             Query parameter name: "sort"
             Default: "events_desc"

Query parameter parsing:
  "query" is a single string value (not multi-value).
  "sort" is a single string value (not multi-value). Invalid values default to "events_desc".


UPDATED FINGERPRINT

The fingerprint string must include query and sort fields. Example format:
  "team_in=growth|team_out=|status_in=|status_out=archived|region_in=|region_out=|query=signup|sort=name_asc"


UPDATED DATA-* ATTRIBUTE CONTRACT (additions only — all PROMPT_50/51 attributes unchanged)

  On the outer layout wrapper:
    data-role="reports-layout"

  On the search input element:
    name="query"
    data-role="search-input"
