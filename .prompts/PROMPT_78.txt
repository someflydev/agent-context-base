1.  name: event-bus-backbone
    title: "Strategy 1: Event Bus as Backbone"
    primary_examples: [35, 68, 49, 50, 56]
    summary: |
      Use NATS JetStream or Kafka as the central nervous system: every service publishes
      domain events to typed subjects and subscribes only to events it cares about.
      Dead letter queues handle failures without blocking producers; replay enables
      debugging and reprocessing. 13 of the 100 projects already use or are compatible
      with event streaming patterns from the base repo.
    when_to_use: "When multiple services need to coordinate without tight coupling and event replay is a debugging requirement."
    tradeoffs: "Adds operational complexity (NATS/Kafka infra) and eventual consistency; removes direct service-to-service dependencies."

2.  name: gateway-mesh
    title: "Strategy 2: Gateway Mesh"
    primary_examples: [47, 57, 51, 60]
    summary: |
      Deploy multiple Go gateways that route to different service clusters: Gateway-A
      to data platform services, Gateway-B to ML/search services, Gateway-C to storage
      experiments. Each gateway owns its routing table and health probes. Cross-gateway
      communication goes through the event bus, never direct HTTP.
    when_to_use: "When scaling beyond 5 services and cross-service HTTP spaghetti becomes a maintenance risk."
    tradeoffs: "Adds Go gateway infrastructure; eliminates direct inter-service calls but requires maintaining routing tables."

3.  name: shared-data-layer
    title: "Strategy 3: Shared Data Layer"
    primary_examples: [67, 1, 3, 63, 42, 44, 83]
    summary: |
      Three storage tiers: Redis for sub-millisecond cached results, PostgreSQL for
      normalized operational data, MinIO+Parquet for analytical and archival data.
      Services write to the appropriate tier based on access pattern. DuckDB queries
      cold-tier Parquet without moving data; Qdrant holds vector embeddings as a
      parallel index.
    when_to_use: "When different services need different access patterns against the same underlying data."
    tradeoffs: "Requires discipline about which tier owns which data; avoids duplication but adds schema coordination overhead."

4.  name: prompt-first-regeneration
    title: "Strategy 4: Prompt-First Regeneration"
    primary_examples: [85, 86, 87, 88, 89, 90]
    summary: |
      Each of the 100 projects was generated from a 2-4 sentence prompt and built via
      a prompt sequence (.prompts/ directory). Any service can be regenerated from
      scratch by replaying its prompts; services can be forked by modifying the initial
      prompt. The prompt evaluator audits prompt quality across all services.
    when_to_use: "When a service needs to be rebuilt, forked for a new domain, or audited for prompt quality drift."
    tradeoffs: "Relies on the generating AI maintaining consistent output given the same prompts; best verified by the test suite."

5.  name: verification-mesh
    title: "Strategy 5: Verification Mesh"
    primary_examples: [22, 23, 24, 25, 26, 27, 73, 77]
    summary: |
      Cross-service verification using the base repo's testing philosophy: each service
      has smoke tests (fast, no external deps), integration tests (real containers via
      docker-compose), and cross-service tests that verify seam contracts end-to-end.
      The reconciliation CLI and config auditor provide offline quality and compliance
      checks.
    when_to_use: "Always — this is the baseline verification posture for any combination of projects."
    tradeoffs: "Integration tests require docker-compose and are slower; smoke tests provide the fast feedback loop."
