TERMINAL TOOLING CONTEXT
================================================================================

This prompt is part of the TERMINAL TOOLING ARC (PROMPT_100–111).

The arc goal: make terminal tooling (CLI tools, TUI apps, dual-mode tools) a
first-class, routed, doctrinally coherent capability area in agent-context-base.

Key architectural principles (do not deviate):
  - Shared domain core separates from CLI/TUI surfaces
  - Non-interactive mode is mandatory for every TUI tool
  - Fixture-first validation: no live network in smoke tests
  - Marker-based or structured output for CLI assertions
  - PTY-based or scripted validation for TUI (with documented fallback)
  - Fixture corpus lives in examples/canonical-terminal/fixtures/
  - All examples must be independently smoke-testable in CI

Structure and content:

  # Terminal Validation Contract

  ## Purpose
  Defines how terminal tools (CLI, TUI, dual-mode) must be tested to satisfy the
  terminal tooling doctrine (context/doctrine/terminal-ux-first-class.md).

  ## 1. CLI Smoke Tests (mandatory for all terminal examples)

  Every terminal example must include at least one CLI smoke test that:
  - Invokes the tool in non-interactive (headless) mode
  - Loads fixture data (not live data) from examples/canonical-terminal/fixtures/
  - Asserts on at least one marker or structured output field
  - Passes without a TTY (exportable TERM=dumb or redirected stdout)
  - Runs in under 5 seconds

  Assertion strategies:
  a) Marker-based: output contains BEGIN_TABLE / END_TABLE, BEGIN_JOB / END_JOB, etc.
     The tool wraps sections in these markers in non-interactive mode.
  b) Structured output: use --output json; assert on parsed JSON fields.
  c) Exit-code: assert exit code 0 for normal operation, non-zero for errors.

  Preferred assertion: --output json + assert parsed fields. Fallback: markers.

  ## 2. CLI Integration Tests (optional, for real-backend extensions)

  When a terminal example supports a live backend (e.g., real database, API),
  integration tests may test against that backend. Integration tests:
  - Live in tests/integration/ (not tests/smoke/ or tests/unit/)
  - Are NOT required to pass without a live backend
  - Are NOT included in --tier fast verification runs
  - Must be gated: skipped if the backend is unavailable

  ## 3. TUI Validation (required; method depends on language)

  TUI applications must have a documented validation path. Options in priority order:

  Option A — PTY + pexpect (Python, usable for others via subprocess):
    Use pexpect or equivalent to spawn the TUI process in a PTY, send keystrokes,
    capture rendered output, assert on text patterns. The TUI must exit cleanly on 'q'.

  Option B — Headless --no-tui flag:
    The TUI tool accepts --no-tui (or equivalent) to run the domain logic and print
    fixture results to stdout without entering TUI mode. Assert on stdout output.
    This is also how CI smoke tests work for TUI tools.

  Option C — Scripted transcript:
    Provide a documented manual script: "Run the tool, press these keys, verify
    these screens appear." Include a TODO for automation.

  Mandatory: every TUI tool must support Option B (--no-tui) regardless of whether
  Option A is also implemented.

  ## 4. Output Assertion Markers

  CLI tools in this repo use marker-based sections when --output default (human-readable).
  Markers allow string-matching assertions without parsing ANSI or terminal formatting.

  Standard markers:
    <!-- BEGIN_JOBS -->      <!-- END_JOBS -->        (job list output)
    <!-- BEGIN_JOB_DETAIL --> <!-- END_JOB_DETAIL --> (single job detail output)
    <!-- BEGIN_STATS -->     <!-- END_STATS -->       (aggregate stats output)
    <!-- BEGIN_ERROR -->     <!-- END_ERROR -->       (error output)

  Alternatively, HTML-comment markers may not suit all languages. Acceptable formats:
    ## BEGIN_JOBS ##    ## END_JOBS ##
    [BEGIN_JOBS]        [END_JOBS]

  The implementing assistant may choose the marker format that fits the language
  idiom, as long as it is consistent within the example and documented in its README.

  ## 5. Fixture-First Rule

  All smoke tests MUST load data from examples/canonical-terminal/fixtures/
  and NOT from any live backend. Smoke tests must work in:
  - CI without any running service
  - Offline environments
  - Fresh clone without configuration

  The fixture corpus (created in PROMPT_102) includes:
    jobs.json       — 20 job records (varied statuses, tags, durations)
    events.json     — 30 event stream records (job state transitions)
    config.json     — sample tool configuration (queue name, refresh interval)

  ## 6. Language-Specific Notes

  Python (Typer+Textual, Click+Blessed):
    - pytest or unittest for CLI smoke tests
    - pexpect for PTY-based TUI tests (if Textual supports PTY; else --no-tui only)
    - Textual provides its own testing app driver for unit-level widget tests

  Rust (clap+ratatui, argh+tui-realm):
    - cargo test for unit and integration tests
    - ratatui Backend::TestBackend for unit-testing draw output
    - Non-interactive --no-tui flag required; PTY via expect or rexpect crate

  Go (Cobra+Bubble Tea, urfave+tview):
    - go test for CLI tests
    - Bubble Tea: bubbletea.Model unit testing with tea.Batch
    - Non-interactive --no-tui flag required

  TypeScript (Commander+Ink, Yargs+Blessed):
    - jest or vitest for CLI tests
    - Ink: @ink-testing-library for component rendering tests
    - Non-interactive --no-tui flag required

  Java (picocli+Lanterna, JCommander+JLine):
    - JUnit 5 + picocli CommandLine for CLI tests
    - Lanterna: VirtualTerminal or MockTerminal for TUI unit tests
    - Non-interactive --no-tui flag required

  Ruby (Thor+TTY, Clamp+TTY):
    - RSpec or minitest for CLI tests
    - TTY::TestPrompt for headless prompt testing
    - Non-interactive --no-tui or --batch flag required

  Elixir (Optimus+Ratatouille, OptionParser+Owl):
    - ExUnit for all tests
    - Ratatouille: ExTermbox.fake_backend/0 for TUI unit tests if available
    - Non-interactive --no-tui or Mix.env :test override required

  ## 7. Smoke Test Structure Convention

  Canonical example smoke tests should be organized as:

    tests/
      smoke/
        test_cli_smoke.*     — non-interactive CLI invocations, output assertions
        test_tui_pty.*       — PTY-based TUI test (optional; adds note if absent)
      unit/
        test_core.*          — domain core unit tests (filtering, state, data loading)

  Language variations are acceptable. The smoke/test_cli_smoke.* file is mandatory.

  ## 8. Verification Integration

  Terminal smoke tests integrate with the repo's existing verification tiers:

    --tier fast:   Runs all smoke tests for all terminal examples.
                   All must pass without network or TTY.
    --tier slow:   Runs integration tests if backends are available.
    --tier tui:    Runs PTY-based TUI tests (added in PROMPT_108).

  The implementing assistant for PROMPT_102–105 must ensure:
    - Smoke tests pass under `python scripts/run_verification.py --tier fast`
    - Test discovery paths are registered in verification/ if required by the
      repo's existing verification runner (inspect verification/ to confirm)

  # Terminal Smoke Baseline

  ## Purpose
  Verifies that all terminal canonical examples pass their CLI smoke tests
  in non-interactive, fixture-backed mode.

  ## Applies To
  All examples in examples/canonical-terminal/

  ## Preconditions
  - examples/canonical-terminal/fixtures/jobs.json exists
  - Each example's dependencies are installed

  ## Test Sequence
  For each implemented language example:
  1. Run CLI smoke test in headless mode
  2. Assert exit code 0
  3. Assert at least one marker or JSON field in output

  ## Pass Criteria
  - All CLI smoke tests exit 0
  - Output contains expected markers or JSON structure
  - No network calls made during smoke run

  ## Failure Modes
  - Missing fixture file → pre-condition failure (not a smoke test failure)
  - Exit code non-zero → implementation error
  - Missing marker → output assertion contract violated
