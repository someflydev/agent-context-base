      cli/
        mod.rs
        commands.rs    # list, inspect, stats, watch --no-tui implementations
        output.rs      # table formatter, json output, marker wrapping
      tui/
        mod.rs
        app.rs         # ratatui App struct, event loop, draw functions
        ui.rs          # layout/widget rendering functions
        state.rs       # AppState: selected job, filter, scroll position
    tests/
      smoke_cli.rs     # cargo test smoke tests (non-interactive, fixture-backed)
    Cargo.toml
    README.md

--- src/core/models.rs ---

  #[derive(Debug, Clone, Serialize, Deserialize)]
  pub enum JobStatus { Pending, Running, Done, Failed }

  #[derive(Debug, Clone, Serialize, Deserialize)]
  pub struct Job {
    pub id: String,
    pub name: String,
    pub status: JobStatus,
    pub started_at: Option<String>,
    pub duration_s: Option<f64>,
    pub tags: Vec<String>,
    pub output_lines: Vec<String>,
  }

  // Stats struct: total, by_status HashMap, avg_duration_s, failure_rate

--- src/cli/commands.rs ---

  clap App with subcommands:
    list: --status, --tag, --output (json|table)
    inspect: JOB_ID, --output (json|plain)
    stats: --output (json|plain)
    watch: --interval, --no-tui

  All commands load from --fixtures-path arg or TASKFLOW_FIXTURES_PATH env var.
  Markers on table/plain output (## BEGIN_JOBS ## etc.).
  --output json: serialize to serde_json, print, exit.

--- src/tui/app.rs ---

  ratatui App with:
  - crossterm raw mode + alternate screen
  - Event loop: keyboard events (q=quit, r=refresh, ↑↓=select, f=filter)
  - Draw function: 3-panel layout (header, job table, detail panel)
  - Job table: id, name, status (colored), duration
  - Detail panel: shows selected job's full info and output_lines
  - On 'q': restore terminal and exit cleanly

  Use ratatui::backend::CrosstermBackend.
  Handle all terminal restore on panic (std::panic::set_hook or scopeguard).

--- tests/smoke_cli.rs ---

  #[cfg(test)]
  Integration tests using std::process::Command:
  1. list_table_output: run binary with list subcommand, check markers in stdout
  2. list_json: run with --output json, parse JSON, assert 20 items
  3. filter_by_status: --status done --output json, assert all done
  4. inspect_job: inspect job-001 --output json, assert id match
  5. stats_json: stats --output json, assert total == 20
  6. watch_no_tui: watch --no-tui, check exit 0 and markers

  Pass fixtures path via --fixtures-path arg pointing to
  examples/canonical-terminal/fixtures/ (use cargo test env or relative path).

  # TaskFlow Monitor — Rust (clap + ratatui)

  Flagship Rust dual-mode terminal example. Demonstrates clap-based CLI parsing
  with a ratatui terminal dashboard sharing a pure domain core.

  ## Stack: clap + ratatui + crossterm

  ## Usage
  ```bash
  cargo build --release
  ./target/release/taskflow list
  ./target/release/taskflow list --status failed --output json
  ./target/release/taskflow watch --no-tui
  ./target/release/taskflow watch   # launches TUI
  ```

  ## Testing
  ```bash
  cargo test
  ```

  ## Architecture
  src/core/: domain logic (models, loading, filtering) — no terminal I/O
  src/cli/:  clap subcommand dispatch and output formatting
  src/tui/:  ratatui app, event loop, and UI rendering

  ## When to Use
  High-performance Rust tools needing both scriptable CLI and a rich terminal UI.

PART 2: RUST SECONDARY — argh + tui-realm (taskflow-rs-tui-realm)
================================================================================

Create: examples/canonical-terminal/rust-argh-tui-realm/

This secondary example demonstrates a component/event-driven TUI architecture using
tui-realm over ratatui, with argh for a simpler CLI parser.

--- Scope ---

Implement the full domain core (same models/loader/filters as flagship).
Implement all 4 CLI commands (list, inspect, stats, watch) using argh.
Implement TUI using tui-realm's component/subscription model.

Key difference from flagship:
  - tui-realm: components register subscriptions to events; updates are dispatched
    via messages. Think Elm architecture. Each panel (JobList, JobDetail) is a Component.
  - argh: lighter argument parsing (struct-based, no builder pattern)

Structural similarity: same directory layout, same fixture path, same markers,
same --no-tui requirement, same smoke tests (adjust for argh CLI parsing).

--- Cargo.toml ---

  [dependencies]
  argh = "0.1"

  ratatui = "0.26"        # tui-realm depends on ratatui
  crossterm = "0.27"
  serde = { version = "1", features = ["derive"] }
  serde_json = "1"
  anyhow = "1"

--- README.md ---

  # TaskFlow Monitor — Rust (argh + tui-realm)

  Secondary Rust terminal example demonstrating tui-realm's component/event
  architecture with argh's lightweight CLI parser.

  ## When to Use vs Flagship
  - Use clap+ratatui for maximum CLI flexibility and direct ratatui control
  - Use argh+tui-realm when the CLI is simpler and the TUI benefits from
    a component-subscription model (easier to extend with more panels)
