================================================================================
BACKGROUND: THE TWO-LAYER MODEL
================================================================================

The current PROMPT_META_RUNNER.txt was generated at an early stage to describe how to
run this repo's own prompt sequence. It is sophisticated — it defines subagent roles
(AGENT_MAIN, AGENT_ARCHITECT, AGENT_BUILDER, AGENT_AUDITOR, AGENT_REFACTOR), execution
loops, artifact gates, testing discipline, commit discipline, and a "Start Procedure."

That sophistication was appropriate when the repo needed a detailed runtime guide. But
it has an implicit assumption that PROMPT_META_RUNNER is the controller — that pasting
it into an assistant session makes the assistant manage its own execution lifecycle,
multi-prompt sequencing, quota awareness, and commit discipline automatically.

This assumption does not hold cleanly:
- Assistants cannot truly enforce fresh-session boundaries
- They cannot observe real quota state
- They cannot guarantee commit discipline without operator oversight
- Multi-prompt sequencing from within a single session is drift-prone

The historical design introduced in PROMPT_92 proposed `work.py` operator-console
commands. If the live repo still has only the base runtime-state commands, treat that
operator-console layer as a proposal to be rebased rather than an already-landed fact.

The repositioning to make in this prompt:

  PROMPT_META_RUNNER.txt becomes:
  - A behavioral contract for ONE bounded assistant session
  - Not an automation layer that sequences multiple prompts
  - Not a quota-awareness tool
  - A clear, principled guide to how an assistant should behave during a single session

  scripts/work.py becomes:
  - The local operator console for managing prompt-to-prompt progression
  - The durable truth layer between sessions
  - The tool the OPERATOR uses (not the assistant)

This distinction must be reflected in PROMPT_META_RUNNER.txt, doctrine, and docs.

--- SECTION 1: Purpose ---

Clearly state what this file is and what it is NOT:

  This is a behavioral contract for a single bounded assistant session executing one
  `.prompts/` file. It is NOT a multi-session scheduler, a quota tracker, or a

  sessions. This file governs what happens INSIDE a session.

--- SECTION 2: Single-Session Invariants ---

Rules that apply to every prompt session, regardless of which prompt is being executed:

  1. One prompt per session
     A session executes exactly one `.prompts/` file. The session ends when that prompt's
     completion condition is met. Do not begin the next prompt in the same session.

  2. Boot before work
     Every session must boot via the sequence in AGENT.md/CLAUDE.md before any
     implementation starts. Check `work.py resume` output early.

  3. Commit discipline
     All commits during a session must begin with [PROMPT_XX] matching the active prompt.
     Use `git add -p` only for logical groupings that cannot be staged cleanly by file. Never batch changes from different prompts.

  4. Validation before done
     A prompt is not complete until its stated verification steps pass. Report `blocked`
     or `incomplete` if they cannot be run or do not pass.

  5. Explicit scope
     Stay within the scope of the active prompt. Improvements to adjacent files are only
     allowed if they are required for the prompt's stated completion condition.

  6. No auto-sequencing
     When a prompt's completion condition is met, stop and report. Do not move to the
     next prompt. The operator manages sequencing outside the assistant session.

--- SECTION 3: Operator Console Integration ---

Explain the two-layer model:

  - The OPERATOR uses the live repo workflow to manage sequencing and session lifecycle.
  - The ASSISTANT uses boot docs, work.py resume, memory/, and prompt content to do the work.
  - These layers are separate. Do not assume `work.py start/pause/done` exist unless the
    repo actually has them.

Provide the expected operator workflow around a session:
  Before:   review runtime state and pick the next prompt
  During:   assistant works; operator may inspect or checkpoint as needed
  After:    write the completion summary and decide what prompt comes next

--- SECTION 4: Artifact Gates ---

A condensed but complete list of what must be true before a prompt is declared done:

  - Promised files exist with correct content (no unfilled placeholders)
  - Committed artifacts match what the prompt described
  - Smoke tests or stated verification steps pass (or are documented as blocked/N/A)
  - Commit prefixes match [PROMPT_XX]
  - No unintended template overwrites
  - Completion condition as stated in the prompt is met

--- SECTION 5: Commit and File Conventions ---

  - git add -p only for logical groupings that share a file

--- SECTION 6: Drift Stop Conditions ---

Retain the key stop conditions from the current file, adapted for single-session scope:

--- SECTION 7: Subagent and Role Conventions (condensed) ---

Keep the essential role taxonomy but simplify. The goal is recognizable role names that

Most sessions use one role. Role-switching mid-session is allowed but must be noted.

--- SECTION 8: Testing Discipline (condensed) ---

Retain the core principle from the current file:

  Testing is not applicable for documentation-only or doctrine-only prompts.
