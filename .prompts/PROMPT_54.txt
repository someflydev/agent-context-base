TRIO STACK DOC CONTENT NOTES

TRIO 1 — Go + Elixir + Python
  The canonical AI platform trio: throughput, coordination, intelligence.
  Seam map:
    [Go Gateway] ── NATS ──► [Elixir Coordinator] ── REST ──► [Python Inference]
  Go owns: external API ingestion, rate limiting, auth, high-throughput HTTP
  Elixir owns: distributed job coordination, fault-tolerant retries, result fan-out,
               real-time delivery to clients via Phoenix Channels
  Python owns: ML model inference, feature extraction, scoring, batch analytics
  Seam A↔B: NATS JetStream — Go publishes inference jobs, Elixir distributes them
  Seam B↔C: REST (HTTP/JSON) — Elixir POSTs to Python scoring endpoint
  Note: Elixir is the coordinator, not just a pass-through. It manages job queues,
        tracks which inferences are in-flight, and fan-outs results.
  Use cases: real-time recommendation engines, fraud detection pipelines,
             content moderation platforms, IoT telemetry + ML scoring

TRIO 2 — Go + Rust + Python
  The performance stack: gateway, hot-path kernel, AI layer.
  Seam map:
    [Go Gateway] ── gRPC ──► [Rust Kernel] ── REST ──► [Python ML]
    (or Go calls both Rust and Python separately — note both topologies)
  Go owns: API gateway, routing, auth, service mesh, ops tooling
  Rust owns: CPU-intensive processing — tokenization, embedding computation,
             search indexing, codec work, data normalization at scale
  Python owns: model training, advanced analytics, ML experimentation, batch jobs
  Primary topology: Go receives a request, calls Rust for compute preprocessing,
                    then calls Python for inference. Rust may return processed
                    features that Python uses directly.
  Seam A↔B: gRPC — Go calls Rust for synchronous compute results
  Seam B↔C: REST — Rust results fed to Python OR Go calls Python directly
             (document both topologies; prefer Go→Rust→Python pipeline)
  Use cases: AI search platforms, large-scale data processing with ML scoring,
             document processing pipelines, embedding pipelines at scale

TRIO 3 — Elixir + Go + Rust
  Fault-tolerant coordination + high-throughput services + low-latency compute.
  Seam map:
    [Elixir Coordinator] ── NATS ──► [Go Services] ── gRPC ──► [Rust Kernel]
  Elixir owns: process supervision, distributed coordination, OTP trees,
               fault isolation, retry orchestration, multi-node state
  Go owns: external-facing HTTP APIs, fast worker pool, ingestion layer
  Rust owns: sub-millisecond compute — financial matching engine, search,
             cryptographic operations, real-time signal processing
  Seam A↔B: NATS — Elixir publishes coordination events, Go workers consume
  Seam B↔C: gRPC — Go calls Rust for latency-critical compute
  Use cases: high-frequency financial systems, game server infrastructure,
             telemetry processing at scale, low-latency data transformation

TRIO 4 — Node + Go + Python
  Consumer product stack: frontend contract, backend services, AI layer.
  Seam map:
    [Node BFF] ── REST ──► [Go Services] ── REST ──► [Python ML]
  Node/TypeScript owns: BFF layer, GraphQL schema, WebSocket sessions, SSE,
                        client auth tokens, frontend-specific data shaping
  Go owns: internal domain services, storage-backed CRUD, background workers,
            fast REST APIs consumed by Node
  Python owns: ML recommendations, analytics, complex scoring, data science
  Seam A↔B: REST — Node calls Go for domain data
  Seam B↔C: REST — Go calls Python for ML features (or Node calls Python directly
            if the ML result is needed at the BFF layer without Go mediation)
  Note: document both topologies (Node→Go→Python vs. Node→Go and Node→Python)
  Use cases: e-commerce with recommendations, social platforms with feed ranking,
             SaaS products with analytics dashboards, consumer apps with AI features

TRIO 5 — Clojure + Python + Go
  Business intelligence stack: event logic, ML, fast APIs.
  Seam map:
    [Clojure Events] ── Kafka ──► [Python Scorer] ── REST ──► [Go API]
    (Clojure produces enriched events; Python scores them; Go serves results)
  Clojure owns: event processing, business rule engines, Kafka Streams topology,
                data transformation, domain event enrichment
  Python owns: ML scoring, model inference, risk assessment, classification
  Go owns: low-latency read API, result serving, operational microservices
  Seam A↔B: Kafka — Clojure produces enriched domain events, Python consumes
  Seam B↔C: REST — Python writes scored results to a store that Go reads,
             or Python calls Go's ingest endpoint with results
  Use cases: insurance underwriting, credit scoring pipelines, fraud detection,
             financial risk analytics, healthcare risk stratification

TRIO 6 — Scala + Python + Go
  Data platform stack: streaming, analytics, lightweight API.
  Seam map:
    [Scala Streaming] ── gRPC ──► [Python Analytics] ── REST ──► [Go API]
  Scala owns: distributed streaming (Akka Streams / Spark), batch analytics,
              actor model coordination, stream topology definition
  Python owns: data science, ad-hoc analysis, model training, Jupyter-style
               exploration, complex aggregations using pandas/polars
  Go owns: lightweight operational API serving computed results,
            fast read endpoints, health monitoring, alerting hooks
  Seam A↔B: gRPC — Scala calls Python for analytical computation on stream windows
             OR REST — Scala writes to a shared store, Python reads for analysis
             (document both; prefer REST for simplicity unless latency matters)
  Seam B↔C: REST — Python writes analytics results, Go serves them
  Use cases: observability platforms, analytics SaaS, data-intensive applications,
             streaming ML feature pipelines, BI tooling backends

TRIO EXAMPLE 1: Go + Elixir + Python

Directory: examples/canonical-multi-backend/trios/go-elixir-python/

go-side.go
  - Receives an HTTP POST /submit-job with a JSON body: {"job_id": str, "features": [...]}
  - Publishes a NATS event to subject "jobs.submitted" with the job payload
  - Exposes /healthz checking NATS connectivity
  - Environment: NATS_URL

elixir-side.ex
  - Subscribes to "jobs.submitted" from NATS (GenServer with Gnat)
  - For each job: calls the Python scoring service via HTTP POST /score
  - Receives the score response and publishes a "jobs.scored" event to NATS
    (demonstrating the coordinator fan-out pattern)
  - Exposes /healthz
  - Environment: NATS_URL, PYTHON_SCORE_URL

python-side.py
  - FastAPI: POST /score accepting {"job_id": str, "features": [...]}
  - Returns {"job_id": str, "score": float, "label": str}
  - Uses a deterministic stub function for scoring (not real ML)
  - GET /healthz
  - Environment: PORT (default 8002)

docker-compose.yml
  - nats service (JetStream enabled)
  - go-service (port 8080), depends_on: nats
  - elixir-service (port 4000), depends_on: nats + python-service
  - python-service (port 8002)
  - All environment variables set correctly

seam.md
  Explains both seams: Go→Elixir (NATS, async), Elixir→Python (REST, sync).
  Explains why Go doesn't call Python directly (Elixir owns coordination and retry).
  How to run, what to observe, how to verify each seam is working.


TRIO EXAMPLE 2: Go + Rust + Python

Directory: examples/canonical-multi-backend/trios/go-rust-python/

go-side.go
  - Receives HTTP POST /process with {"document": str, "doc_id": str}
  - Calls Rust gRPC service to preprocess (tokenize/normalize) the document
  - Takes the preprocessed features from Rust and calls Python REST to score
  - Returns the final scored result
  - Exposes /healthz that checks both Rust gRPC and Python REST health
  - Environment: RUST_GRPC_URL, PYTHON_SCORE_URL

rust-side.rs
  - tonic gRPC server implementing ProcessService.Preprocess(PreprocessRequest)
  - PreprocessRequest: document string
  - PreprocessResponse: repeated double features, int32 token_count
  - Stub implementation: splits on spaces, converts word lengths to float features
  - Boots on GRPC_PORT env var
  - Exposes a /healthz HTTP handler (separate tokio task on HTTP_PORT)

python-side.py
  - FastAPI: POST /score accepting {"doc_id": str, "features": [...], "token_count": int}
  - Returns {"doc_id": str, "score": float, "category": str}
  - Stub scoring: score = mean(features), category based on token_count
  - GET /healthz
  - Environment: PORT (default 8003)

docker-compose.yml
  - rust-service (gRPC port 50051, HTTP health port 8090)
  - python-service (port 8003)
  - go-service (port 8080), depends_on: rust-service + python-service

service.proto (shared, placed in this directory for reference)
  - ProcessService.Preprocess RPC
  - Same format as kotlin-rust-grpc/service.proto but for document processing

seam.md
  Explains both seams: Go→Rust (gRPC, sync), Go→Python (REST, sync).
  Notes the pipeline topology: Go orchestrates, not Rust or Python.
  The proto file is the Rust↔Go contract. The JSON schema is the Go↔Python contract.
  How to run, how to regenerate proto stubs, what to observe.
