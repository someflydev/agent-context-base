Content notes:

When GraphQL IS the right choice:
- The BFF layer must serve multiple client types (mobile, web, third-party) with different
  field subsets — clients request exactly what they need
- Multiple underlying services (Go domain services, Python ML, Elixir real-time) need to be
  aggregated into a single typed API surface for the frontend
- Subscriptions are needed for real-time updates — GraphQL subscriptions over WebSocket
  are the standard pattern for pushing updates to subscribers
- The frontend team controls the GraphQL schema and drives API shape (schema-first)
- API introspection is a requirement — clients can self-discover the schema

When GraphQL is NOT the right choice:
- Simple CRUD APIs with one client — REST is simpler to implement and consume
- The caller is a non-frontend service calling another service — REST or gRPC between
  backend services; GraphQL as a BFF, not as an internal service protocol
- Real-time requires stateful bidirectional streams (voice, game frames) — WebSocket
  without GraphQL is more efficient for high-frequency binary streams
- The team is unfamiliar with GraphQL — the learning curve (resolvers, N+1, schema evolution)
  adds risk; REST first, GraphQL later

Typical repo surface (Node/TypeScript with Apollo Server):
  - src/schema.graphql — SDL schema definition (schema-first) or
    src/schema.ts — code-first schema with Pothos or Nexus
  - src/resolvers/ — resolver implementations per type
  - src/dataloaders/ — DataLoader instances for batching downstream calls
  - src/generated/ — codegen output (graphql-codegen outputs)
  - codegen.yml — graphql-codegen configuration

Schema design principles:
  - Schema-first preferred: write the .graphql SDL file, generate types for the server
    (graphql-codegen) and the client (gql.tada or graphql-codegen client preset)
  - Use GraphQL enums for finite value sets — avoids stringly-typed anti-patterns
  - Prefer nullable fields for optional data (matching GraphQL's null semantic)
  - Non-null (!) only on fields that are truly never null — overly strict non-null causes
    partial responses where callers expected a full response
  - Use Relay cursor-based pagination (Connection/Edge/Node) for paginated lists
  - Avoid returning raw error strings in schema fields — use a union type for result variants:
      union CreateUserResult = User | UserAlreadyExistsError | ValidationError

Query and mutation resolvers (Apollo Server 4, TypeScript):
  Schema-first with graphql-codegen:
  ```typescript
  // src/resolvers/User.ts
  import { Resolvers } from './generated/types';

  export const userResolvers: Resolvers = {
    Query: {
      user: async (_, { id }, { goUsersClient }) => {
        const resp = await fetch(`${goUsersClient}/users/${id}`);
        if (!resp.ok) return null;
        return resp.json();
      },
      users: async (_, __, { goUsersClient }) => {
        const resp = await fetch(`${goUsersClient}/users`);
        return resp.json();
      },
    },
  };
  ```

  Context type includes downstream client URLs / DataLoader instances:
  ```typescript
  interface Context {
    goUsersUrl: string;
    pythonScoringUrl: string;
    userLoader: DataLoader<string, User>;
  }
  ```

Subscriptions (WebSocket with graphql-ws):
  Apollo Server 4 does not bundle a WebSocket server. Use graphql-ws + ws:
  ```typescript
  import { createServer } from 'http';
  import { WebSocketServer } from 'ws';
  import { useServer } from 'graphql-ws/lib/use/ws';
  import { makeExecutableSchema } from '@graphql-tools/schema';

  const schema = makeExecutableSchema({ typeDefs, resolvers });
  const httpServer = createServer(app);
  const wsServer = new WebSocketServer({ server: httpServer, path: '/graphql' });
  useServer({ schema }, wsServer);
  ```

  Schema definition for subscriptions:
  ```graphql
  type Subscription {
    stateUpdated(docId: ID!): StateUpdate!
  }

  type StateUpdate {
    docId: ID!
    eventType: String!
    data: JSON
  }
  ```

  Resolver: subscribe to NATS subject and yield updates:
  ```typescript
  Subscription: {
    stateUpdated: {
      subscribe: async function* (_, { docId }) {
        const sub = natsConn.subscribe(`state.${docId}.updates`);
        for await (const msg of sub) {
          const event = JSON.parse(msg.string());
          yield { stateUpdated: event };
        }
      },
    },
  }
  ```

DataLoader pattern:
  DataLoader batches multiple resolver calls into one downstream request.
  Without DataLoader: a query for 10 users' orders calls the orders API 10 times.
  With DataLoader: all 10 user IDs are batched into one call.

  ```typescript
  import DataLoader from 'dataloader';

  const createUserOrdersLoader = (goOrdersUrl: string) =>
    new DataLoader<string, Order[]>(async (userIds) => {
      const resp = await fetch(`${goOrdersUrl}/orders/bulk`, {
        method: 'POST',
        body: JSON.stringify({ user_ids: userIds }),
      });
      const orderMap: Record<string, Order[]> = await resp.json();
      return userIds.map(id => orderMap[id] ?? []);
    });

  // In context factory (runs per request):
  context: ({ req }) => ({
    userOrdersLoader: createUserOrdersLoader(GO_ORDERS_URL),
  })
  ```

Code generation (graphql-codegen):
  The graphql-code-generator package generates typed resolvers and TypeScript types from
  the .graphql schema and resolver signatures.

  ```yaml
  schema: src/schema.graphql

  ```

Federation (Apollo Federation v2):
  Federation splits a large GraphQL schema across multiple subgraph services, each
  owned by a different team. The Router merges them into a supergraph.
  When to use:
    - Multiple teams each own a segment of the GraphQL schema
    - The schema exceeds what one BFF can reasonably own
    - Different parts of the schema have different deployment cycles
  When NOT to use:
    - Single-team BFF calling 2-3 downstream services — a single Apollo Server is simpler
    - The schema is stable and not growing across team boundaries
  Key concepts: @key directive for entity relationships, @external, @requires, @provides.
  Use rover CLI to compose the supergraph schema locally.

Language coverage (brief — full guidance in language-specific stack docs):
  Node/TypeScript (primary): Apollo Server 4, Pothos (code-first), graphql-codegen
  Go: gqlgen (code-first, generates Go types from schema, no reflection)

  Python: Strawberry (code-first, Pythonic) or Ariadne (schema-first)

Testing expectations:
- Test schema validation: ensure schema compiles with makeExecutableSchema
- Test resolvers: mock the downstream client URLs (not the entire service) in unit tests
- Integration test: docker compose up go-service + node-bff; run a real GraphQL query
  against http://localhost:3000/graphql; assert the response shape
- Test subscriptions: use a WebSocket client in tests; publish an event to NATS;
  assert the subscription yields the expected message
- Test DataLoader: verify batch function is called once for multiple resolver calls
  (not once per resolver call)
- Do NOT mock the GraphQL layer itself — test at the HTTP/WebSocket level

Common assistant mistakes:
- Putting business logic in resolvers — resolvers should only call downstream services
  and transform responses; business logic belongs in services or downstream APIs
- Not using DataLoader for any field that involves a nested list — any list that loads
  child objects will have an N+1 problem without DataLoader
- Using auto-commit session state in subscriptions — subscription connections are long-lived;
  clean up subscriptions and free resources on disconnect
- Returning 200 for GraphQL errors — GraphQL returns 200 even on errors (the error is in
  the "errors" field); do not confuse HTTP status codes with GraphQL error handling
- Over-using mutations for read operations — queries are idempotent; mutations are not;
  use the correct operation type
- Not generating types from the schema — hand-writing TypeScript types for GraphQL responses
  leads to drift; always use graphql-codegen

  - Uses rustler crate
  - Implements:
      compute_hash(input: &str) -> (Atom, String)
        → returns {:ok, hex_md5_of_input}
        → use md5 crate or a deterministic stub: format!("{:x}", input.len() as u64 * 0xDEADBEEF)
        → (use the stub to avoid a real crypto dep in the example; comment explains)
      normalize_vector(values: Vec<f64>) -> (Atom, Vec<f64>)
        → subtracts min, divides by (max - min); if all equal, returns zeros
        → returns {:ok, normalized_values}

  - Comment explaining the dirty NIF annotation for longer operations

  - Setup instructions (replaces docker-compose in this example):
    1. mix.exs dep: {:rustler, "~> 0.31", runtime: false}
    2. Place the Rust source at: native/compute_nif/src/lib.rs and native/compute_nif/Cargo.toml

    4. Call from IEx: MyApp.ComputeNif.compute_hash("test")

  - Why NIF (not Port, not a separate service):

    - Never block the BEAM scheduler for more than 1ms in a NIF — use dirty NIF for longer ops
