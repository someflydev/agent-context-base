BACKGROUND: OPERATOR CONSOLE
================================================================================

The current work.py has four commands that serve ASSISTANTS well:
  init        scaffold missing runtime files
  status      read-only health check
  resume      read-only session start briefing
  checkpoint  scaffold + report drift/compaction pressure

These commands treat the assistant as the active agent and the human as the observer.

This prompt introduces an additional layer: commands that serve the OPERATOR (the human
running the prompt-first workflow) as the active agent. The operator needs to:

  - know which prompt to run next and whether conditions are right to start
  - mark a session as started, paused, or done
  - log quota state when approaching rate limits
  - verify the repo is in a consistent state before starting a new session
  - review a concise prompt-aware commit digest

This is NOT a framework replacement. The existing four commands remain unchanged.
The new commands add an explicit operator-console layer on top of the existing model.

The data they read and write lives in `work/projects/<slug>/` — a gitignored directory
that holds local operator state. Do not use it as the home for prompt-session planning;
that belongs in local `tmp/*.md` files, while repo-root `PLAN.md` stays roadmap-only.

================================================================================
ARCHITECTURAL PRINCIPLES FOR THIS EXTENSION
================================================================================

Follow these principles throughout the implementation:

1. Explicit over magic
   Every state transition writes a real file. No hidden databases, no opaque registries.
   The entire work/ tree is readable with a text editor.

2. Backward compatible
   Existing commands (init, status, resume, checkpoint) must remain identical in behavior.
   Do not modify their output format or exit codes.

3. Graceful degradation
   If work/projects/<slug>/WORK_STATE.json does not exist, new commands print a helpful
   message and exit cleanly rather than crashing. `work.py init-project` creates it.

4. Read-only and write-only separation
   `status`, `resume`, `next`, `recent-commits`, and `verify` are read-only.
   `start`, `pause`, `done`, `log-quota`, and `init-project` are write commands.
   Never surprise the operator by mutating state during a read command.

5. Work state truth
   WORK_STATE.json is the single machine-readable truth for operator state.
   SESSION_QUEUE.md, COMMIT_LOG.md, RUN_HISTORY.md, and RATE_LIMITS.json are
   human-friendly views derived from or complementary to WORK_STATE.json.

6. No auto-running prompts
   work.py never reads, interprets, or executes .prompts/ files. It only tracks filenames
   as opaque strings. The operator and the coding assistant execute the prompts.

  Project: <slug>
  Branch:  <branch>
  Head:    <short hash> <subject>

  Next Prompt:   <prompt_file or "none pending">

------------------------------------------------------------------------

Write command. Marks a prompt as done.

------------------------------------------------------------------------
work.py verify [--slug SLUG]
------------------------------------------------------------------------

Read-only. Runs a structured consistency check.

Checks to perform:

1. Repo cleanliness
   - Report number of staged/unstaged changes
   - Warn if more than 10 changed files (broad surface)

2. Last commit prefix discipline
   - Check the last 5 commits for [PROMPT_XX] subject prefix pattern
   - Report which have it and which do not
   - Warn if none of the last 3 commits follow the convention

3. WORK_STATE.json consistency (if work/ exists)
   - current_prompt matches a queue entry with status "in_progress" or null
   - no queue entry has status "in_progress" and does not match current_prompt

4. Memory summary artifacts
   - For each "done" prompt in the queue, check if a completion summary exists
   - Report missing summaries as warnings

5. Validation readiness
   - Does python scripts/validate_context.py exist? Report its path.
   - Does python scripts/run_verification.py exist? Report its path.
   - Print the commands the operator should run before starting a new prompt.

Output format:

  Verify Report
  =============
  Repo cleanliness:   <clean | N changes (M staged, N unstaged)>
  Commit discipline:  <N/5 recent commits follow [PROMPT_XX] convention>
  Work state:         <consistent | warning: ...>
  Memory summaries:   <complete | missing: PROMPT_XX, PROMPT_YY, ...>

  Validation commands to run:
    python scripts/validate_context.py
    python scripts/run_verification.py --tier fast

  Overall: <OK | WARNINGS (see above)>
