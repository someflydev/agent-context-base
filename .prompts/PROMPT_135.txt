BACKGROUND: WHY A SEPARATE DOMAIN SPEC PROMPT
================================================================================

The canonical auth domain is more complex than a data-analytics domain spec
because it involves multiple interacting entities, role hierarchies, a
platform-managed permission catalog, canonical flows with specific actor
sequences, and a shared contract for /me output shape.

Getting this right once — before any implementation — means all 8 language
examples will be members of the same family rather than independent
interpretations. The spec is the parity contract.

The canonical domain is called TenantCore (the same SaaS platform used in
the faker arc). This is intentional dogfooding: the faker arc generates
realistic TenantCore data; the auth arc implements TenantCore's access
control. Fixture data in this arc should be consistent with the TenantCore
entity definitions established in the faker arc.

Required sections:

  # TenantCore IAM Domain Specification

  ## Purpose
  3–4 sentences. What TenantCore IAM is. Why it is the shared domain for
  the jwt-auth arc. How it connects to the TenantCore entity graph from
  the faker arc. What makes it a good teaching domain for auth patterns.

  ## Core Entities

  ### users
  Fields: id (UUID), email (unique), display_name, tenant_id (FK → tenants),
  created_at, is_active, acl_ver (int, incremented on permission change)

  ### tenants
  Fields: id (UUID), slug (unique), name, created_at, is_active

  ### memberships
  Fields: id (UUID), user_id (FK → users), tenant_id (FK → tenants),
  tenant_role (enum: super_admin / tenant_admin / tenant_member),
  created_at, is_active
  Note: super_admin has no tenant_id (it is NULL / omitted). Model this
  explicitly — super_admin is a global role.

  ### groups
  Fields: id (UUID), tenant_id (FK → tenants), slug (unique per tenant),
  name, created_at

  ### permissions
  Fields: id (UUID), name (format: {service}:{resource}:{action}),
  description, created_at
  Note: This is the platform-owned permission catalog. No tenant-local
  additions. Owned and seeded by the platform.

  ### group_permissions
  Fields: id (UUID), group_id (FK → groups), permission_id (FK → permissions),
  granted_at

  ### user_groups
  Fields: id (UUID), user_id (FK → users), group_id (FK → groups), joined_at

  ### audit_events
  Fields: id (UUID), actor_id (FK → users), actor_role, tenant_id (nullable),
  action, resource_type, resource_id, outcome (success/failure), created_at,
  metadata (JSON blob)

  ## Role Hierarchy

  A clear description of:
  - super_admin: global; not tenant-scoped; can create tenants, assign
    tenant_admins, inspect cross-tenant state. No tenant_id in token.
  - tenant_admin: tenant-scoped; creates users, groups, assigns permissions
    to groups from catalog, assigns users to groups within their tenant.
  - tenant_member: tenant-scoped; acts within their group-granted permissions.

  ## The Permission Catalog

  Define the canonical 20+ permission atoms used across all examples.
  Organized by service. All implementations must seed these exact atoms.

  Service: iam
    iam:user:read
    iam:user:create
    iam:user:update
    iam:user:delete
    iam:user:invite
    iam:group:read
    iam:group:create
    iam:group:update
    iam:group:delete
    iam:group:assign_permission
    iam:group:assign_user
    iam:tenant:read
    iam:tenant:create
    iam:tenant:update
    iam:permission:read

  Service: billing
    billing:invoice:read
    billing:invoice:create
    billing:subscription:read
    billing:subscription:update

  Service: reports
    reports:usage:read
    reports:usage:export
    reports:audit:read

  Service: admin (super-admin only)
    admin:tenant:create
    admin:tenant:suspend
    admin:audit:read

  ## JWT Claim Shape Standard

  The canonical claim shape for all 8 implementations:

    {
      "iss": "tenantcore-auth",
      "aud": "tenantcore-api",
      "sub": "<user UUID>",
      "exp": <unix timestamp — max 15 minutes from iat>,
      "iat": <unix timestamp>,
      "nbf": <unix timestamp — same as iat>,
      "jti": "<UUID — unique token ID>",
      "tenant_id": "<tenant UUID or null for super_admin>",
      "tenant_role": "tenant_admin | tenant_member | super_admin",
      "groups": ["<group-slug>", ...],
      "permissions": ["iam:user:read", ...],
      "acl_ver": <int>
    }

  Notes:
  - permissions are embedded for fast request-time checks
  - acl_ver must match users.acl_ver or the token is treated as stale
  - super_admin tokens omit tenant_id (null or absent)
  - groups carries slugs (not IDs) for human-legibility
  - jti enables token replay prevention (optional in smoke tests)
  - signing: RS256 default; HS256 acceptable only in test/dev mode

  ## Canonical Flows

  The following flows must be implemented by all language examples.
  Each flow lists the actor, inputs, expected outcome, and which permissions
  are required.

  ### Flow 1: Issue Token
  Actor: any authenticated user (via username + password or test endpoint)
  Input: user credentials
  Output: signed JWT with claims per shape above
  Permissions: none (public endpoint for credential exchange)
  Notes: In canonical examples, a /token or /auth/token endpoint accepts
         email + password (plaintext in dev fixtures — hash in real systems).

  ### Flow 2: GET /me
  Actor: any valid token holder
  Input: Bearer token
  Output: JSON with sub, tenant context, groups, permissions,
          allowed_routes (filtered route registry), guide_sections
  Permissions: none beyond a valid token
  Error: 401 if token missing or invalid

  ### Flow 3: Super Admin — Create Tenant
  Actor: super_admin
  Input: tenant name, slug, first tenant_admin email
  Output: new tenant record; new user record; membership record with tenant_admin role
  Permissions: admin:tenant:create
  Error: 403 if not super_admin

  ### Flow 4: Tenant Admin — Create Group
  Actor: tenant_admin
  Input: group name, group slug, optional permissions (must be from catalog)
  Output: new group in actor's tenant
  Permissions: iam:group:create
  Error: 403 if not tenant_admin; 400 if permission not in catalog

  ### Flow 5: Tenant Admin — Assign Permission to Group
  Actor: tenant_admin
  Input: group_id, permission_id (must be catalog permission)
  Output: group_permissions record created
  Permissions: iam:group:assign_permission
  Error: 403 if not tenant_admin; 404 if group not in actor's tenant

  ### Flow 6: Tenant Admin — Invite User
  Actor: tenant_admin
  Input: email, display_name, initial group slugs
  Output: new user record, membership, user_group records
  Permissions: iam:user:invite
  Error: 403 if not tenant_admin

  ### Flow 7: Tenant Admin — Assign User to Group
  Actor: tenant_admin
  Input: user_id, group_id (both must be in actor's tenant)
  Output: user_groups record created
  Permissions: iam:group:assign_user
  Error: 403 if not tenant_admin; 404 if user or group not in actor's tenant

  ### Flow 8: Tenant Member — Read Users in Tenant
  Actor: tenant_member with iam:user:read
  Input: Bearer token
  Output: list of users in actor's tenant
  Permissions: iam:user:read
  Error: 403 if missing permission; 401 if invalid token

  ### Flow 9: Cross-Tenant Denial
  Actor: tenant_member from tenant A
  Input: valid token, request targeting tenant B resource
  Output: 403
  Purpose: This is a negative test. It must be implemented as a test in
           every language example.

  ### Flow 10: Stale ACL Version Denial
  Actor: tenant_member with an acl_ver behind current users.acl_ver
  Input: valid (non-expired) token with stale acl_ver
  Output: 401 (token treated as expired due to stale acl_ver)
  Purpose: This negative test verifies authorization versioning works.

  ## Route Metadata Specification

  The route registry is the shared source of truth for all implementations.
  Every implementation must register (at minimum) these routes:

  | Method | Path                         | Permission              | Tenant Scoped | Super Admin Only | Public |
  |--------|------------------------------|-------------------------|---------------|-----------------|--------|
  | POST   | /auth/token                  | (none)                  | false         | false           | true   |
  | GET    | /me                          | (none — valid token)    | true          | false           | false  |
  | GET    | /users                       | iam:user:read           | true          | false           | false  |
  | POST   | /users                       | iam:user:invite         | true          | false           | false  |
  | GET    | /users/{id}                  | iam:user:read           | true          | false           | false  |
  | PATCH  | /users/{id}                  | iam:user:update         | true          | false           | false  |
  | GET    | /groups                      | iam:group:read          | true          | false           | false  |
  | POST   | /groups                      | iam:group:create        | true          | false           | false  |
  | POST   | /groups/{id}/permissions     | iam:group:assign_permission | true     | false           | false  |
  | POST   | /groups/{id}/users           | iam:group:assign_user   | true          | false           | false  |
  | GET    | /permissions                 | iam:permission:read     | true          | false           | false  |
  | GET    | /admin/tenants               | admin:tenant:create     | false         | true            | false  |
  | POST   | /admin/tenants               | admin:tenant:create     | false         | true            | false  |
  | GET    | /health                      | (none)                  | false         | false           | true   |

  Implementations may add additional routes but must implement all of the above.

  ## /me Response Shape

  The exact JSON structure all implementations must return for GET /me:

    {
      "sub": "user-uuid",
      "email": "user@example.com",
      "display_name": "Alice Example",
      "tenant_id": "tenant-uuid",
      "tenant_name": "Acme Corp",
      "tenant_role": "tenant_admin",
      "groups": ["iam-admins", "billing-readers"],
      "permissions": ["iam:user:read", "iam:user:create", "billing:invoice:read"],
      "acl_ver": 3,
      "allowed_routes": [
        {
          "method": "GET",
          "path": "/users",
          "permission": "iam:user:read",
          "description": "List users in the active tenant"
        },
        ...
      ],
      "guide_sections": ["User Management", "Billing"],
      "issued_at": "2025-01-01T00:00:00Z",
      "expires_at": "2025-01-01T00:15:00Z"
    }

  Super admin /me response:
  - tenant_id: null
  - tenant_role: "super_admin"
  - groups: []
  - allowed_routes: includes only /admin/* routes

  ## Fixture Contract

  All implementations load fixture data from
  examples/canonical-auth/domain/fixtures/ at startup (or test time).

  Fixture files are JSON. The fixture set must include:
