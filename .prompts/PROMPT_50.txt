BEHAVIORAL SPECIFICATION

Before reading the sections below, internalize these three rules. Every deliverable
must enforce them.


RULE 1 — INCLUDE OPTION FOR AN EXCLUDED VALUE

When a value V is present in dimension D's exclude set (i.e., V is in `D_out`):

  - The INCLUDE option for V in the filter panel MUST render with count 0.
    Do not compute a non-zero include count for an excluded value. The user
    cannot add V to the include set while it is being excluded; showing a
    non-zero count implies they could. It is misleading.

  - The INCLUDE option MUST be visually greyed out and functionally disabled.
    Apply `opacity-50 cursor-not-allowed` (Tailwind) or equivalent CSS.
    The include checkbox MUST carry the `disabled` attribute.
    Add `data-excluded="true"` to the include option element.

  - Do NOT grey out or zero-out the EXCLUDE option for V.
    The exclude option is the active filter. It tells the user something useful.
    It should remain visually normal and show a non-zero count (see RULE 2).

  - Do NOT hide the include option. The user should see it exists and understand
    why it is unavailable.


RULE 2 — EXCLUDE OPTION COUNT SEMANTICS

The count shown next to an exclude option for value V in dimension D must answer:
"How many rows does this exclusion remove from the current result set?"

Computation:
  - Apply all active filters for all OTHER dimensions fully (both their _in and _out).
  - Apply all active excludes for dimension D EXCEPT the specific value V being counted.
  - Do NOT apply any includes for dimension D.
  - Count rows where row[D] == V.

This is a new computation distinct from the existing include-count/facet-count logic.
Name it clearly in code: `exclude_impact_counts(state, dimension)` or equivalent.

When no excludes are active for D, exclude_impact_counts still gives a meaningful
preview count (how many results this option would remove if you were to exclude it),
which is useful for the quick excludes strip (see RULE 3).

For a value V that is both in _in and _out for the same dimension, resolve it as
PROMPT_31 doctrine specifies (explicit and deterministic). The exclude wins.
The include option for V must be greyed out and zeroed. The exclude option count
uses the RULE 2 computation.


RULE 3 — QUICK EXCLUDES STRIP

The filter panel may optionally include a quick-excludes strip at the top. When
present, the following rules are absolute:

  a) Every quick exclude value MUST also exist as an explicit exclude option inside
     the corresponding dimension's group in the main filter panel below. The quick
     excludes strip is a convenience layer on top of the main filters, not a
     replacement for them.

  b) When a quick exclude is ACTIVE (its value is in the active `D_out`):
     - The quick exclude toggle must visually indicate it is active (checked,
       highlighted, or clearly marked as "on").
     - The corresponding exclude option in the main filter section MUST render
       with a checked checkbox (or equivalent active state marker).
     - Both the quick exclude toggle and the main-section exclude option must
       reflect the same filter state. They are not two separate controls; they
       are two renderings of the same state. Use shared form inputs, shared
       `data-*` attributes, or OOB swaps to keep them in sync.

  c) When a quick exclude is INACTIVE:
     - The quick exclude toggle is unchecked / visually off.
     - The main-section exclude option is unchecked.
     - The exclude impact count (RULE 2) may still be shown as a preview.

  d) The quick excludes strip MUST be declared explicitly in the backend.
     It is NOT auto-derived from the data at query time.
     Define a constant or config like `QUICK_EXCLUDES = [("status", "archived"), ...]`.

  e) The quick exclude toggle element must carry:
       data-role="quick-exclude"
       data-quick-exclude-dimension="<dimension>"
       data-quick-exclude-value="<value>"
     and a visual indicator of active state (e.g., data-active="true/false").

  f) The main-section exclude option element must carry:
       data-filter-option="<value>"
       data-filter-mode="exclude"
       data-filter-dimension="<dimension>"
       data-option-count="<N>"
       data-active="true/false"
     A checked checkbox inside this element (or an equivalent input) reflects
     whether the value is in the active `D_out`.


DATA-* ATTRIBUTE CONTRACT

All filter panel option elements MUST carry the following attributes to allow
Playwright assertions and backend verification:

  Include option:
    data-filter-dimension="<dimension>"
    data-filter-option="<value>"
    data-filter-mode="include"
    data-option-count="<N>"        -- must be 0 when value is excluded (RULE 1)
    data-excluded="true"           -- present only when the value is in D_out

  Exclude option:
    data-filter-dimension="<dimension>"
    data-filter-option="<value>"
    data-filter-mode="exclude"
    data-option-count="<N>"        -- rows this exclude removes (RULE 2)
    data-active="true"             -- present when value is in D_out

  Quick exclude toggle:
    data-role="quick-exclude"
    data-quick-exclude-dimension="<dimension>"
    data-quick-exclude-value="<value>"
    data-active="true/false"

These attributes are the verification contract. Do not omit them. Do not vary
the naming by language or template system. Playwright tests rely on them.


LANGUAGE SCOPE

This capability applies to any stack the repo supports: Python/FastAPI, Elixir/Phoenix,
and any future stacks. The rendering rules and data-* attribute contract are
stack-agnostic. The canonical examples use Python/FastAPI (consistent with PROMPT_31),
but all doctrine and workflow language must be explicitly neutral so future Phoenix
or other examples can follow the same spec.

Add a new workflow:

  context/workflows/render-filter-panel-include-exclude-ui.md
    Purpose: Render a correct filter panel with split include/exclude options
             per dimension, proper visual states, and optional quick excludes.

    When to use it:
    - building any HTMX-served filter panel with include/exclude semantics
    - adding quick excludes to an existing filter panel
    - verifying that a filter panel's visual state matches backend query state

    Sequence:
    1. Render the quick-excludes strip (if configured) at the top of the panel.
       For each configured quick exclude: render a toggle with data-* attrs.
       Show exclude impact count as a preview badge.
    2. For each facet dimension, render a group with a heading and two sub-sections:
       - Include sub-section: one labeled checkbox per option with include count.
         Apply RULE 1: if the value is in D_out, show count=0, add disabled/greyed state.
       - Exclude sub-section: one labeled checkbox per option with exclude impact count.
         Apply RULE 3b: if the value is in D_out, render as checked.
         If this value is also a configured quick exclude, the visual state here
         must mirror the quick exclude toggle.
    3. Ensure all elements carry the required data-* attributes.
    4. Include the filter panel in the same HTMX form as the result list,
       or use hx-swap-oob if the panel updates independently.
    5. Write Playwright assertions verifying at least:
       - an excluded value's include option has data-option-count="0" and data-excluded="true"
       - an excluded value's exclude option has data-active="true" and a non-zero count
       - an active quick exclude's toggle has data-active="true"
       - the corresponding main-section exclude option also has data-active="true"

    Outputs:
    - A backend function that renders a complete include/exclude filter panel fragment.
    - A QUICK_EXCLUDES config constant if quick excludes are used.
    - Playwright tests covering RULE 1, RULE 2, and RULE 3 assertions.

    Related doctrine:
    - context/doctrine/filter-panel-rendering-rules.md
    - context/doctrine/faceted-query-count-discipline.md
    - context/doctrine/filter-state-and-query-state.md

    Common pitfalls:
    - Computing include counts for excluded values using the normal facet_counts path,
      which returns non-zero. Must short-circuit to 0 when value is in D_out.
    - Greying the exclude option in addition to or instead of the include option.
    - Quick exclude toggles using separate hidden inputs that can desync from
      main section checkboxes. Prefer shared inputs or OOB sync.
    - Rendering the quick excludes strip without corresponding entries in the
      main exclude section.
    - Omitting data-excluded or data-active attributes that break assertions.

    Stop conditions:
    - A Playwright test confirms RULE 1 for at least one excluded value.
    - A Playwright test confirms RULE 2: exclude option count is non-zero.
    - A Playwright test confirms RULE 3: quick exclude and main section are in sync.
