SECTION 1 — akka-streams.md SPECIFICATION

File: context/stacks/akka-streams.md

Akka Streams is the Scala streaming library for building reactive, backpressure-aware
stream topologies. It is used in Scala services (especially for the duo-scala-rust and
trio-elixir-go-rust patterns) where a streaming topology must delegate hot-path compute
to an external service via gRPC. This doc covers Akka Streams as a standalone concern —
not as part of an HTTP framework.

Sections:

  ## When Akka Streams Is the Right Choice
  ## When Akka Streams Is NOT the Right Choice
  ## Typical Repo Surface
  ## Core Primitives (Source, Flow, Sink, RunnableGraph)
  ## Materialized Values
  ## Backpressure and Demand Signaling
  ## GraphDSL for Complex Topologies
  ## Connecting to External Services (gRPC, HTTP, Kafka)
  ## Local Dev Composition
  ## Testing Expectations
  ## Common Assistant Mistakes
  ## Related

Content notes:

When Akka Streams IS the right choice:
- Building a multi-stage processing pipeline where each stage can have different concurrency,
  throttling, or parallelism requirements
- Streaming data with backpressure semantics where producers must not overwhelm consumers
- Complex topologies with fan-out (Broadcast), fan-in (Merge, Zip), or cycles (GraphDSL)
- Connecting multiple sources (HTTP, Kafka, file, NATS) into a unified processing graph
- The team already uses Akka/Pekko for actor-based coordination
- Integrating with Alpakka connectors (Kafka, AMQP, AWS S3, JDBC, gRPC)

When Akka Streams is NOT the right choice:
- Simple sequential data processing where ZIO Streams or fs2 would be lighter
- The team is not on Akka/Pekko — introduce ZIO Streams or fs2 for a pure-functional alternative
- The streaming workload is stateless and high-throughput without complex topology — Kafka
  consumer groups may be sufficient
- Scala 3 is the target without legacy Akka Classic — prefer Pekko (Apache fork of Akka)
  or ZIO Streams to avoid Lightbend licensing concerns

Typical repo surface:
  - build.sbt with com.typesafe.akka::akka-stream, com.typesafe.akka::akka-actor-typed
  - src/main/scala/{domain}/Pipeline.scala — source → flow → sink chain
  - src/main/scala/{domain}/StreamGraphs.scala — complex topologies using GraphDSL
  - src/main/scala/{domain}/KernelClient.scala — gRPC client for delegating to Rust kernel
  - src/test/scala/{domain}/PipelineSpec.scala — TestKit-based stream tests

Core primitives:
  Source — produces elements (Source[T, Mat])
  Flow — transforms elements (Flow[In, Out, Mat])
  Sink — consumes elements (Sink[In, Mat])
  RunnableGraph — a connected Source+Sink that can be materialized

  Show a minimal example:
  ```scala
  import akka.actor.ActorSystem
  import akka.stream.scaladsl._

  implicit val system: ActorSystem = ActorSystem("pipeline")

  val source: Source[Int, NotUsed] = Source(1 to 100)
  val flow: Flow[Int, Double, NotUsed] = Flow[Int].map(n => n * 1.5)
  val sink: Sink[Double, Future[Done]] = Sink.foreach(println)

  source.via(flow).runWith(sink)
  ```

Materialized values:
  The third type parameter in Source/Flow/Sink is the materialized value — the
  value produced when the graph is run. Common materialized values:
  - Sink.head → Future[T] (the first emitted element)
  - Sink.fold → Future[U] (accumulated result)
  - Sink.queue → SinkQueueWithCancel (allow imperative pull)
  - Source.queue → SourceQueueWithComplete (allow imperative push)
  When composing Source.via(flow).runWith(sink), the materialized value is that of the Sink
  (right side). Use .toMat(sink)(Keep.both) to preserve both materialized values.

Backpressure:
  Akka Streams implements the Reactive Streams specification. When a downstream operator
  is slower than upstream, it signals demand upstream and the source slows automatically.
  There is no need to manually handle overflow for most use cases.
  Key operators for controlling backpressure:
  - buffer(n, OverflowStrategy.backpressure) — buffer n elements, then signal demand stop
  - throttle(elements, per, burst, mode) — rate limiting with bursting
  - conflate — merge slow-consumer messages using a function (lossy, use with care)

GraphDSL for complex topologies:
  ```scala
  import akka.stream.{ClosedShape, FlowShape}
  import akka.stream.scaladsl.GraphDSL.Implicits._

  val graph = RunnableGraph.fromGraph(GraphDSL.create() { implicit b =>
    val broadcast = b.add(Broadcast[Event](2))
    val merge     = b.add(Merge[ProcessedEvent](2))

    source ~> broadcast
              broadcast ~> fastFlow  ~> merge
              broadcast ~> slowFlow  ~> merge
                                        merge ~> sink
    ClosedShape
  })
  ```

Connecting to external services:

  gRPC with Akka gRPC (Alpakka gRPC):
  Akka gRPC generates Scala stubs from proto files and provides streaming gRPC support.
  Add sbt-akka-grpc plugin. The generated stub integrates naturally with Akka Streams —
  the streaming RPCs return Source[Response, NotUsed] or accept Source[Request, NotUsed].

  Show a flow that calls an external gRPC service for each element:
  ```scala
  // In the streaming pipeline, delegate compute to Rust kernel
  val kernelFlow: Flow[InputRecord, KernelResult, NotUsed] =
    Flow[InputRecord].mapAsync(parallelism = 4) { record =>
      val request = TransformRequest(operation = "normalize", values = record.values)
      kernelClient.transform(request)  // returns Future[TransformResponse]
    }
  ```

  Kafka with Alpakka Kafka:
  ```scala
  // Consumer source
  val kafkaSource: Source[ConsumerMessage.CommittableMessage[String, Array[Byte]], Consumer.Control] =
    Consumer.committableSource(consumerSettings, Subscriptions.topics("input.topic"))

  // Process and commit
  kafkaSource
    .mapAsync(parallelism = 4) { msg => processEvent(msg.record.value()) }
    .map(result => result.committableOffset)
    .runWith(Committer.sink(CommitterSettings(system)))
  ```

Local dev composition:
  Akka Streams services run in JVM containers. Key considerations:
  - JVM startup takes longer than Go/Rust — set start_period: 40s in healthchecks
  - Set JVM heap: -Xmx512m -Xms256m in JAVA_OPTS for dev
  - Use eclipse-temurin:21-jre-alpine or openjdk:21-slim as base image
  - sbt assembly to build a fat JAR for Docker

  ```yaml
  services:
    scala-pipeline:
      build:
        context: ./services/pipeline
        dockerfile: Dockerfile
      ports:
        - "8080:8080"
      environment:
        RUST_KERNEL_ADDR: "rust-kernel:50051"
        JAVA_OPTS: "-Xmx512m"
      depends_on:
        rust-kernel:
          condition: service_healthy
      healthcheck:
        test: ["CMD", "curl", "-f", "http://localhost:8080/healthz"]
        interval: 10s
        timeout: 5s
        retries: 3
        start_period: 40s
  ```

Testing expectations:
- Use AkkaSpec or ScalaTestWithActorTestKit for stream tests
- Prove a Source → Flow → Sink pipeline with a real graph
- Prove backpressure: use Sink.lazyInit or a slow sink and verify the source throttles
- Prove GraphDSL broadcast/merge topologies by asserting element counts
- Prove the gRPC integration: use a test gRPC server (testcontainers or embedded)
- Keep stream tests deterministic: avoid time-based assertions; prefer element count assertions

Common assistant mistakes:
- Creating ActorSystem on every test — create once in test setup (it is expensive to spin up)
- Not materializing the graph — calling .via() or .to() does NOT run the stream;
  call .run() or .runWith(sink) to start it
- Forgetting to shut down ActorSystem in tests — causes resource leaks and hanging test suites
- Using Sink.foreach for production code — it blocks the stream; prefer Sink.foreachAsync
  for async side effects
- Ignoring materialized values — if you want the result of a Sink, capture the materialized value
- Using Akka Classic instead of Akka Typed when starting a new service

- context/stacks/scala-tapir-http4s-zio.md (ZIO HTTP + Tapir for Scala HTTP services)

SECTION 2 — zio-streams.md SPECIFICATION

File: context/stacks/zio-streams.md

ZIO Streams is the streaming subsystem of ZIO 2. It provides functional, type-safe
stream processing with ZIO's effect system integration. Unlike Akka Streams, ZIO Streams
has no actor system overhead — it runs on ZIO's fiber-based runtime. It is the natural
choice for streaming in ZIO-based Scala services (those using scala-tapir-http4s-zio.md).

Sections:

  ## When ZIO Streams Is the Right Choice
  ## When ZIO Streams Is NOT the Right Choice
  ## Typical Repo Surface
  ## Core Primitives (ZStream, ZSink, ZPipeline)
  ## Channel and Hub (fan-out)
  ## Error Handling in Streams
  ## Integration with ZIO HTTP (streaming HTTP endpoints)
  ## Connecting to External Services
  ## Testing Expectations
  ## Common Assistant Mistakes
  ## Related

Content notes:

When ZIO Streams IS the right choice:
- The service is already built on ZIO 2 (using scala-tapir-http4s-zio.md stack)
- Pure functional effect composition is preferred over Akka's imperative mutable state
- Streaming HTTP responses (chunked transfer, server-sent events) from http4s
- No actor system overhead needed — ZIO fiber runtime is lighter than Akka
- The team wants typed error channels throughout the stream (ZStream[R, E, A])
- Integration with ZIO services (ZIO Kafka via zio-kafka, ZIO HTTP)

When ZIO Streams is NOT the right choice:
- The codebase is Akka-based — Akka Streams integrates better with Akka actors/gRPC
- Complex graph topologies (fan-out, cycles) — Akka GraphDSL is more mature for this
- The team needs Alpakka connectors (ZIO ecosystem has zio-kafka, zio-amqp, but
  Alpakka has a larger connector catalog)

Typical repo surface:
  - build.sbt with dev.zio::zio-streams, dev.zio::zio
  - src/main/scala/{domain}/Processor.scala — ZStream pipeline definition
  - src/main/scala/{domain}/Hub.scala — fan-out to multiple fibers via Hub
  - src/main/scala/{domain}/routes/ — ZIO HTTP streaming endpoints
  - src/test/scala/{domain}/ProcessorSpec.scala — ZIO Test stream tests

Core primitives:
  ZStream[R, E, A] — a description of a stream that requires R, may fail with E, produces A
  ZSink[R, E, In, L, Z] — consumes a stream and produces Z (leftover L if partial)
  ZPipeline[R, E, In, Out] — transforms a stream (ZStream → ZStream)

  Minimal example:
  ```scala
  import zio._
  import zio.stream._

  val program: ZIO[Any, Throwable, Unit] =
    ZStream.fromIterable(1 to 100)
      .map(_ * 1.5)
      .tap(n => ZIO.debug(n.toString))
      .run(ZSink.drain)
  ```

  Producing a stream from a resource (queue, database, file):
  ```scala
  ZStream.fromQueue(queue)         // from a ZQueue
  ZStream.fromFile(path)           // from a file
  ZStream.paginateZIO(cursor)(fetchPage)  // from paginated API
  ```

  Pipelines:
  ```scala
  val normalize: ZPipeline[Any, Nothing, Double, Double] =
    ZPipeline.map(x => x / x.max(1.0))

  val pipeline: ZStream[Any, Nothing, Double] =
    ZStream.fromIterable(data) >>> normalize
  ```

Hub (fan-out to multiple subscribers):
  ```scala
  for {
    hub    <- Hub.bounded[Event](capacity = 64)
    _      <- ZStream.fromHub(hub).run(ZSink.foreach(processA)).fork
    _      <- ZStream.fromHub(hub).run(ZSink.foreach(processB)).fork
    _      <- ZStream.fromIterable(events).foreach(hub.publish)
  } yield ()
  ```
  Hub is the ZIO equivalent of Akka Streams Broadcast — all subscribers receive all messages.

Error handling:
  ZStream has a typed error channel E. Errors cause the stream to fail and stop by default.
  Key operators:
  - catchAll: recover from any error, continuing with a new stream
  - retry(Schedule): retry the failing stream on a schedule
  - orElse: switch to a fallback stream on failure
  - ensuring: run a cleanup effect when the stream ends (success or failure)

  ```scala
  ZStream.fromSocketServer(...)
    .catchAll(e => ZStream.logError(e.getMessage) *> ZStream.empty)
  ```

Integration with ZIO HTTP (http4s streaming endpoint):
  ```scala
  // Stream JSON array from http4s endpoint
  import org.http4s.server.Router
  import zio.interop.catz._

  val routes = HttpRoutes.of[Task] {
    case GET -> Root / "events" =>
      Ok(
        ZStream.fromIterable(events)
          .map(e => ServerSentEvent(Some(e.toJson)))
          .toFs2Stream
      )
  }
  ```

Connecting to external services:

  zio-kafka (canonical Kafka integration for ZIO):
  ```scala
  // deps: dev.zio::zio-kafka
  import zio.kafka.consumer._
  import zio.kafka.serde._

  val stream: ZStream[Consumer, Throwable, CommittableRecord[String, String]] =
    Consumer.subscribeAnd(Subscription.topics("input.topic"))
      .plainStream(Serde.string, Serde.string)

  stream
    .mapZIO(record => processEvent(record.value).as(record.offset))
    .aggregateAsync(Consumer.offsetBatches)
    .mapZIO(_.commit)
    .runDrain
  ```

Testing expectations:
- Use ZIO Test and ZTestEnvironment for stream tests
- ZStream.runCollect: materialize a stream to a Chunk[A] for assertion
- Prove fan-out with Hub: verify each subscriber receives all messages
- Prove error recovery: inject a failure into a stream and verify catchAll fires
- Prove streaming HTTP: use ZIO HTTP test client to stream from an endpoint

Common assistant mistakes:
- Treating ZStream as a ZIO — ZStream is a description of a stream, not a single effect;
  use .run(sink) to execute it
- Not using .fork when running multiple streams — blocking on one stream prevents others
- Confusing Hub and Queue: Hub broadcasts to all subscribers; Queue distributes (work queue)
- Not typing the error channel — using ZStream[Any, Nothing, A] masks real errors;
  keep E explicit until the boundary
- Using Throwable as the error type throughout — define domain-specific error types
  and map to Throwable only at the boundary (e.g., HTTP response encoding)

Related:
- context/stacks/akka-streams.md (alternative: Akka Streams for actor-based pipelines)
- context/stacks/scala-tapir-http4s-zio.md (ZIO HTTP + Tapir for the HTTP service layer)
- context/stacks/kafka.md (zio-kafka for Kafka integration)
- context/stacks/duo-scala-rust.md (Scala in multi-backend context)

SECTION 4 — CANONICAL NODE + ELIXIR NATS EXAMPLE

Directory: examples/canonical-multi-backend/duos/node-elixir-nats/

This example demonstrates the NATS broker seam between Node/TypeScript (client connection
layer, publishes client actions) and Elixir (distributed state, processes and rebroadcasts).
It fills the canonical example gap for duo-node-elixir.md.

This is the collaborative app seam: Node connects WebSocket clients and publishes their
actions to NATS; Elixir processes the actions, updates distributed state, and publishes
state updates back through NATS to Node for delivery to clients.

Files:
  node-side.ts      — Node/TypeScript NATS publisher + subscriber (bidirectional)
  elixir-side.ex    — Elixir GenServer with Gnat (bidirectional NATS)
  docker-compose.yml — Full local dev composition
  seam.md           — Companion doc

Note: this example shows the bidirectional NATS seam described in duo-node-elixir.md.
Both sides publish AND subscribe. This is different from the go-elixir-nats duo where
only Go publishes and only Elixir subscribes.


node-side.ts
  - Comment at top: seam example, not full scaffold
  - Uses nats.js (npm: nats@2)
  - Connects to NATS from NATS_URL env var (default "nats://nats:4222")
  - Subscribes to "state.*.updates" — receives state broadcasts from Elixir
  - On startup: publishes one demo action to "actions.sess-demo.cursor.moved" with payload:
      {"payload_version": 1, "session_id": "sess-demo", "correlation_id": "demo-001",
       "published_at": "...", "action_type": "cursor.moved",
       "data": {"x": 100, "y": 200, "doc_id": "doc-abc"}}
  - Logs received state updates from Elixir: event_type, doc_id, data
  - GET /healthz — checks NATS connection status, returns 200 or 503
  - Reads PORT env var (default 3000) for the HTTP endpoint
  - Uses Hono (matching typescript-hono-bun.md conventions) for the health endpoint


elixir-side.ex
  - Comment at top: seam example, not full scaffold
  - Uses Gnat (GenServer)
  - Subscribes to "actions.>" — receives client actions from Node
  - On receiving an action: decodes JSON, logs session_id, action_type, doc_id
  - Publishes a state update to "state.{doc_id}.updates":
      {"payload_version": 1, "doc_id": "doc-abc", "published_at": "...",
       "event_type": "presence.updated",
       "data": {"cursors": [{"session_id": "sess-demo", "x": 100, "y": 200}]}}
  - Implements the bidirectional seam pattern: subscribes to actions, publishes state
  - GET /healthz via Phoenix router returning {"status": "ok"}
  - Reads NATS_URL env var


docker-compose.yml
  - nats service: nats:2.10-alpine with -js flag, ports 4222 and 8222
    - healthcheck: wget --spider http://localhost:8222/healthz
  - node-service: built from node-side.ts, port 3000
    - depends_on: nats (service_healthy)
    - NATS_URL: nats://nats:4222
    - healthcheck: wget --spider http://localhost:3000/healthz
  - elixir-service: built from elixir-side.ex, port 4000
    - depends_on: nats (service_healthy)
    - NATS_URL: nats://nats:4222
    - healthcheck: wget --spider http://localhost:4000/healthz


seam.md
  - Purpose: Node owns client connection layer; Elixir owns distributed state
  - Why bidirectional NATS (not unidirectional): client actions go Node→Elixir;
    state updates go Elixir→Node; both directions use NATS subjects
  - Why not REST for the state updates: Elixir must push updates to Node asynchronously
    (client may be connected but idle); REST would require polling
  - The two subject namespaces: "actions.{session_id}.{type}" (Node→Elixir)
    and "state.{doc_id}.updates" (Elixir→Node)
  - How to run; what to observe: Node logs the action publish; Elixir logs the
    received action; Node logs the received state update
  - The JetStream note: this example uses core NATS (no JetStream), which is sufficient
    for ephemeral real-time coordination. For durable queues, use JetStream.
  - Related docs
