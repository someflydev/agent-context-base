Content notes:

When Kafka IS the right choice:
- Message volumes that benefit from horizontal partition scaling (millions/day threshold)
- Cross-team consumer compatibility requires a schema registry (Avro, Protobuf, JSON Schema)
- Strict ordering guarantees needed within a partition key
- Long-term message retention (days/weeks) needed for replay or audit
- The consumer ecosystem already uses Kafka client libraries
- Multiple independent consumer groups reading the same topic at different offsets

When Kafka is NOT the right choice:
- Lightweight event bus with simple per-subject routing → use NATS JetStream
- In-memory streams with consumer groups on existing Redis infrastructure → use Redis Streams
- Small volume (< 100k messages/day) without schema registry need
- Local dev environment where ZooKeeper/KRaft setup adds friction without benefit
- Real-time request/response patterns → use gRPC or REST
- Ordering not needed and fan-out is simple → NATS is operationally simpler

Topic naming convention:
- Pattern: {domain}.{entity}.{event_type}  (dot-separated, lowercase)
- Examples: payments.orders.created, analytics.users.churned, inventory.items.updated
- Partition key: tenant_id or entity_id (the field that should co-locate in a partition)
- Avoid: action-verb topic names like "process-order" — topics are event logs, not queues

Schema strategy:
- JSON (no registry): fast start, readable, no tooling required. Good for internal
  single-team services. Document the schema in seam-contract/.
- Avro + Schema Registry (Confluent or Apicurio): best for cross-team or cross-service
  compatibility. Schema evolution with backward/forward compatibility guarantees.
  Use confluent_kafka with SchemaRegistryClient.
  Show the Avro schema file format (JSON-encoded .avsc) and registration pattern.
- JSON Schema registry: an intermediate option — human-readable schema with registry
  enforcement. Use when Avro binary encoding is not needed but cross-team schema
  agreement is.

Producer patterns (Python example using confluent-kafka-python):
```python
# pip install confluent-kafka
from confluent_kafka import Producer
import json, time

conf = {
    "bootstrap.servers": os.getenv("KAFKA_BOOTSTRAP_SERVERS", "kafka:9092"),
    "acks": "all",           # strongest durability guarantee
    "enable.idempotence": True,
    "retries": 5,
    "retry.backoff.ms": 200,
}
producer = Producer(conf)

def publish_event(topic: str, key: str, event: dict) -> None:
    payload = json.dumps({
        "payload_version": 1,
        "correlation_id": event.get("correlation_id", ""),
        "published_at": time.strftime("%Y-%m-%dT%H:%M:%SZ", time.gmtime()),
        **event
    }).encode()
    producer.produce(topic, key=key.encode(), value=payload, callback=delivery_report)
    producer.poll(0)

def delivery_report(err, msg):
    if err:
        log.error(f"delivery failed: {err}")
    else:
        log.info(f"delivered to {msg.topic()} [{msg.partition()}] @ offset {msg.offset()}")
```

Consumer patterns (Python example, manual offset commit):
```python
from confluent_kafka import Consumer, KafkaError
import json

conf = {
    "bootstrap.servers": os.getenv("KAFKA_BOOTSTRAP_SERVERS"),
    "group.id": "my-consumer-group",
    "auto.offset.reset": "earliest",
    "enable.auto.commit": False,   # manual commit for at-least-once
}
consumer = Consumer(conf)
consumer.subscribe(["payments.orders.created"])

while True:
    msg = consumer.poll(timeout=1.0)
    if msg is None:
        continue
    if msg.error():
        if msg.error().code() == KafkaError._PARTITION_EOF:
            continue
        raise KafkaException(msg.error())
    event = json.loads(msg.value())
    try:
        process_event(event)
        consumer.commit(asynchronous=False)   # commit only after successful processing
    except Exception as e:
        send_to_dlq(event, str(e))
        consumer.commit(asynchronous=False)   # commit DLQ'd messages too; don't stall
```

Dead letter topics:
- Pattern: {original_topic}.dlq
- Examples: payments.orders.created.dlq
- Send to DLQ when: deserialization fails, schema version unsupported, business rule
  rejects the event, max retries exceeded
- Include in DLQ payload: original message, error message, retry count, failed_at timestamp
- Consumer groups should have a dedicated DLQ monitor service

Testing expectations:
- Run integration tests against a real Kafka broker (Docker-backed bitnami/kafka KRaft)
- Prove one produce + consume round-trip with a real message
- Prove DLQ delivery for a malformed message
- Test idempotent re-delivery: produce the same message twice, verify consumer processes once
  (or handles the duplicate correctly)
- Do NOT mock the Kafka client in integration tests — the broker behavior is what's tested
- Keep test topics isolated per test run using a unique group.id suffix or unique topic names

Common assistant mistakes:
- Using auto.commit=true in consumer code — this commits before processing is confirmed,
  leading to message loss on crash. Always use manual commit.
- Creating topics at message time inside the consumer — create topics at startup with
  explicit partition/replication config, not auto-create on first consume.
- Naming topics as verbs ("process-payment") instead of event log nouns ("payments.transactions.created")
- Reusing consumer group IDs across environments — dev and test must have separate group IDs
- Ignoring DLQ: assuming all messages will process successfully without a dead-letter path
- Using confluentinc/cp-kafka in docker-compose without a schema registry when Avro is planned

SECTION 2 — kafka-trino.md SPECIFICATION

File: context/stacks/kafka-trino.md

This is the combo pack for using Kafka as a Trino-queryable data source.
Trino's Kafka connector maps topic messages to queryable table rows. This
is useful for ad-hoc stream analytics, debugging topic contents, and joining
Kafka topic data with relational or object-store sources in Trino.

Cover:

  ## Kafka Catalog Config (trino/etc/catalog/kafka.properties)
  ## Topic-to-Table Mapping (topic table definition JSON files)
  ## Schema File Format
  ## Example Queries
  ## Local Dev Composition
  ## Limitations and Anti-Patterns
  ## Related

Content notes:

When to use this combo:
- Joining Kafka topic data with Postgres or MongoDB tables in a single Trino query
  (e.g., enrich Kafka payment events with user records from Postgres)
- Ad-hoc analytics over Kafka topics without building a consumer pipeline
- Debugging and inspecting topic contents at SQL level
- BI tools that speak SQL can query Kafka data without a dedicated consumer

Architecture — how the Kafka connector works:
- The Kafka connector maps each Kafka topic to a Trino table
- Topic messages are decoded at query time (not indexed) — no consumer group
- Each row is one Kafka message: the connector exposes built-in columns
  (_partition_id, _partition_offset, _message_key, _timestamp) plus columns
  defined in a topic table definition file
- Trino reads all partitions in parallel — scans can be expensive for large topics
- Messages are decoded using the configured decoder (JSON, Avro, CSV)

Kafka catalog config file (etc/catalog/kafka.properties):
```
connector.name=kafka
kafka.nodes=kafka:9092
kafka.table-names=payments.orders.created,analytics.users.churned
kafka.hide-internal-columns=false
kafka.default-schema=default
```

Topic table definition (etc/kafka/payments.orders.created.json):
This file maps the JSON fields of a Kafka message to Trino column definitions.
Show the full format:
```json
{
  "tableName": "payments.orders.created",
  "schemaName": "default",
  "topicName": "payments.orders.created",
  "message": {
    "dataFormat": "json",
    "fields": [
      {"name": "payload_version", "mapping": "payload_version", "type": "BIGINT"},
      {"name": "correlation_id", "mapping": "correlation_id", "type": "VARCHAR"},
      {"name": "published_at", "mapping": "published_at", "type": "VARCHAR"},
      {"name": "tenant_id", "mapping": "tenant_id", "type": "VARCHAR"},
      {"name": "entity_id", "mapping": "entity_id", "type": "VARCHAR"},
      {"name": "risk_score", "mapping": "data/risk_score", "type": "DOUBLE"},
      {"name": "risk_tier", "mapping": "data/risk_tier", "type": "VARCHAR"}
    ]
  }
}
```

Note: the "mapping" field uses "/" for nested JSON path navigation.
Built-in columns (_partition_id, _partition_offset, _message_key, _timestamp) are
always available when hide-internal-columns=false.

Example queries:
- Simple topic scan:
  ```sql
  SELECT correlation_id, tenant_id, risk_score, _timestamp
  FROM kafka.default."payments.orders.created"
  WHERE risk_tier = 'high'
  ORDER BY _timestamp DESC
  LIMIT 100;
  ```

- Cross-source join (Kafka topic + Postgres table):
  ```sql
  SELECT k.entity_id, k.risk_score, k.risk_tier,
         p.user_email, p.account_age_days
  FROM kafka.default."payments.orders.created" k
  JOIN postgresql.app.users p ON k.entity_id = p.order_id
  WHERE k.published_at >= '2026-03-01'
    AND k.risk_tier = 'high'
  ORDER BY k.risk_score DESC;
  ```

- Partition offset inspection:
  ```sql
  SELECT _partition_id, MIN(_partition_offset) AS min_offset,
         MAX(_partition_offset) AS max_offset, COUNT(*) AS message_count
  FROM kafka.default."payments.orders.created"
  GROUP BY _partition_id;
  ```

Local dev composition — show how to add the Kafka catalog to the existing Trino setup:
```yaml
services:
  kafka:
    # bitnami/kafka with KRaft (see kafka.md)
    ...
  trino:
    image: trinodb/trino:446
    ports:
      - "8085:8080"
    volumes:
      - ./etc/catalog:/etc/trino/catalog
      - ./etc/kafka:/etc/trino/kafka   # topic table definition files
    depends_on:
      kafka:
        condition: service_healthy
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/v1/info"]
      interval: 15s
      timeout: 10s
      retries: 5
      start_period: 30s
```

Limitations and anti-patterns:
- The Kafka connector scans the entire topic on every query — it is NOT suitable for
  high-frequency queries on large topics. Use it for ad-hoc analysis, not a hot path.
- No filter pushdown to Kafka: Trino reads all messages and filters locally.
  The _partition_offset range filter is the only available scan optimization.
- Message ordering: Trino does not guarantee row order matches Kafka partition ordering
  unless you explicitly ORDER BY _partition_id, _partition_offset.
- Avro-encoded topics: the JSON decoder will not work on Avro messages. The Avro decoder
  requires the Confluent Schema Registry. Document this clearly when using Avro.
- Do not use Kafka+Trino for operational dashboards with SLA latency requirements.
  Use it for analytical exploration and cross-source joins.

    just log the failure clearly)
