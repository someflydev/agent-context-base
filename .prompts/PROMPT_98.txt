================================================================================
BACKGROUND: WHY HEURISTIC ROUTING
================================================================================

The repo already has routers (task-router.md, stack-router.md, archetype-router.md)
that an ASSISTANT reads during boot to decide what to load. Those are normative docs —
guidance for humans and assistants.

What's missing is an executable tool that can take a prompt text and produce a
structured report of:
  - what archetype(s) the prompt likely implies
  - what primary stack it suggests
  - what capabilities are implicated (from profile-rules.json)
  - what budget profile the implied context would require
  - warnings if the prompt is likely "juicy" (multi-stack, multi-capability)

This tool (`work.py route-check`) does NOT replace the routers. It is a heuristic
assistant that helps operators ask "is this repo well-configured to handle this prompt,
and what should the assistant load?" It is useful for:
  1. Pre-session planning: what will this prompt actually require?
  2. Post-session validation: did the session declare loading what route-check expected?
  3. Repo evaluation: is the repo actually leveraging its capabilities when given
     complex prompts that should activate multiple stacks/workflows?
  4. Testing the routing infrastructure: do the canonical juicy-prompt scenarios produce
     stable routing suggestions?

ALL output must be labeled HEURISTIC. The tool never claims certainty. It proposes.

  "postgres", "postgresql", "mysql", "sqlite", "database", "schema" -> storage
  "redis", "keydb", "cache", "caching"                              -> storage

  "qdrant", "meilisearch", "vector", "embedding", "rag", "search"   -> rag, search
  "celery", "worker", "queue", "background task", "scheduler"       -> workers
  "kafka", "rabbitmq", "event", "stream", "pubsub"                  -> eventing
  "pipeline", "etl", "transform", "ingest"                          -> pipelines
  "scrape", "scraping", "crawler", "crawl", "fetch"                 -> scraping

------------------------------------------------------------------------
1.2 Capability-to-archetype mapping
------------------------------------------------------------------------

Read the `archetype_capabilities` mapping from `profile-rules.json` at runtime.
Invert it: for each capability tag, which archetypes support it?

  capability -> [archetype1, archetype2, ...]

Use this to infer candidate archetypes from the prompt's implied capabilities.

Expose: `load_capability_to_archetype_map(repo_root: Path) -> dict[str, list[str]]`

------------------------------------------------------------------------
1.3 Capability-to-manifest mapping
------------------------------------------------------------------------

Similarly, invert `manifest_capabilities` from profile-rules.json:

  capability -> [manifest1, manifest2, ...]

Expose: `load_capability_to_manifest_map(repo_root: Path) -> dict[str, list[str]]`

------------------------------------------------------------------------
1.4 Route inference
------------------------------------------------------------------------

  @dataclass
  class RouteInference:
      implied_capabilities: tuple[str, ...]     # deduplicated, sorted
      candidate_archetypes: tuple[str, ...]      # ranked by match count
      candidate_manifests: tuple[str, ...]       # ranked by match count
      primary_archetype: str | None              # highest-ranked archetype
      primary_manifest: str | None               # highest-ranked manifest
      is_multi_capability: bool                  # True if >= 3 capability tags
      is_likely_juicy: bool                      # True if capabilities span 3+ domains
      suggested_budget_profile: str              # tiny/small/medium/large/cross-system
      confidence: float                          # 0.0-1.0 (based on keyword match density)
      warnings: tuple[str, ...]                  # routing concerns
      keyword_hits: dict[str, list[str]]         # keyword -> [capability tags] for transparency

Profile suggestion heuristic:
  1 capability tag  → tiny or small
  2 capability tags → small or medium
  3 capability tags → medium or large
  4+ capability tags or is_likely_juicy → large or cross-system

Domain categories for is_likely_juicy:
  data-domain:       storage, rag, pipelines
  infra-domain:      workers, eventing, cloud-deployment
  interface-domain:  api, cli
  quality-domain:    verification

is_likely_juicy = True if capabilities span 3+ domains OR include both data-domain and
interface-domain and cloud-deployment.

Confidence heuristic:
  strong (0.85-1.0):    >= 5 keyword hits with consistent capability signals
  usable (0.70-0.84):   3-4 keyword hits, mild diversity
  weak (0.55-0.69):     1-2 keyword hits or high diversity
  very weak (< 0.55):   0 keyword hits or highly ambiguous

Expose: `infer_route(prompt_text: str, repo_root: Path) -> RouteInference`

------------------------------------------------------------------------
1.5 Suggested context files (heuristic)
------------------------------------------------------------------------

Given a RouteInference, suggest a minimal context bundle to load. This is the core of
"capability validation" — comparing what an assistant declared it loaded against what
route-check suggests it should have loaded.

  def suggest_context_bundle(
      inference: RouteInference,
      repo_root: Path,
  ) -> list[str]:
      """Return a heuristic list of recommended file paths for this route."""

Logic:
  - Always include: AGENT.md
  - Include primary manifest file if it exists in manifests/
  - Include archetype file if it exists in context/archetypes/
  - Include primary workflow file(s) implied by capability (use a small
    capability-to-workflow map: api -> add-api-endpoint, storage -> add-storage-layer,
    pipelines -> add-data-pipeline, cli -> add-cli-command, etc.)
  - Include at most 2 stack files from context/stacks/ matching primary manifest/stack
  - Include core doctrine: context/doctrine/context-complexity-budget.md always
  - If cloud-deployment in capabilities: include deployment doctrine
  Only suggest files that actually exist in the repo (check Path.exists()).

PART 2: work.py route-check subcommand
================================================================================

Add `route-check` to `scripts/work.py`.

  python3 scripts/work.py route-check "<prompt text>"
  python3 scripts/work.py route-check @path/to/prompt.txt
  python3 scripts/work.py route-check --json "<prompt text>"
  python3 scripts/work.py route-check --compare-bundle path1 path2 ... "<prompt text>"

Argument handling:
  - If first positional arg starts with @: read prompt text from that file path
  - Otherwise: use the positional string as prompt text directly
  - --json: output JSON instead of human-readable
  - --compare-bundle: after routing, score the provided bundle and compare against
    the suggested bundle (useful for validating that a declared trace loaded what
    was recommended)

Human-readable output format:

  Route Check (HEURISTIC)
  =======================
  Prompt:          "<first 80 chars of prompt>..."

  Implied Capabilities:
    api, storage, verification  (from keywords: endpoint, postgres, smoke test)

  Candidate Archetypes:
    1. backend-api-service    (matches: api, storage)
    2. data-pipeline          (matches: storage)

  Candidate Manifests:
    1. backend-api-fastapi-polars  (matches: api, storage)

  Routing Decision (HEURISTIC):
    Primary Archetype:  backend-api-service
    Primary Manifest:   backend-api-fastapi-polars
    Budget Profile:     medium
    Confidence:         strong (0.90)

  Suggested Context Bundle:
    - AGENT.md
    - manifests/backend-api-fastapi-polars.yaml
    - context/archetypes/backend-api-service.md
    - context/workflows/add-api-endpoint.md
    - context/doctrine/context-complexity-budget.md
    (+ 2 more stack files)

  [if --compare-bundle provided:]
  Bundle Comparison:
    Declared bundle score:   <N> points (<profile>)
    Suggested bundle score:  <N> points (<profile>)
    Files in suggested not in declared: <list | none>
    Files in declared not in suggested: <list | none>
    Verdict: <"declared bundle aligns with routing" | "declared bundle under-loaded"
              | "declared bundle over-loaded for this route">

  Warnings:
    - [none | e.g., "cross-domain: deployment capability implied but no trigger detected"]
    - [HEURISTIC: this is not a verified route, only a keyword-based suggestion]

JSON output must include all the same fields in a structured format.

Always include "HEURISTIC" prominently in both output formats.
