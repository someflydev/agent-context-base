================================================================================
BACKGROUND: WHAT GRAFT IS
================================================================================

The ACB workflow (work.py, memory/, runtime context files, commit discipline,
CLAUDE.md boot sequence) has until now only been available to repos that start
from agent-context-base. There is no way to bring this discipline to a repo that
already exists and is being actively worked on.

`work.py graft` solves this. It takes a path to any locally cloned repo and
installs the portable ACB discipline layer into it. It does NOT copy stack
definitions, archetypes, canonical examples, verification suites, or any
content specific to agent-context-base. It installs only the infrastructure
that makes assistants faster and more traceable regardless of what the repo does.

After graft, an operator can open the target repo in any Claude Code session and
immediately get work.py resume output, session state tracking, memory promotion,
and a commit convention. The `.prompts/analyze-and-reverse-engineer.txt` file that
graft installs gives the operator (or their assistant) a ready-made first prompt
that will analyze the target repo and produce a real, high-quality promptset for
the work being planned there.

1.2  handle_graft implementation
--------------------------------------------------------------------------------

Implement `handle_graft` with this exact behavior:

  def handle_graft(
      repo_root: Path,
      *,
      target: Path,
      dry_run: bool,
      force: bool,
      no_prompts: bool,
  ) -> int:

Print header: "ACB Graft"

Step A — Validate the target:
  - Resolve target to an absolute path.
  - If it does not exist or is not a directory: print an error, return 1.
  - Check if target/.git exists. If not, print a warning (not an error):
      "Warning: target does not appear to be a git repository. Continuing anyway."
  - Check that target != repo_root. If they are equal, print an error and return 1:
      "Error: target is the agent-context-base repo itself."

Step B — Build the graft manifest:
  Collect a list of GraftItem objects. A GraftItem has:
    - dest: Path  (relative to target)
    - content: str | None  (if None, the item is a directory marker)
    - reason: str  (human-readable explanation, printed in dry-run and verbose output)

  The manifest is built by the helper `build_graft_manifest(repo_root, target, no_prompts)`.
  See Section 1.3 for exact items.

Step C — Report planned actions:
  Print the list of items, grouped by whether they already exist in the target:
    - "Will write (new):"  — items whose dest does not exist in target
    - "Will write (overwrite):" — items whose dest exists and --force is set
    - "Will skip (already exists):" — items whose dest exists and --force is NOT set
  If dry_run, print "Dry run — no files were written." and return 0.

Step D — Execute:
  For each GraftItem:
    - If dest is a directory marker (content is None):
        Create the directory (parents=True, exist_ok=True), plus a .gitkeep if empty.
    - If dest already exists and not force: skip (already reported).
    - Otherwise: write content to dest (create parent directories as needed).
  Print a confirmation line for each file written.

Step E — Print .gitignore guidance:
  After writing, print:

    Graft complete. Add these lines to {target}/.gitignore if not already present:

      work/
      PLAN.md
      context/MEMORY.md

  Do not modify the target's .gitignore automatically. This is the operator's decision.

Step F — Print next steps:
  Print:

    Next steps:
      1. Review the grafted files in {target}
      2. Add the .gitignore entries above
      3. Commit the grafted files: git add -p && git commit
      4. Open a new session in {target} and run:
           python3 scripts/work.py resume
      5. Then run the analysis prompt:
           cat .prompts/analyze-and-reverse-engineer.txt

  (Omit step 5 if --no-prompts was set.)

Return 0 on success.

--------------------------------------------------------------------------------
Design principles for this prompt
--------------------------------------------------------------------------------

1. It is a GUEST in the target repo. It should read aggressively and write
   conservatively. Reading every file it needs is expected. Creating files that
   conflict with existing ones is not.

2. It produces REAL artifacts, not plans to make artifacts. By the end of the
   session, there must be at minimum: a PLAN.md, a populated context/TASK.md,
   and at least one real .prompts/PROMPT_01.txt.

3. It models the domain, not just the tech. A repo that does payroll processing
   deserves a promptset rooted in payroll concepts, not just "add tests" or
   "refactor." The analysis must surface what the software is actually for.

4. It is honest about uncertainty. If the repo's purpose is unclear after reading,
   the prompt instructs the assistant to say so and ask the operator before
   generating prompts.

5. It is calibrated to scope. A 200-file monorepo needs more prompts than a
   500-line CLI. The assistant should generate as many prompts as makes sense
   (neither underfitting with 2 nor padding with 30).

================================================================================
PHASE 2 — PROMPTSET DESIGN
================================================================================

Design a promptset. A promptset is a sequence of numbered prompt files, each
representing one focused session of assistant work. Design it with these rules:

SESSION SIZING
  Each prompt should be completable in one session (roughly 30–120 minutes of
  real work, or 1–5 meaningful commits). If a task is too large, split it.
  If two tasks are too small individually, merge them.

ORDERING
  Prompts must be ordered so that each builds on what the previous established.
  Foundation before features. Hardening after implementation. Documentation after
  both. Do not put documentation before there is something to document.

DOMAIN FIRST
  The first prompt after any necessary setup should establish the domain model:
  the core entities, their relationships, and the data shapes they produce.
  Even if models already exist in the repo, documenting them in a domain spec
  makes every subsequent prompt faster and more consistent.

HONEST SCOPE
  Generate as many prompts as the repo actually needs. A healthy range is 5–15.
  Do not pad. Do not compress. If the repo needs 3 prompts, generate 3.
  If it needs 20, generate 20.

NAMING
  Name each prompt by what it produces, not by what it touches.
  Good: "Add pagination and filtering to the jobs API"
  Bad:  "Work on routes.py"

IMPLEMENTATION NOTES
================================================================================

- The `build_graft_manifest` function reads the analyze-and-reverse-engineer.txt
  template file at call time from repo_root. If the file does not exist, raise
  a clear RuntimeError: "Graft template not found: {path}. Re-clone agent-context-base."

- The stub constants (MEMORY_INDEX_STUB, CONTEXT_TASK_STUB, etc.) must be defined
  as module-level string literals, not inline in the function. This keeps handle_graft
  readable and makes the stubs easy to update later.

- For the directory marker items (content=None), write only a .gitkeep if the
  directory is empty after creation. Do not overwrite existing .gitkeep files.

- The graft command must work when called from inside agent-context-base (i.e.,
  repo_root is agent-context-base and target is some other path). The detect_repo_root()
  function always returns agent-context-base's root — that is fine. Target resolution
  is fully independent of repo_root.

- Do not add import shutil or any new imports unless strictly necessary.
  Use Path.read_text() and Path.write_text() for all file I/O.
  Path.mkdir(parents=True, exist_ok=True) for directories.

- The analyze-and-reverse-engineer.txt template must be written as plain text, not
  wrapped in any outer format. The file is the prompt. An assistant opening it
  should be able to paste it directly into a session.
