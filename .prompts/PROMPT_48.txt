The `data-acquisition-invariants.md` doctrine says: "source adapters stay narrow
and source-specific instead of leaking upstream field oddities into canonical
models." `tag_name` and `html_url` are GitHub-specific. `external_slug` and
`canonical_url` are source-agnostic. The four-language group has the right names;
the nine-language group has alignment debt.

In a polyglot system where records from multiple language services feed a
shared downstream consumer (search index, aggregation service, report store),
Nim produces `{"external_slug": "v1.2.0"}` while Go produces
`{"tag_name": "v1.2.0"}` for the same semantic concept. The consumer must handle
two schemas for one concept.

Your task is to align the nine Group A languages so that `NormalizedReleaseRecord`

IMPORTANT CONSTRAINTS

- The raw GitHub API access (reading `item.get("tag_name")`, `item["html_url"]`
  from the upstream JSON response) must NOT change. GitHub returns `tag_name`
  and `html_url`. The source adapter reads those field names from the raw payload
  and maps them into the canonical model. The mapping line stays correct — only
  the canonical model field name changes.
- Language-idiomatic casing rules apply to the field names:
  - Python, Go, Rust, OCaml, Crystal: snake_case → `external_slug`, `canonical_url`
  - Go struct fields: PascalCase → `ExternalSlug`, `CanonicalURL`; JSON tag: `external_slug`, `canonical_url`
  - TypeScript, Kotlin, Dart: camelCase → `externalSlug`, `canonicalUrl`
  - Ruby symbols: `:external_slug`, `:canonical_url`
  - Clojure keywords: `:external-slug`, `:canonical-url` (Clojure idiomatic kebab-case)
  - SQL column names (Kotlin Exposed, Clojure next.jdbc, Ruby Sequel): `external_slug`, `canonical_url`
- Do not change any other field in `NormalizedReleaseRecord`. Only rename the
  two fields specified: the tag/slug field and the URL field.
- `canonical_url` is the right replacement for all of: `html_url`, `htmlUrl`,
  `:html-url`, `release_url`, `releaseUrl`, `canonicalUrl`. The downstream
  semantic is "the canonical external URL for this release."
- Do not change any verification test that checks for `tag_name` or `html_url`
  as *input* data (GitHub API response mock keys). Those are correct — GitHub
  returns those field names. Only update tests that assert `tag_name` or
  `html_url` as *output* field names on `NormalizedReleaseRecord`.
- After reading each source file, check whether there are verification tests
  for that language that assert field names on `NormalizedReleaseRecord`. Update
  those tests to assert the new names.
