================================================================================
BACKGROUND: THE JWT AUTH, RBAC, AND MULTI-TENANT PATTERNS ARC
================================================================================

agent-context-base has canonical examples for API services, CLI tools, TUI
applications, data pipelines, observability, schema validation, synthetic data
generation, and analytics visualization. It does not yet have a canonical area
for a concern that underpins nearly every production API: authentication,
role-based access control, and multi-tenant request scoping.

This arc adds first-class status to JWT-backed auth patterns across 8 backend
language ecosystems. The canonical examples are constrained and teachable:
each demonstrates enough to understand the whole pattern without becoming a
full product build.

THE CORE INSIGHT THIS ARC MUST PRESERVE:
  JWT tokens are short-lived snapshots of authorization state. The database
  is the source of truth. Tokens carry effective permissions for fast
  request-time checks, but the auth system must be able to invalidate stale
  tokens by version-checking the authorization state. Route metadata — not
  string literals on routes — is the durable permission namespace. The /me
  endpoint is not a convenience endpoint; it is the discoverability surface
  that makes auth self-documenting.

KEY NON-NEGOTIABLES for the arc:
  - permission naming: {service}:{resource}:{action} — never raw URL paths
  - database-backed source of truth; token is a short-lived snapshot
  - asymmetric signing preferred where the library supports it easily
  - /me endpoint is first-class, not an afterthought
  - route metadata enables docs, enforcement, and guide generation
  - tenant-admin composes from a platform-defined permission catalog;
    tenant-admins do NOT define new permission atoms
  - super-admin is a global construct, never a tenant-local role

A. Create: context/doctrine/jwt-auth-request-context.md
--------------------------------------------------------------------------------

This doctrine defines rules for JWT validation and request-time auth context.

Title: JWT Auth and Request Context

Required sections:

  ## Core Principle
  3–4 sentences. A JWT token is a short-lived, signed snapshot of
  authorization state. The database is the source of truth. The token
  accelerates request-time checks. Stale tokens must be detectable via
  authorization version tracking (acl_ver or equivalent).

  ## Rules  (number each rule)

  Rule 1 — Validate before trusting.
    Every protected endpoint must validate the token signature, expiry,
    issuer, and audience before reading claims. Libraries that make this
    opt-in are dangerous defaults. The canonical examples always validate
    all four.

  Rule 2 — Asymmetric signing is preferred.
    Use RS256 or ES256 when the signing key can be rotated independently
    of verification. HS256 is acceptable for single-service deployments
    but must be documented as a deliberate constraint, not a default.

  Rule 3 — Short-lived access tokens only.
    Access token expiry must be ≤15 minutes in canonical examples. Longer
    expiry must be justified in inline comments. Refresh token patterns
    are noted but out of scope for canonical auth examples.

  Rule 4 — Standard claims are not optional.
    Every token must carry: iss, aud, sub, exp, iat, nbf, jti.
    Missing any of these is a defect, not a simplification.

  Rule 5 — Auth context is request-scoped, not global.
    Auth state must be attached to the request context (middleware result,
    request object, or equivalent per language). Global mutable auth state
    is forbidden. In frameworks with dependency injection, auth context
    must flow through the DI container, not a thread-local or process global.

  Rule 6 — Authorization version enables fast invalidation.
    Tokens should carry an acl_ver claim matching the user's current
    authorization version. If acl_ver in the token does not match the
    stored version, the token is treated as expired regardless of exp.
    Increment acl_ver on any group/permission/membership change.

  Rule 7 — Token claims are a snapshot, not a live query.
    Embedded permissions are for fast request-time checks. Sensitive or
    high-stakes operations should still verify against the database.
    Document which endpoints perform a live check vs. token-only.

  Rule 8 — Tenant context must be explicit.
    For tenant-scoped tokens, tenant_id must be validated on every request.
    An endpoint that accepts any valid token without checking tenant context
    is a cross-tenant access bug, not a feature.

  ## Anti-Patterns
  A bulleted list of 4–5 things to avoid (long-lived tokens, HS256 in
  multi-service, skipping nbf/jti, global auth state, raw URL-path strings
  as permission names).

  ## Canonical Library Choices
  Brief table: language → preferred JWT library → why → adjacent alternative.
  Include all 8 languages. See PART 2 for library rationale.

B. Create: context/doctrine/rbac-permission-taxonomy.md
--------------------------------------------------------------------------------

Title: RBAC and Permission Taxonomy

Required sections:

  ## Core Principle
  The canonical permission namespace uses the form {service}:{resource}:{action}.
  This namespace is owned by the platform, not by tenant-admins. Tenant-admins
  compose groups from a catalog of available atoms. They do not define new atoms.

  ## Rules

  Rule 1 — Permission atoms follow {service}:{resource}:{action}.
    Examples: iam:user:read, iam:group:create, billing:invoice:read,
    reports:usage:export. Actions are expressive verbs close to CRUD but
    not restricted to it. Never use raw URL paths as permission identifiers.

  Rule 2 — The permission catalog is platform-owned.
    The platform defines what permissions exist. Tenant-admins assign
    permissions to groups. They cannot create permissions that are not
    in the platform catalog. This prevents permission sprawl and privilege
    escalation through creative naming.

  Rule 3 — RBAC works through group assignment.
    Users get permissions because they are in groups. Groups have permission
    sets. Users are added to groups by tenant-admins. Direct user-permission
    assignment is not in the canonical model (it creates policy maintenance debt).

  Rule 4 — Super-admin is a global role, not a tenant role.
    Super-admin capabilities (create tenants, assign tenant-admins, inspect
    cross-tenant state) must be modeled as a global role. Never add
    super-admin logic inside the tenant-scoped permission check path.

  Rule 5 — Effective permissions are computed, not stored raw.
    The user's effective permissions = union of all permissions from all
    groups the user belongs to within their active tenant. Compute this at
    token issuance or on demand. Do not store a flat list permanently; recompute
    when group memberships change.

  Rule 6 — Hierarchical role labels are supplemental, not authoritative.
    tenant_role (e.g., "admin", "member") in the JWT is a convenience label.
    Authorization decisions must use the permission list, not the role label.
    Role labels help UI hints and logging but must never be the sole gate.

  Rule 7 — Permission coverage must be verifiable.
    Route metadata (see route-metadata-registry.md) enables a check that
    every protected endpoint has a corresponding permission atom. Missing
    coverage is a defect, not acceptable drift.

  ## Anti-Patterns
  4–5 things to avoid: URL-path permissions, tenant-defined atoms,
  direct user permissions bypassing groups, role-only gates, unchecked
  permission catalog drift.

  ## Canonical Permission Catalog (example)
  A small table with at least 10 example permissions across iam, billing,
  and reports services to illustrate the naming convention concretely.

Target length: 70–90 lines.

C. Create: context/doctrine/tenant-boundary-enforcement.md
--------------------------------------------------------------------------------

Title: Tenant Boundary Enforcement

Required sections:

  ## Core Principle
  Every resource in a multi-tenant system belongs to exactly one tenant.
  The token's tenant_id claim scopes all operations. No endpoint may read
  or write resources from a different tenant based on the authenticated
  token — this must be enforced structurally, not by convention.

  ## Rules

  Rule 1 — tenant_id must be validated at every request boundary.
    Middleware that extracts auth context must verify the token's tenant_id
    matches the resource being accessed. Skipping this check for "internal"
    endpoints is a historical source of real-world breaches.

  Rule 2 — Tenant context flows via the request context, not query params.
    tenant_id must come from the validated JWT claim, not from a URL
    parameter or request body field. Accepting tenant_id from the caller
    allows tenant impersonation.

  Rule 3 — Super-admin calls are explicitly distinguished.
    A super-admin request that accesses cross-tenant data must go through
    a separate, explicitly super-admin-gated code path. The tenant-user
    and tenant-admin path must never include a "unless super-admin" branch.
    Model these as separate handlers or a clearly-labeled bypass token.

  Rule 4 — Tenant isolation in data access must be structural.
    Database queries for tenant-scoped data must include a WHERE tenant_id = ?
    clause derived from the validated auth context, not from the request body.
    ORMs must be instrumented to ensure this clause is never missing.

  Rule 5 — Membership must be verified, not assumed.
    Possessing a valid token for tenant X does not automatically mean the user
    is an active member of tenant X. Membership status should be verified
    from the database on sensitive operations. Token may carry membership
    snapshot for speed; always version-check it.

  Rule 6 — Audit every cross-tenant boundary touch.
    Super-admin operations touching multiple tenants must produce audit events.
    Log: acting principal, target tenant_id, action, timestamp, outcome.

  ## Anti-Patterns
  4–5 items: tenant_id from request body, implicit tenant from URL,
  super-admin branches inside tenant paths, unscoped DB queries,
  trusting membership without version checks.

Target length: 60–80 lines.

D. Create: context/doctrine/route-metadata-registry.md

Title: Route Metadata Registry

Required sections:

  ## Core Principle
  Every protected route must have a machine-readable metadata record that
  names its required permission, tenant scope, and documentation context.
  Route metadata is the durable contract between authorization policy and
  implementation. Enforcement, docs generation, /me output, and coverage
  checks all read from this single source.

  ## Canonical Metadata Fields
  A table:
  | Field             | Type    | Required | Description                             |
  | method            | string  | yes      | HTTP method (GET, POST, etc.)           |
  | path              | string  | yes      | Route path pattern                      |
  | permission        | string  | yes      | Required permission atom                |
  | tenant_scoped     | bool    | yes      | Whether tenant_id is enforced           |
  | description       | string  | yes      | Human-readable description              |
  | service           | string  | yes      | Owning service name                     |
  | resource          | string  | yes      | Resource type                           |
  | action            | string  | yes      | Action verb                             |
  | htmx_target       | string  | no       | HTMX target selector (if applicable)   |
  | docs_section      | string  | no       | Docs/guide section name                 |
  | super_admin_only  | bool    | no       | Only accessible to super-admin          |
  | public            | bool    | no       | No auth required (explicitly opted in) |

  ## Rules

  Rule 1 — Metadata is the single source of truth for route auth requirements.
    Middleware that enforces auth must read from the route metadata registry,
    not from inline annotations that can drift. If metadata says
    iam:user:read is required, the middleware checks that claim.

  Rule 2 — Every protected route has exactly one required permission.
    Routes requiring multiple permissions simultaneously are a design smell.
    Model a higher-level permission atom instead.

  Rule 3 — Public routes are explicitly declared, not implicitly assumed.
    A route without a metadata record is a defect, not a public endpoint.
    Public endpoints must be listed in the registry with public: true.

  Rule 4 — Route metadata powers /me output.
    The /me endpoint must filter the route registry to return only the
    routes accessible to the calling user based on their effective permissions.
    This is not a separate list; it is derived from the same registry.

  Rule 5 — Metadata enables a coverage check.
    A verification script must be able to read the route registry, list all
    permission atoms referenced, and verify each exists in the permission catalog.
    Missing atoms are coverage gaps that must be caught in CI.

  Rule 6 — htmx_target documents backend-to-frontend contracts.
    For HTMX-served routes, htmx_target names the DOM selector the response
    targets. This field documents the contract so frontend fragment boundaries
    stay in sync with backend responses.

  ## Anti-Patterns
  4–5 items: inline permission strings instead of registry, unregistered public
  routes, routes referencing non-existent permission atoms, /me built from
  a separate hardcoded list, HTMX targets undocumented.

  ## Navigation
  The canonical implementation of route metadata in each language lives in
  examples/canonical-auth/<language>/ — see CATALOG.md for status.
  The shared route metadata spec is in examples/canonical-auth/domain/spec.md.

Title: /me Endpoint and Auth Discoverability

Required sections:

  ## Core Principle
  The /me endpoint is the primary discoverability surface for authentication
  and authorization state. A user who calls /me should be able to understand
  who they are, which tenant they belong to, what groups they are in, what
  they can do, and which endpoints or UI sections are available to them.
  /me is not a convenience endpoint — it is the foundation of self-documenting
  auth systems and HTMX-friendly permission-aware UIs.

  ## Required /me Response Fields
  A table:
  | Field              | Type     | Description                                      |
  | sub                | string   | User subject identifier                          |
  | email              | string   | User email                                       |
  | tenant_id          | string   | Active tenant identifier                         |
  | tenant_name        | string   | Active tenant display name                       |
  | tenant_role        | string   | Broad role label (admin / member / etc.)         |
  | groups             | array    | Group slugs or IDs the user belongs to           |
  | permissions        | array    | Effective permission atoms                       |
  | acl_ver            | int      | Current authorization version                   |
  | allowed_routes     | array    | Route metadata records the user may access       |
  | guide_sections     | array    | Docs/UI sections available to the user           |
  | issued_at          | string   | Token issuance time (ISO 8601)                   |
  | expires_at         | string   | Token expiry time (ISO 8601)                     |

  ## Rules

  Rule 1 — /me is computed from the validated token plus live data.
    The base claims (sub, tenant_id, permissions) come from the validated token.
    tenant_name, groups, and allowed_routes are derived from the database
    or route registry at request time. Serving stale /me from a cached token
    alone is wrong.

  Rule 2 — allowed_routes is filtered, not enumerated.
    /me must not return all routes. It returns only routes where the required
    permission is in the user's effective permission list. This filtering is
    done at request time from the route metadata registry.

  Rule 3 — guide_sections supports HTMX-driven UI.
    guide_sections is the list of docs/UI sections the user can access,
    derived from allowed_routes' docs_section fields. A server-rendered HTMX
    dashboard can render navigation from this list without client-side logic.

  Rule 4 — /me must be protected.
    /me requires a valid token. It is not public. An unauthenticated caller
    receives 401.

  Rule 5 — /me response is a teaching tool.
    The response format should be self-explanatory to a developer reading it
    for the first time. Field names must not be opaque abbreviations.
    Description fields in allowed_routes must explain the operation in plain
    language.

  Rule 6 — /me response may be rendered as HTML.
    For HTMX backends, /me should support both JSON response (for API clients)
    and HTML fragment response (for HTMX-driven profile/guide pages).
    Content negotiation via Accept header is the canonical approach.

  ## Anti-Patterns
  4–5 items: /me as a simple JWT decode echo, serving /me without token
  validation, hardcoded allowed_routes list, missing guide_sections,
  /me rendered only as JSON with no HTMX variant.

  ## Navigation
  Route metadata registry: context/doctrine/route-metadata-registry.md
  Domain spec (response shape detail): examples/canonical-auth/domain/spec.md
  Canonical implementations: examples/canonical-auth/<language>/

Target length: 65–80 lines.
