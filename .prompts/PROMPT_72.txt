PHASE 3 — UPDATES

Apply changes in this order.

3A. Create: context/skills/manifest-selection.md
  This skill fills boot-sequence step 5: "Select a manifest: prefer the manifest with the
  strongest repo-signal and router match."

  The skill must cover:
  - How to score signal strength: lockfile match > framework entrypoint > doc-only mention.
    A manifest backed by two or more strong signals outranks one backed by a single weak signal.
  - The tie-breaking rule: when signal strength is equal, prefer the manifest with the narrower
    scope (fewer optional context items) because it reflects a more precise task match.
  - When repo-signal-hints.json exists, it is machine-readable authority — read it before
    comparing manifests by hand.
  - What to do when no manifest is clearly dominant: stop, state the ambiguity, and do not merge
    near-match manifests. Raising the ambiguity is more valuable than a plausible guess.
  - The Avoid list: choosing a manifest from one transient file, merging two near-matches, picking
    the manifest that loads the most context as a hedge.

  Format: follow the existing skill format (heading, purpose sentence, ## Procedure, ## Good
  Triggers, ## Avoid). Under 50 lines.

3B. Create: context/skills/verification-path-selection.md
  This skill fills boot-sequence step 8 (verify the changed boundary).

  The skill must cover:
  - The decision tree for which verification level the current change surface requires:
      storage write or read     → real integration test against docker-compose.test.yml
      UI state or HTMX fragment → Playwright e2e test against running app
      pure logic, transforms    → unit test, no docker needed
      boot health, happy path   → smoke test, can run without full infra
  - The rule that verification is not complete until a test, harness, or check was actually run.
    "Plausible generation" is not verification.
  - The escalation rule: when in doubt, use the higher-confidence level, not the lower one.
  - When to load a specific workflow instead of implementing ad hoc:
      storage boundary → context/workflows/add-storage-integration.md
      smoke check      → context/workflows/add-smoke-tests.md
      Playwright e2e   → context/workflows/add-playwright-ui-verification.md
  - The Avoid list: calling a mock-backed test an integration test, claiming the smoke test
    covers a storage boundary, skipping verification because "the logic is simple".

  Format: same lightweight format as other skills. Under 60 lines.

3C. Create: context/skills/context-bundle-assembly.md
  This skill fills boot-sequence step 6: "Assemble the minimal bundle: required context first,
  optional context only when activated by the task."

  The skill must cover:
  - What makes context required: the active change surface directly touches it. A storage write
    requires the storage stack doc. A route change requires the stack's API surface doc.
  - What makes context optional: the content is nearby but not on the change surface. The
    Postgres doc is not required when the change is a pure data-transform function.
  - Load-order discipline:
      doctrine before workflow (invariants before steps)
      workflow before stack (what to do before how to do it)
      stack before example (framework rules before implementation patterns)
      one example over several near-matches
  - The one-dominant-path invariant: load one workflow, one stack surface, one archetype.
    If two workflows look equally relevant, pick the one that touches the change surface first.
  - When to add a second context item: only when the task has a clear orthogonal concern (e.g.,
    a storage write plus a smoke test). Two clearly distinct surfaces can each load one item.
  - The Avoid list: loading several near-match examples as a hedge, loading doctrine "just in
    case", widening the bundle when routing is still ambiguous (stop instead).

  Format: same lightweight format. Under 60 lines.

3D. Create: context/skills/memory-continuity-discipline.md
  This skill covers both boot step 3 (recover continuity) and step 8 (update continuity).

  The skill must cover two sub-procedures:

  READING MEMORY.md:
  - Check whether the stated objective still matches the current repo signals and task.
  - Check the working set: do the files listed still exist and still look active?
  - Check verification status: is the claimed verification state still consistent with the
    test files present?
  - If MEMORY.md is stale (objective completed, working set gone, or contradicts current state),
    treat it as outdated and do not use it to constrain the current task.

  WRITING MEMORY.md:
  - Capture: current objective, active working set (3–7 files max), key decisions made,
    verification status (what was run, what passed), and the next concrete step.
  - Prune: completed sub-tasks, superseded plans, files that are no longer on the change surface.
  - Update at meaningful stop points, not continuously. The right moment is when a phase
    boundary is crossed or a verification step completes.
  - Never write implementation details that are already in the code or commit history.
    MEMORY.md is operational state, not a second source of truth.

  Format: same lightweight format, two clearly labeled subsections. Under 70 lines.
