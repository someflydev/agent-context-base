  | Field         | Type      | Constraints           | Generation Notes                         |
  |---------------|-----------|-----------------------|------------------------------------------|
  | id            | UUID v4   | unique, primary key   |                                          |

  | role          | enum      | admin/member/viewer   |                                          |

  ## Cross-Field Rules (enforced in validation)

  Rule A: membership.joined_at >= organization.created_at

  Rule C: project.created_by must be a member of project.org_id
  Rule D: audit_event.user_id must be a member of audit_event.org_id
  Rule E: audit_event.project_id must belong to audit_event.org_id

  Rule H: invitation.invited_by must be a member of invitation.org_id
  Rule I: invitation.invited_email must not match any member's email for that org

Required content:

  # TenantCore Generation Order

  ## Canonical Generation Sequence

  The following order must be followed in every language implementation.
  It satisfies all foreign key constraints and cross-field rules.

  Stage 1 — Root entities (no parents):
    1. Generate organizations
       → Build org_id pool
       → Build org metadata map (plan, region, created_at, member_ids)

  Stage 2 — Global entities (reference orgs but are globally unique):
    2. Generate users
       → Build user_id pool
       → Build user metadata map (locale, timezone)

  Stage 3 — Bridge entities (join roots to global entities):
    3. Generate memberships (org_id × user_id × role × joined_at)
       → For each org: sample from user pool
       → Apply cardinality rule: 3–50 members per org, skewed toward 3–10
       → Build org_member_map: org_id → set of user_ids

  Stage 4 — Owned entities (require org + member):
    4. Generate projects (org_id, created_by from org_member_map)
       → Build project_id pool
       → Build project metadata map (org_id, created_at)

  Stage 5 — Activity entities (require org + member + project):
    5. Generate audit_events (org_id, user_id from org_member_map,
                               project_id from org's projects,
                               ts >= project.created_at)

  Stage 6 — Support entities (optional, generated last):
    6. Generate api_keys (org_id, created_by from org_member_map)
    7. Generate invitations (org_id, invited_by from org_member_map,
                             invited_email NOT in org_member_map emails)

  ## ID Pool Pattern

  Every stage must maintain an ID pool. Do not inline ID generation.

  Python pseudocode (conceptual only — actual implementation in PROMPT_121):
    org_ids = []
    for _ in range(profile.num_orgs):
        org = generate_org(seed_state)
        org_ids.append(org["id"])
        orgs.append(org)

  ## Why This Order Matters

  (3–4 sentences) Explain that generating children before parents creates
  forward references that cannot be resolved, forces retry loops, and produces
  FK violations. The canonical order ensures every foreign key can be resolved
  at generation time without rollback or retry.

  # Seed Registry

  ## Purpose

  Every canonical faker example must use a fixed seed for its smoke profile.
  This file documents the canonical seeds used across the arc.

  ## Canonical Seeds

  | Profile   | Seed  | Notes                                      |
  |-----------|-------|--------------------------------------------|
  | smoke     | 42    | Used in all smoke tests. Must not change.  |
  | small     | 1000  | Stable reference small dataset.            |
  | medium    | 5000  | Stable reference medium dataset.           |
  | large     | 9999  | For benchmark runs. May be overridden.     |

  ## Per-Language Seeding

  Each language has a different RNG API. The stack files document the seeding
  API. Key principles:
  - The seed must be applied before the first faker/random call.
  - The seed must flow through the entire generation pipeline.
  - No language-native random call should bypass the seeded generator.
  - Two runs with the same seed must produce byte-identical JSONL output.

  ## Reproducibility Test

  Every smoke test must include a reproducibility assertion:
    run1 = generate(seed=42, profile="smoke")
    run2 = generate(seed=42, profile="smoke")
    assert run1 == run2  (or compare output files)

Create: examples/canonical-faker/domain/generation_patterns.py

This is the reference Python implementation of the generation pattern library.
Language implementations in PROMPT_121 will use this as their core.
Other language implementations (PROMPT_122+) are NOT required to use this file —
they implement the same patterns idiomatically in their own language.

This module must implement:

  1. class Profile(dataclass):
     num_orgs, num_users: int
     seed: int
     SMOKE, SMALL, MEDIUM, LARGE = class methods returning Profile instances

  2. build_org_pool(profile: Profile, rng) -> list[dict]:
     Generates organizations using Faker with the given RNG.
     Returns list of org dicts. Each has: id, name, slug, plan, region, created_at, owner_email.
     plan distribution: 50% free, 35% pro, 15% enterprise (use weighted random choice).
     region distribution: 40% us-east, 25% us-west, 20% eu-west, 15% ap-southeast.

  3. build_user_pool(profile: Profile, rng) -> list[dict]:
     Generates users. Returns list of user dicts.
     locale distribution: 60% en-US, 20% en-GB, 10% de-DE, 10% fr-FR.
     timezone: consistent with locale (not pure random — use a lookup table).

  4. build_membership_pool(orgs, users, rng) -> list[dict]:
     For each org: sample 3–50 users (skewed toward 3–10 with Poisson or triangular
     distribution). Assign roles with weights: 5% owner, 10% admin, 60% member, 25% viewer.
     Ensure at least one owner per org. joined_at >= org.created_at.
     Returns memberships list AND org_member_map (org_id → list of member user_ids).

  5. build_project_pool(orgs, org_member_map, rng) -> list[dict]:
     For each org: sample 1–20 projects (skewed 1–5). created_by from org_member_map.
     Returns projects list AND project_org_map (project_id → org_id).

  6. build_audit_event_pool(orgs, org_member_map, projects, project_org_map, rng) -> list[dict]:
     For each project: generate 5–500 events (skewed, active projects have more).
     user_id from org_member_map[project.org_id]. ts >= project.created_at.

  7. build_api_key_pool(orgs, org_member_map, rng) -> list[dict]:
     0–10 per org. created_by from org_member_map. key_prefix unique.

  8. build_invitation_pool(orgs, org_member_map, users, rng) -> list[dict]:
     0–5 per org. invited_email must NOT be an existing member's email.

  9. generate_dataset(profile: Profile) -> dict:
     Runs all eight stages in the canonical order.
     Returns: {"organizations": [...], "users": [...], ...}
     Must use a single seeded Faker instance and single seeded random.Random.
     Must pass validate_dataset() before returning.

 10. validate_dataset(dataset: dict) -> ValidationReport:
     Checks all Cross-Field Rules A–I from domain/schema.md.
     Returns a ValidationReport dataclass with:
       ok: bool
       violations: list[str]
       counts: dict[str, int]
       seed: int
       profile: str

 11. write_jsonl(dataset: dict, output_dir: Path, profile_name: str):
     Writes one JSONL file per entity to output_dir/{entity}.jsonl.
     Uses chunked writes (chunk_size=1000).

 12. write_csv(dataset: dict, output_dir: Path, profile_name: str):
     Writes one CSV per entity. Includes header row.

Implementation requirements:
  - Use `from faker import Faker` and `import random`
  - Seed Faker with: `fake = Faker(); Faker.seed(profile.seed)`
  - Seed random with: `rng = random.Random(profile.seed)`
  - Do NOT use random.seed() (global state) — use the rng instance
  - Import uuid (standard library) for ID generation
  - No other dependencies in this file

Add a `__main__` block:
  if __name__ == "__main__":
      import sys
      profile_name = sys.argv[1] if len(sys.argv) > 1 else "smoke"
      profiles = {"smoke": Profile.SMOKE, "small": Profile.SMALL,
                  "medium": Profile.MEDIUM, "large": Profile.LARGE}
      profile = profiles[profile_name]
      dataset = generate_dataset(profile)
      print(dataset["report"])

Create: examples/canonical-faker/domain/validate_output.py

A standalone validation script that reads JSONL output files and validates
FK integrity and cross-field rules. This is the language-agnostic validator —
it can validate output from any language's generator.

Implementation:

  Reads from: output_dir/{entity}.jsonl (one record per line)
  Checks:
    - All FK references resolve (org_id, user_id, project_id)
    - All Cross-Field Rules A–I from schema.md
    - Uniqueness constraints (org.id, user.email, api_key.key_prefix)
    - Row count sanity (at least 1 record per entity)
    - Temporal ordering (joined_at >= org.created_at, etc.)
  Outputs:
    - Console summary: PASS/FAIL per rule
    - Exit code 0 on pass, 1 on fail
    - Optional --json flag that outputs a machine-readable report

CLI:
  python3 examples/canonical-faker/domain/validate_output.py \
    --input-dir output/smoke/python \
    [--json]

Dependencies: only Python standard library (json, pathlib, argparse, sys).
