This prompt adds comprehensive critical user journey (CUJ) Playwright tests for the
full faceted filter UI: include/exclude filters, quick excludes, text search, sort order,
and independent scroll layout — the complete capability set as of PROMPT_64.

The existing tests (playwright-split-filter-panel-example.spec.ts,
playwright-search-sort-example.spec.ts) test individual features in isolation.
This prompt tests the features as a user actually uses them: in sequence, in combination,
and with state carried across multiple interactions. These are not unit tests of the
filter logic — they are end-to-end tests of the user experience.

A critical user journey (CUJ) is a multi-step interaction sequence that represents a
realistic task a user would perform. Each CUJ in this spec corresponds to a named
scenario with a clear user goal, a sequence of actions, and a set of assertions that
verify the system behaved correctly at each step and at the end.

CUJ-1: "Discover and narrow: facet filter followed by text search"
  Goal: User starts with an unfiltered view, applies a facet filter to narrow by
        dimension, then uses the search bar to narrow further within that filtered view.
  Steps:
    1. Load /ui/reports with no params.
       Verify: result count = 6, all facet option counts are at baseline.
    2. Click/activate status_out=archived (exclude archived).
       Verify: result count = 5, archived include option is greyed with count=0 (RULE 1),
               archived exclude option shows data-active="true" with count=1 (RULE 2).
    3. Type "growth" into the search input.
       Verify: result count = 2 (daily-signups: growth/active/us, trial-conversions: growth/active/us — api-latency and checkout-failures are growth/active/eu but wait: let me clarify)
       Wait — restate: with status_out=archived and query=growth (substring of team value
       NOT report_id). If search matches only report_id: no rows have "growth" in report_id.
       Use query="signup" for this step instead.
       With status_out=archived and query=signup: matches daily-signups, trial-conversions.
       Verify: result count = 2.
       team facet: growth=2, platform=0.
       The archived include option: still greyed (RULE 1, exclude still active) with count=0.
    4. Clear the search (set query to "").
       Verify: result count returns to 5 (status_out=archived still active).
       Facet counts return to the archived-excluded baseline.
  Key assertions:
    - Counts update at each step without page reload.
    - RULE 1 (include greyed when excluded) holds throughout.
    - The search input value is pre-populated from state at each step.
    - Clearing search does not clear the active facet filter.

SECTION 1 — Page object model

The test file must use a page object model to avoid repeating DOM selectors across tests.
Define a ReportsFilterPage class (or namespace) in the spec file with the following
typed helper methods:

  goto(params?: Record<string, string | string[]>): Promise<void>
    Navigates to /ui/reports with the given query params.

  resultCount(): Promise<number>
    Reads and returns the numeric value from [data-role="result-count"].

  facetCount(dimension: string, option: string, mode: "include" | "exclude"): Promise<number>
    Finds the element:
      [data-filter-dimension="{dimension}"][data-filter-option="{option}"][data-filter-mode="{mode}"]
    Returns the numeric value of data-option-count.

  isIncludeGreyed(dimension: string, option: string): Promise<boolean>
    Returns true if the include option for the given value has data-excluded="true".

  isIncludeCheckboxDisabled(dimension: string, option: string): Promise<boolean>
    Returns true if the include checkbox for the given value has the disabled attribute.

  isExcludeActive(dimension: string, option: string): Promise<boolean>
    Returns true if the exclude option for the given value has data-active="true".

  isExcludeCheckboxChecked(dimension: string, option: string): Promise<boolean>
    Returns true if the exclude checkbox for the given value is checked.

  quickExcludeActive(value: string): Promise<boolean>
    Returns true if [data-role="quick-exclude"][data-quick-exclude-value="{value}"]
    has data-active="true".

  quickExcludeCount(value: string): Promise<number>
    Returns the numeric text content of the count badge inside the quick exclude toggle.

  searchInputValue(): Promise<string>
    Returns the value attribute of [data-role="search-input"].

  sortSelectValue(): Promise<string>
    Returns the data-sort-order attribute of [data-role="sort-select"].

  resultsSectionSearchQuery(): Promise<string>
    Returns the data-search-query attribute of #report-results.

  resultsSectionSortOrder(): Promise<string>
    Returns the data-sort-order attribute of #report-results.

  firstResultCardText(): Promise<string>
    Returns the trimmed text content of the first result card in the results list.

  allResultCardTexts(): Promise<string[]>
    Returns the trimmed text content of all result cards in order.

  filterPanelScrollTop(): Promise<number>
    Returns document.getElementById('filter-panel').scrollTop.

  resultsContainerScrollTop(): Promise<number>
    Returns document.getElementById('report-results-container').scrollTop.

  scrollFilterPanelTo(px: number): Promise<void>
    Sets filter-panel scrollTop to the given value via page.evaluate.

  scrollResultsContainerTo(px: number): Promise<void>
    Sets report-results-container scrollTop to the given value via page.evaluate.

  overflowY(elementId: string): Promise<string>
    Returns the computed overflow-y style of the element with the given id.

Each method should use `this.page` to access the Playwright Page object.
The class is instantiated once in beforeEach with `const rp = new ReportsFilterPage(page)`.

SECTION 2 — Test file structure

FILE: examples/canonical-integration-tests/playwright-cuj-filter-example.spec.ts

The file must:
  - Import { test, expect, Page } from '@playwright/test'.
  - Define the ReportsFilterPage class as described in SECTION 1.
  - Use test.describe for each CUJ (10 describe blocks, one per CUJ).
  - Each describe block may contain one or more test() calls representing sub-steps
    or related assertions within that CUJ. Prefer grouping the full CUJ sequence
    into a single test() for each CUJ, with step labels in comments. Use sub-tests
    (separate test() calls) only when the sub-steps are truly independent.
  - Use beforeEach to create a new ReportsFilterPage instance.
  - Use await page.waitForSelector or await page.waitForResponse where needed to
    handle HTMX partial swaps before asserting. Do not assert immediately after
    triggering an HTMX action without waiting.
  - Use a BASE_URL constant (default: "http://localhost:8000") for all navigation.
    It can be overridden by the PLAYWRIGHT_BASE_URL environment variable.

Debounce handling:
  For tests that type into the search input, use:
    await rp.goto({ query: "signup" })
  rather than simulating typing into the input when possible, to avoid debounce delays.
  When the test must simulate typing (CUJ-6, CUJ-7), use:
    await page.fill('[data-role="search-input"]', 'query-term');
    await page.waitForTimeout(400); // exceed the 300ms debounce
    await page.waitForSelector('#report-results[data-search-query="query-term"]');

HTMX swap waiting pattern (to use after triggering form changes):
    await page.waitForSelector('#report-results[data-query-fingerprint]');
  Or wait for a specific expected attribute value to appear:
    await page.waitForSelector('#report-results[data-result-count="2"]');

Scroll tests (CUJ-9):
  Use page.evaluate() calls as described in the CUJ-9 definition.
  Add await page.waitForTimeout(100) after scroll operations to allow browser layout
  to stabilize before asserting scroll positions.

SECTION 5 — Important constraints

  - Do NOT test implementation details. Tests assert data-* attributes, visible text,
    result counts, and CSS properties — not internal function calls or HTML class names
    that may differ between language implementations.

  - Data-* attribute selectors are the stable test contract. CSS classes may vary
    between implementations; data-* attributes must not.

  - Do NOT use page.waitForTimeout as the primary synchronization mechanism (only use
    it for debounce handling and scroll stabilization as specified). Prefer
    waitForSelector with a specific expected state as the sync point.

  - Do NOT assert on exact HTML structure (e.g., "the label contains a span which
    contains a checkbox"). Assert on data-* attributes and logical states. Structure
    may evolve; the data contract must not.

  - The page object model methods must not throw if an element is absent — they should
    return a sensible null/undefined so callers can handle missing elements gracefully
    in non-required assertions.

  - Every CUJ test must reach its final assertion state even if intermediate steps
    require state to be set via URL params (navigate to the full URL) rather than
    simulating every interaction from a blank page. Use direct navigation where
    it makes the test more readable and faster to execute.

  - Expected count values in the tests are derived from the shared dataset defined in
    README-faceted-filter-examples.md. Do not invent count values — compute them from
    the dataset and verify against the README's reference counts.

  - The PLAYWRIGHT_BASE_URL default must be "http://localhost:8000" to match the
    existing test suite's convention. Do not hard-code a different port.
