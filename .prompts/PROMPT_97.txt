================================================================================
BACKGROUND: WHAT THE STARTUP TRACE IS
================================================================================

The Session Context Briefing (added in PROMPT_94) reports the OBJECTIVE STATE of the
repo at the moment `work.py resume` is run: files present, git state, memory base,
operator queue, complexity posture. It is generated automatically from file system state.

The startup trace is different: it records what the ASSISTANT DECLARES it loaded during
boot. This is NOT automatic logging of internal model reads — that is not observable.
It IS a structured self-declaration that the assistant fills in at the end of its boot
sequence when it follows the protocol defined here.

Why both?
  - The Session Context Briefing answers: "what was the state of the repo at session start?"
  - The startup trace answers: "what did the assistant say it loaded and why?"

The trace enables:
  - Post-session review of what context was declared (and comparing against expected)
  - Budget validation: run budget-report against declared files to score the session
  - Pattern analysis: over many sessions, do traces show consistent loading discipline?
  - Debugging: when a session goes wrong, what did the assistant say it had loaded?

Important constraint: the trace is self-reported and cannot be proven. Document this
clearly. The value is in the discipline of declaration, not in verified truth.

================================================================================
PART 1: DEFINE THE STARTUP TRACE FORMAT
================================================================================

Create: `context/doctrine/startup-trace-protocol.md`

This is the normative specification for the startup trace format. Cover:

1. What a startup trace is
   A structured self-declaration written by the assistant at the end of its boot sequence.
   Not automated. Not verified. Valuable because it makes loading decisions inspectable
   and forces intentional declaration.

2. When to write one
   Write a startup trace when:
   - The session loads a substantial context bundle (medium profile or larger)
   - The task spans multiple stacks or workflows
   - The operator is debugging a previous session's context load
   - startup_trace_enabled is true in the repo's .acb/ config (if that flag exists)

   Do not write one for:
   - Tiny prompt-only sessions with minimal context
   - Quick status queries or one-file fixes
   - When the Session Context Briefing already captures everything needed

3. Trace file format
   Plain text with clearly delimited sections (no YAML frontmatter — keep it simple).
   Stored as: logs/startup/<YYYY-MM-DD-HHMMSS>-trace.md
   Distinguish from the Session Context Briefing logs which end in -resume.log

4. Required sections
   The format is:

   ================================================================
   Startup Trace
   ================================================================
   Created:         <ISO datetime>
   Session:         <short description of the task or prompt being started>
   Entry Point:     <AGENT.md | CLAUDE.md>

   Declared Files Read:
     1. <path> — <why loaded: e.g., "boot anchor", "active workflow", "primary stack">
     2. <path> — <why>
     ...

   Routers Consulted:
     - <none | router name(s) and what they resolved>

   Doctrines Loaded:
     - <doctrine file names, or "none beyond boot sequence">

   Memory Artifacts Seen:
     - <memory/summaries/PROMPT_XX_completion.md: present | not present>
     - <context/MEMORY.md: present | missing>

   Budget Estimate:
     Files: <N>   Estimated Profile: <tiny|small|medium|large|cross-system|unknown>
     Note: run `work.py budget-report --bundle <files>` for a scored evaluation.

   Router Decision:
     Archetype:      <inferred | not applicable>
     Primary Stack:  <inferred | not applicable>
     Workflow:       <selected | not applicable>
     Confidence:     <strong (>= 0.85) | usable | weak | not routed>

   Warnings:
     - <none | list any loading concerns, drift, or missing expected files>
   ================================================================

5. Anti-patterns
   - Writing a trace that just copies the Session Context Briefing (they serve different purposes)
   - Listing files not actually loaded (false declarations are worse than no trace)
   - Omitting files that were loaded (incomplete traces undermine their value)
   - Inventing "Routers Consulted" entries without actually consulting them

6. Relationship to the budget model
   The "Declared Files Read" section is the input to `work.py budget-report`.
   After writing a trace, an operator can run:
     `work.py budget-report --bundle <declared-files...>`
   to score the declared bundle against the budget profiles.

Keep this file tight: 60–90 lines. It is doctrine, not a tutorial.

PART 2: work.py startup-trace SUBCOMMAND
================================================================================

Add a `startup-trace` command to `scripts/work.py` with two subsubcommands:

  python3 scripts/work.py startup-trace write [options]
  python3 scripts/work.py startup-trace show

------------------------------------------------------------------------
2a. startup-trace write
------------------------------------------------------------------------

  python3 scripts/work.py startup-trace write \
      --session "brief description" \
      --files path1 path2 ... \
      [--entry-point AGENT.md] \
      [--routers "task-router resolved to add-api-endpoint workflow"] \
      [--doctrines doctrine1 doctrine2 ...] \
      [--archetype backend-api-service] \
      [--primary-stack python-fastapi] \
      [--workflow add-api-endpoint] \
      [--confidence 0.9] \
      [--warnings "cross-domain file loaded without explicit trigger"]

Behavior:
  - Assembles the trace format from arguments
  - Writes to: logs/startup/<YYYY-MM-DD-HHMMSS>-trace.md
  - Creates logs/startup/ if it does not exist
  - Prints the output path on success
  - Exits 0 on success, 1 on error

All flags except --session are optional. When --files is omitted, the trace is written
with "Declared Files Read: none declared" (a valid minimal trace).

For "Budget Estimate": compute it automatically using context_budget.py.
If --files is provided, score the bundle (with --primary-stack and --archetype as
ModifierContext if supplied) and include the profile in the estimate. If --files is not
provided, write "Files: 0   Estimated Profile: unknown".

The --confidence flag accepts a float 0.0-1.0. Map to the text labels:
  >= 0.85: "strong (>= 0.85)"
  0.70-0.84: "usable (0.70–0.84)"
  0.55-0.69: "weak (0.55–0.69)"
  < 0.55: "very weak (< 0.55)"
  not provided: "not declared"

------------------------------------------------------------------------
2b. startup-trace show
------------------------------------------------------------------------

  python3 scripts/work.py startup-trace show [--last N]

Behavior:
  - Lists the N most recent trace files in logs/startup/ that end in -trace.md
    (distinct from -resume.log files written by --write-startup-log)
  - Default N=1: shows the content of the most recent trace file
  - If no trace files exist: prints "No startup traces found in logs/startup/"
  - If N > 1: prints just the filenames and their Created: date, not full content

------------------------------------------------------------------------
2c. Parser integration
------------------------------------------------------------------------

`startup-trace` is a subcommand of work.py. The subsubcommands `write` and `show`
should be implemented using argparse subparsers nested under startup-trace.

Match the style of other subcommands in work.py. Avoid adding a third level of nesting
if argparse makes it awkward — a flat `startup-trace-write` and `startup-trace-show`
pair is acceptable if nested subparsers complicate the parser structure significantly.
Use your judgment but document the chosen approach in a comment.
