BACKGROUND: ELIXIR TEACHING SURFACE AND HTMX vs LIVEVIEW
================================================================================

This implementation uses Phoenix with standard controllers + HEEx templates +
HTMX. It does NOT use LiveView. This is an intentional design choice for this
arc — see context/stacks/elixir-phoenix-htmx-plotly.md for the rationale.

The Elixir implementation teaches:

  - Phoenix controllers for server-rendered analytics pages and HTMX fragments
  - HEEx templates for typed, component-friendly HTML rendering
  - Elixir map-based figure building (no Elixir Plotly library needed; pure maps
    + Jason for JSON serialization is idiomatic)
  - Phoenix context modules for analytics aggregation (OTP/BEAM-friendly design)
  - Detecting the HX-Request header to route full-page vs. fragment responses
  - Jason.encode! for figure serialization in the controller
  - ExUnit for all testing

Key difference from other stacks:
  Elixir figure builders return plain Elixir maps. Jason.encode!() in the
  controller serializes them. The map structure mirrors the Plotly.js figure JSON.

  Example figure builder return:
    %{
      data: [
        %{type: "scatter", mode: "lines+markers", name: "production",
          x: dates, y: counts, hovertemplate: "%{y} events on %{x}"}
      ],
      layout: %{
        title: %{text: "Event Volume Over Time"},
        xaxis: %{title: %{text: "Date"}},
        yaxis: %{title: %{text: "Event Count"}}
      },
      meta: %{
        total_count: total,
        filters_applied: filters_applied,
        generated_at: DateTime.utc_now() |> DateTime.to_iso8601()
      }
    }

This is the three-layer model applied idiomatically in Elixir.

PART 3: FIXTURE LOADING
================================================================================

Implement lib/analytics_workbench/data/loader.ex:

  defmodule AnalyticsWorkbench.Data.Loader do
    @doc """
    Loads fixture data from the canonical-analytics domain fixtures directory.
    Dogfoods examples/canonical-faker/ via the shared fixture files produced
    by examples/canonical-analytics/domain/generate_fixtures.py.
    """

    def load_fixtures(profile \\ :smoke) do
      path = fixture_path(profile)
      path |> File.read!() |> Jason.decode!(keys: :atoms)
    end

    def fixture_path(profile) do
      base = System.get_env("FIXTURE_PATH") ||
        Path.join([__DIR__, "../../../../domain/fixtures"])
      Path.join(base, "#{profile}.json")
    end
  end

Store the loaded fixture data in the application's supervision tree using an
Agent or a simple GenServer. Access via a module-level function:

  defmodule AnalyticsWorkbench.Data.Store do
    use Agent

    def start_link(_) do
      Agent.start_link(fn -> AnalyticsWorkbench.Data.Loader.load_fixtures(:smoke) end,
        name: __MODULE__)
    end

    def get, do: Agent.get(__MODULE__, & &1)
  end

PART 4: FILTER STATE
================================================================================

Implement lib/analytics_workbench/filters.ex:

  defmodule AnalyticsWorkbench.Filters do
    defstruct date_from: nil, date_to: nil,
              services: [], severity: [], environment: []

    def parse(params) do
      %__MODULE__{
        date_from: Map.get(params, "date_from"),
        date_to: Map.get(params, "date_to"),
        services: parse_multi(params, "services"),
        severity: parse_multi(params, "severity"),
        environment: parse_multi(params, "environment"),
      }
    end

    defp parse_multi(params, key) do
      case Map.get(params, key) do
        nil -> []
        val when is_list(val) -> val
        val when is_binary(val) -> String.split(val, ",") |> Enum.reject(&(&1 == ""))
      end
    end
  end

Phoenix query params for multi-select <select multiple> are sent as
services[]=a&services[]=b — ensure parse_multi handles both [] suffix and plain key.

PART 5: ANALYTICS CONTEXT MODULES
================================================================================

Implement all six analytics modules in lib/analytics_workbench/analytics/.

Each module exposes one or more public functions following the pattern:
  def aggregate_<name>(data, %FilterState{} = filters) :: result_map

Return plain Elixir maps (not structs) for aggregate results. This keeps the
return type inspectable and pattern-matchable in tests.

Required functions and result shapes:

  Analytics.Events.aggregate_event_counts(events, filters)
    Returns: %{dates: [String], counts: [Integer], by_environment: %{String => [Integer]}}

  Analytics.Services.aggregate_service_error_rates(events, services, filters)
    Returns: %{services: [String], error_rates: [Float], total_events: [Integer]}

  Analytics.Distributions.aggregate_latency_histogram(samples, filters)
    Returns: %{values: [Float], p50: Float, p95: Float, p99: Float}

  Analytics.Distributions.aggregate_latency_by_service(samples, services, filters)
    Returns: %{services: [String], p50s: [Float], p95s: [Float], p99s: [Float]}

  Analytics.Heatmap.aggregate_event_heatmap(events, filters)
    Returns: %{hours: [Integer], days: [String], counts: [[Integer]]}

  Analytics.Funnel.aggregate_funnel_stages(sessions, filters)
    Returns: %{stages: [String], counts: [Integer], drop_off_rates: [Float]}

  Analytics.Incidents.aggregate_incident_severity(incidents, filters)
    Returns: %{severities: [String], counts: [Integer], mttr_by_severity: %{String => Float}}

For percentile calculations: sort with Enum.sort/1, index by floor(length * p / 100).
Implement a private percentile(values, p) helper in distributions.ex.

All functions: return a zero-valued result map (not raise) when input list is empty.

Filter application pattern:
  Use Enum.filter/2 for each non-empty FilterState field.
  Example: filter by environment:
    if filters.environment != [] do
      Enum.filter(events, &(&1.environment in filters.environment))
    else
      events
    end

           environment_list: environment_list(store),
           severity_list: severity_list(store))
