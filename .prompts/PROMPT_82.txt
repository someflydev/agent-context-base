You are working inside the agent-context-base repository.

This prompt adds a second derived-generation mode for `scripts/new_repo.py`.
The current derived flow is intentionally selective: it vendors the chosen
base manifests and the files those manifests point at. That is now good enough
for a compact downstream repo, but it is not the "everything local" mode.

Build a new derived-generation mode that maximally vendors the relevant
agent-context-base context into each generated derived repo so a fresh
assistant session inside that repo has an aggressively complete local bundle.

The default mode must remain the current compact behavior. This prompt adds a
new explicit operator opt-in for maximal vendoring.

Output changes as commit-ready batches labeled `[PROMPT_82]` unless a later
prompt in this sequence changes that instruction.

I. OPERATOR EXPERIENCE
===============================================================================

Add a new explicit mode to `scripts/new_repo.py` for derived generation.

Pick a clear CLI surface and use it consistently. Preferred shape:

  --derived-context-mode compact
  --derived-context-mode maximal

Rules:

1. The default must remain `compact`.
2. The new option only needs to affect derived generation.
3. Non-derived generation may ignore the option or reject it cleanly; do not
   create ambiguous semantics.
4. The selected mode must be recorded in generated profile metadata.

Examples of the target operator experience:

  python3 scripts/new_repo.py \
    --derived-example operator-surface \
    --derived-context-mode maximal \
    --target-dir /tmp/operator-surface-max

    --derived-example team-a \
    --derived-context-mode maximal \
    --target-dir /tmp/team-a-max

===============================================================================
II. WHAT MAXIMAL MODE MUST INCLUDE
===============================================================================

In maximal mode, every generated derived repo should vendor a materially larger
repo-local context bundle than compact mode.

At minimum, include all of the following when they exist in the base repo and
are relevant to the prompt-first orchestration shape:

1. The selected manifests under `manifests/base/`.
2. All files referenced by those manifests under:
     - `required_context`
     - `optional_context`
     - `preferred_examples`
     - `recommended_templates`
3. The prompt-first doctrine, workflows, archetype docs, stack docs, and
   anchors needed to route and continue prompt work locally.
4. The canonical prompt examples and canonical workflow examples that a
   downstream assistant would plausibly consult while extending `.prompts/`.
5. The prompt-generation and prompt-governance materials needed to continue
   prompt-first work inside the generated repo without improvising.

The generated repo should not need to "remember" where agent-context-base kept
these materials; it should have them locally.

===============================================================================
III. MAXIMAL MODE SHOULD BE THOUGHTFUL, NOT JUST BIG
===============================================================================

Do not implement maximal mode as "copy the whole repository" or "copy every
file under context/ and examples/".

Instead, define a thoughtful maximal bundle policy that is still bounded and
defensible. The downstream repo should be rich enough to work from locally, but
not a blind mirror of the source repository.

Good candidates to include:

- prompt-first doctrine and workflow anchors
- repo-signal and task-inference guidance used during startup
- canonical prompt examples
- canonical workflow checklists
- prompt-first templates and manifest/profile templates
- any additional context explicitly needed to interpret the vendored manifest
  metadata in a repo-local session

Potentially unnecessary unless justified:

- unrelated stacks not implicated by prompt-first orchestration
- unrelated archetype docs with no derived relevance
- verification fixtures for the base repo itself
- unrelated example registries or policies that do not guide downstream work

Make the selection model explicit in code and docs.
