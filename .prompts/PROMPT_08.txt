You are designing a “Context Complexity Budget” system for an agent-context-based repository used by coding assistants such as Codex, Claude, and Gemini.

This repository already has:
- doctrine docs
- workflows
- stack packs
- archetype packs
- routers
- manifests
- canonical examples
- templates
- assistant operating manual
- context boot sequence
- deployment and isolation doctrine for Docker-backed local infrastructure

Your task is to design a rigorous but practical budgeting system that mathematically limits how much context an assistant is allowed to load during a task.

The goal is to reduce context sprawl, prevent cross-stack contamination, reduce hallucinated architecture, and make assistant behavior more deterministic.

This system should act as a control layer between:
- router proposals
- manifest-defined context bundles
- actual context loading

In other words:
routers and manifests may suggest candidate context,
but the budget system decides what is actually allowed to load in the first pass,
and how bounded escalation may work if more context is needed.

The design must be concrete, insightful, and implementation-oriented.

The budget system should reflect the real architecture and working style of this repository, including:
- prompt-first repos
- backend APIs
- CLI tools
- local RAG systems
- multi-storage experiments
- data pipelines
- FastAPI + uv + ruff + orjson + polars
- TypeScript + Hono + Bun + Drizzle ORM + TSX
- Rust + Axum
- Go + Echo
- Elixir + Phoenix
- Redis/KeyDB + MongoDB
- DuckDB + Trino + Polars
- NATS Jetstream
- Meilisearch
- TimescaleDB (PostgreSQL extension for real-time analytics on time-series)
- Elasticsearch
- Qdrant vector database
- Dokku-oriented deployment
- repo-derived Compose `name:` values and isolated primary/dev versus test local stacks
- smoke-test-heavy philosophy
- canonical example guidance
- minimal relevant context loading
- deterministic routing and boot sequence behavior
