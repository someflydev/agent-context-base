BACKGROUND: CANONICAL EXAMPLE DESIGN GOALS
================================================================================

These examples are intentionally constrained:
- No live database required — fixtures are loaded from JSON at startup
- No Redis/session storage — acl_ver check is done via in-memory user store
- No external service dependencies — runs with a single command
- The in-memory store is a teaching simplification; each README must clearly
  note where a production system would use a real database

What these examples teach:
- How to wire JWT validation middleware correctly in this ecosystem
- How route metadata drives authorization (not inline if-chains)
- How tenant context is extracted and enforced structurally
- How /me is computed from token + route registry
- How to test auth behavior with crafted tokens

Aim for ~350–500 lines of application code per example (not counting tests).
Prefer clarity over cleverness. Comments should explain auth-specific choices,
not boilerplate.

PART B: Key Implementation Requirements

  auth/claims.py:
    @dataclass
    class AuthContext:
        sub: str
        email: str
        tenant_id: str | None
        tenant_role: str        # "super_admin" | "tenant_admin" | "tenant_member"
        groups: list[str]
        permissions: list[str]
        acl_ver: int
        issued_at: datetime
        expires_at: datetime

    AuthContext must be created from a validated decoded token dict.
    Include a method: has_permission(self, permission: str) -> bool

  auth/token.py:
    issue_token(user: User, store: InMemoryStore) -> str

    Must:
    - load RSA private key (or HS256 secret in test mode via env var)
    - compute effective permissions from store (user's groups → group_permissions)
    - set exp = iat + 900 (15 minutes)
    - embed all claims per domain spec (iss, aud, sub, exp, iat, nbf, jti, tenant_id,
      tenant_role, groups, permissions, acl_ver)
    - sign with PyJWT: jwt.encode(payload, key, algorithm="RS256")

    For test mode: accept TENANTCORE_TEST_SECRET env var → use HS256.
    Document this clearly in a comment.

  auth/middleware.py:
    get_auth_context(token: HTTPAuthorizationCredentials = Depends(HTTPBearer())) -> AuthContext

    Must:
    - decode and verify: signature, exp, nbf, iss, aud
    - raise 401 if any check fails
    - load user from store by sub
    - compare token acl_ver to store user.acl_ver; raise 401 if stale
    - extract tenant_id and verify user is an active member of that tenant
    - raise 403 if membership check fails
    - return AuthContext

  registry/routes.py:
    ROUTE_REGISTRY: list[dict] — one entry per route per domain spec.
    Each entry has: method, path, permission, tenant_scoped, description,
    service, resource, action, public (bool).

    Implement get_allowed_routes(permissions: list[str]) -> list[dict]
    that filters ROUTE_REGISTRY to entries where either public=True or
    the required permission is in the permissions list.

  routes/me.py:
    GET /me: requires get_auth_context dependency.
    Build /me response from AuthContext + store (for tenant_name) +
    get_allowed_routes(auth.permissions).
    Must include: sub, email, display_name, tenant_id, tenant_name, tenant_role,
    groups, permissions, acl_ver, allowed_routes, guide_sections, issued_at,
    expires_at.
    guide_sections: unique docs_section values from allowed_routes (if the
    route registry includes docs_section — omit if not present).

  RBAC enforcement pattern (NOT inline if-chains):
    Create a FastAPI dependency factory:
      def require_permission(permission: str):
          def checker(auth: AuthContext = Depends(get_auth_context)):
              if not auth.has_permission(permission):
                  raise HTTPException(status_code=403, detail="Forbidden")
          return checker

    Use it on routes:
      @router.get("/users", dependencies=[Depends(require_permission("iam:user:read"))])

  domain/store.py:
    InMemoryStore class:
    - loads from examples/canonical-auth/domain/fixtures/ at init
    - provide methods: get_user_by_id, get_user_by_email, get_tenant_by_id,
      get_groups_for_user(user_id, tenant_id), get_effective_permissions(user_id),
      verify_membership(user_id, tenant_id) -> bool, get_tenant_name(tenant_id)
    - for tenant isolation: all multi-user queries MUST filter by tenant_id

  admin.py routes:
    Gate with a super_admin check dependency (separate from require_permission):
      def require_super_admin(auth: AuthContext = Depends(get_auth_context)):
          if auth.tenant_role != "super_admin":
              raise HTTPException(status_code=403, ...)
