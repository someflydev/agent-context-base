SECTION 3 — Solo stack-pack expectations

Each new stack pack should follow the existing house style:

- short title
- one-paragraph “Use this pack when...” framing
- `Typical Repo Surface`
- `Change Surfaces To Watch`
- `Testing Expectations`
- `Common Assistant Mistakes`

But the content must be more specific than the current combined packs.

Mongo requirements:

- present MongoDB as a document-store and reporting-friendly operational datastore, not just “simple persistence”
- explicitly cover aggregation pipeline usage for reporting and operational analytics
- include the request/response log use case:
  - cleaned request/response logs dumped to MongoDB
  - weekly collections such as date-bucketed log collections
  - old collections can be dropped whole when retention expires
- explain when `partialFilterExpression` is useful:
  - sparse or shape-known documents
  - indexes targeted to the subset of documents that actually need a query path
- call out collection naming, compound indexes, document-shape drift, and retention strategy as primary change surfaces
- discourage lazy “just store arbitrary JSON forever” guidance

Redis requirements:

- treat Redis as a first-class solo pack, not a footnote to MongoDB
- make data-structure selection central:
  - strings with expiry
  - hashes
  - sets
  - sorted sets
  - lists
  - streams
- explicitly mention credible use cases such as:
  - leaderboards
  - expiring keys
  - counters / rate limits
  - coordination primitives
  - append-like event or job-adjacent streams
- mention Redis modules as optional capability expansion, but do not assume they are always present
- warn against flattening everything into opaque serialized blobs when native structures would be better

Trino requirements:

- present Trino as a federated analytics query layer, not as a general transactional database
- explain that when a repo uses multiple databases, Trino should often be considered as an analytical overlay
- instruct that compatible allowed databases may be added as catalogs when that materially improves cross-source analytics
- make the “compatible” constraint explicit:
  - do not imply every storage system automatically belongs behind Trino
  - stay connector-aware and realistic
- describe analytics queries that are meaningfully Trino-shaped compared with ordinary PostgreSQL/MySQL usage:
  - cross-catalog joins
  - analytical rollups across multiple sources
  - larger scan-style aggregations
  - warehouse-style read paths that do not belong on OLTP traffic
- call out catalog config, type mismatches, pushdown assumptions, and query cost as primary risks

PostgreSQL requirements:

- create a real baseline PostgreSQL stack pack
- present it as the default transactional SQL foundation for many backend repos in this ecosystem
- cover schema and migration discipline, indexing, transactional behavior, and representative reporting queries
- mention `jsonb` pragmatically without turning the doc into “Mongo in Postgres”
- include guidance for connection setup, migration files, query modules, seed/reset surfaces, and integration tests against real isolated PostgreSQL
- explain the relationship to `timescaledb.md`:
  - `postgresql.md` is the base pack
  - `timescaledb.md` adds extension-specific time-series concerns
  - when a repo uses TimescaleDB, assistants should usually load both

SECTION 4 — Combined-pack redesign expectations

Update the existing combined packs so their purpose is explicit.

`context/stacks/redis-keydb-mongo.md` should become a mixed-storage pack for cases such as:

- Redis plus Mongo in one service

- migration or comparison decisions between similar storage layers

`context/stacks/duckdb-trino-polars.md` should become a pack for:

- local analytical workflows plus federation
- DuckDB + Polars local transforms

`context/stacks/timescaledb.md` should be updated so it clearly assumes PostgreSQL fundamentals are already relevant and should normally pair with `postgresql.md`.
