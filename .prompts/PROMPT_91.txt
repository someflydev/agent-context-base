================================================================================
BACKGROUND: WHY THIS UPGRADE
================================================================================

The current memory model has one local layer: context/MEMORY.md (gitignored, per-machine,
ephemeral). It handles within-session and short-term across-session continuity well. However:

- There is no committed place for durable conceptual knowledge that outlives a session
- Prompt-boundary compaction artifacts have no committed structural home
- `artifacts/handoffs/` snapshots are useful but general-purpose, not prompt-aware
- The work.py resume/checkpoint cycle is strong but produces no persisted artifacts
- Fresh sessions must re-discover things that previous sessions already resolved

This prompt creates a committed `memory/` directory at the repo root — the Structured Knowledge
Base. It works alongside the existing model:

  context/MEMORY.md          local runtime state, gitignored, mutable, task-scoped
  artifacts/handoffs/        general timestamped handoff snapshots (existing, unchanged)
  memory/                    committed, structured, tiered knowledge base (NEW)

The committed memory base is NOT:
- a log of every action
- a replacement for doctrine (that stays in context/doctrine/)
- a replacement for MEMORY.md (that stays local and current-task focused)
- hidden state (everything here is committed, inspectable, version-controlled)

It IS:
- explicit
- version-controlled
- structured by purpose (concepts vs sessions vs summaries)
- a durable index point for fresh sessions

--------------------------------------------------------------------------------
memory/concepts/README.md
--------------------------------------------------------------------------------

Write a concise guide to this directory. Include:

- What belongs here:
  Durable knowledge unlikely to change with each session. Stable conventions not obvious
  from reading code. Architectural decisions whose rationale is not in comments. Known
  recurring pitfalls. Patterns that have come up in multiple sessions.

- What does NOT belong here:
  Active task state (context/TASK.md), session logs (memory/sessions/), checkpoint
  summaries (memory/summaries/), doctrine (context/doctrine/), ephemeral working notes.

- File format convention:
  Filename: kebab-case descriptive slug, e.g., prompt-numbering-discipline.md
  Header: `# Concept: <Title>`
  Body: dense, fact-first, no padding
  Last Updated: date stamp at bottom

- When to create a new concept file:
  After the second time you look something up and wish it were written down.
  After resolving a non-obvious confusion that affected implementation.
  After a session that produced a durable finding worth preserving across prompts.

--------------------------------------------------------------------------------
memory/sessions/README.md
--------------------------------------------------------------------------------

Write a concise guide. Include:

- What belongs here:
  Session logs for meaningful work sessions. Prompt execution notes when the execution
  produced non-obvious findings. Exploration results that should not be re-discovered.
  Decision rationale that is too specific to be a concept.

- File format convention:
  Filename: YYYY-MM-DD-<slug>.md  OR  PROMPT_XX-session.md
  Header: `# Session: <description>`
  Required sections: Date, Prompt Context, Key Findings, Decisions Made, Open Questions

- Pruning convention:
  Sessions older than ~30 days whose findings have been promoted to concepts/ or
  summaries/ may be removed. Never prune a session that is the sole record of a
  decision not captured elsewhere.

- Relationship to artifacts/handoffs/:
  memory/sessions/ holds prompt-first session notes. artifacts/handoffs/ holds
  general-purpose timestamped transfer snapshots. Both can coexist.

--------------------------------------------------------------------------------
memory/summaries/README.md
--------------------------------------------------------------------------------

Write a concise guide. Include:

- What belongs here:
  Prompt-boundary compaction artifacts. Checkpoint summaries created when pausing
  mid-prompt or completing a prompt. Resume handoffs for fresh sessions.

- Standard filename patterns:
  PROMPT_XX_completion.md    written when a prompt is fully done
  PROMPT_XX_resume.md        written when a prompt is paused mid-execution

- Required sections for summary files:
  ## Objective
  ## Completed
  ## Remaining                (omit or mark "none" if fully done)
  ## Tests Status
  ## Files Touched
  ## Key Commits
  ## Pitfalls and Caveats
  ## Next Recommended Action
  ## Created At

- Relationship to context/MEMORY.md:
  Summaries are immutable once written. MEMORY.md is mutable live state. After
  writing a summary, MEMORY.md should be pruned to stay current, not archival.

DOCTRINE UPDATE
================================================================================

Create this new doctrine file:

  context/doctrine/memory-compaction-discipline.md

The file must cover:

1. Compaction Defined
   Compaction in this repo is explicit, file-based, and inspectable. It is NOT hidden
   automation. It means deliberately summarizing session state into a persistent artifact
   at a meaningful boundary.

2. The 3-Tier Memory Structure
   Describe the memory/ directory hierarchy and when each tier serves which purpose.
   Contrast with context/MEMORY.md (local, mutable, task-scoped).

3. Compaction Triggers
   Conditions that should cause an assistant to write a summary or concept:
   - A prompt is paused mid-execution (write PROMPT_XX_resume.md)
   - A prompt completes (write PROMPT_XX_completion.md)
   - A non-obvious finding was resolved that will come up again
   - The context/MEMORY.md is getting bloated and sections are stale

4. Prompt-Boundary Convention
   After every completed or paused prompt, a summary artifact should be placed in
   memory/summaries/. This is the primary source for fresh sessions resuming that work.

5. Anti-Patterns
   - Writing summaries that are just MEMORY.md copies with different names
   - Growing sessions/ without ever promoting findings to concepts/
   - Treating memory/ as a backup of context/MEMORY.md
   - Creating summaries that reference placeholder text or <TODO> markers

6. Integration with work.py
   work.py checkpoint and work.py resume should reference memory/summaries/ when
   relevant. (Full integration comes in PROMPT_92. This doctrine establishes the
   conceptual contract.)

Keep this file tight. It is doctrine, not a tutorial. Target: 80–150 lines.
