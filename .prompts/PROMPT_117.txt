BACKGROUND: KOTLIN, RUBY, AND ELIXIR FRAMING
================================================================================

KOTLIN — Two Design Philosophies on the JVM

Kotlin's position in this arc is unique because it has two distinct validation
approaches with meaningfully different designs:

  Konform — a Kotlin-first DSL. You define validation rules as a lambda using
  an infix DSL. The rules live in your code, not in annotations. You can compose
  validators, reference complex conditions, and return structured error objects.
  This is idiomatic Kotlin: expressive, null-safe, no reflection.

    val validateWorkspace = Validation<WorkspaceConfig> {
        WorkspaceConfig::name {
            minLength(3); maxLength(100)
        }
        WorkspaceConfig::ownerEmail {
            pattern(".+@.+\\..+")
        }
    }

  Hibernate Validator / Jakarta Bean Validation — the JVM ecosystem standard.
  Rules live in annotations on data class fields. This integrates with Spring,
  Quarkus, Jakarta EE. Framework-driven, reflection-based, familiar to Java devs.

    data class WorkspaceConfig(
        @field:NotBlank @field:Size(min = 3, max = 100) val name: String,
        @field:Email val ownerEmail: String,
        @field:Min(1) @field:Max(1000) val maxSyncRuns: Int
    )

  Neither replaces the other. Choose Konform for pure Kotlin code with complex
  rules; choose Hibernate Validator for framework-integrated Spring/Jakarta apps.
  The examples must make this tradeoff explicit — do not reduce Kotlin to
  "just JVM annotation-based."

RUBY — The dry-rb Ecosystem's Intentional Separation

dry-schema and dry-validation are designed to separate two concerns that other
libraries conflate:

  dry-schema: defines the structure and basic type coercion of data. It handles
  "is this field present, what type is it, can it be coerced to the right type?"
  It can also export a JSON Schema-like description of the schema.

  dry-validation: adds semantic business rules on top of a dry-schema. It handles
  "is this email actually valid, is this value within range, does this cross-field
  rule hold?" dry-validation uses a Contract class that wraps a schema.

Using dry-validation without understanding dry-schema misses the intentional
layering. The examples must show both and explain why the separation exists.

ELIXIR — Ecto.Changeset's Cast-Then-Validate Pipeline

Elixir's approach is fundamentally different from all the above:

  Ecto.Changeset is not a schema library. It is a pipeline for casting, changing,
  and validating structured data. A Changeset holds the data, the changes, and
  the accumulated errors. You chain operations:
    |> cast(params, [:name, :slug, :owner_email])
    |> validate_required([:name, :slug])
    |> validate_format(:slug, ~r/^[a-z][a-z0-9-]{1,48}[a-z0-9]$/)
    |> validate_inclusion(:plan, ["free", "pro", "enterprise"])

  Norm is a different design: instead of a pipeline, you define a spec (similar
  to Clojure's spec) and conform data against it. More declarative, less stateful.

  ex_json_schema is the JSON Schema library in Elixir. It validates data against
  a JSON Schema document. This is the contract-first / Lane B path in Elixir:
  you have a schema (possibly generated elsewhere), and you validate data against it.

The Elixir examples must treat all three as serious, well-chosen tools with
distinct design philosophies — not as a tier list of "good → better → best."

  # Kotlin — Konform (Lane A: Kotlin-First DSL Validation)

  ## Lane
  Lane A: rules defined in a Kotlin DSL, not in annotations. Data classes are clean.

  ## The Konform DSL
  4–5 sentences. Validation<T> { } is the entry point. Inside, reference properties
  using WorkspaceConfig::name { ... }. Chain constraint functions: minLength,
  maxLength, pattern, minimum, maximum, maxItems. For cross-field rules, use
  addConstraint with a predicate lambda. Results are ValidationResult<T> — either
  Valid or Invalid with a list of ValidationError.

  ## Comparison with Hibernate Validator
  5–6 sentences. Konform: rules in code, composable validators, Kotlin-idiomatic,
  no reflection overhead, no framework dependency. Hibernate Validator: rules in
  annotations, integrates with Spring/Jakarta EE, familiar to Java developers,
  framework-managed lifecycle. Choose Konform for pure Kotlin codebases and complex
  business rules. Choose Hibernate Validator when using Spring Boot or Jakarta EE.

  ## Cross-Field Rules
  Shown using addConstraint. Konform does not have a native "watch two fields"
  primitive — addConstraint with a predicate that accesses both fields is the
  idiomatic approach.

--- workspace_config_contract.rb ---

  require 'dry-validation'
  require 'dry/validation/contract'

  class WorkspaceConfigContract < Dry::Validation::Contract
    params do
      required(:id).filled(:string)
      required(:name).filled(:string)
      required(:slug).filled(:string)
      required(:owner_email).filled(:string)
      required(:plan).filled(:string, included_in?: %w[free pro enterprise])
      required(:max_sync_runs).filled(:integer)
      required(:settings).hash do
        required(:retry_max).filled(:integer)
        required(:timeout_seconds).filled(:integer)
        required(:notify_on_failure).filled(:bool)
        optional(:webhook_url).maybe(:string)
      end
      optional(:tags).array(:string)
      required(:created_at).filled(:string)
      optional(:suspended_until).maybe(:string)
    end

    rule(:slug) do
      key.failure('must be a valid slug (lowercase, hyphens, digits)') unless
        value.match?(/\A[a-z][a-z0-9-]{1,48}[a-z0-9]\z/)
    end

    rule(:owner_email) do
      key.failure('must be a valid email') unless value.match?(URI::MailTo::EMAIL_REGEXP)
    end

    rule(:max_sync_runs) do
      key.failure('must be between 1 and 1000') unless value.between?(1, 1000)
    end

    rule(:tags) do
      if values[:tags]
        key.failure('must have at most 20 tags') if values[:tags].length > 20
        values[:tags].each_with_index do |tag, i|
          key([:tags, i]).failure('each tag must be 1..50 chars') unless tag.length.between?(1, 50)
        end
      end
    end

    # Cross-field: plan vs max_sync_runs
    rule(:max_sync_runs, :plan) do
      limit = case values[:plan]
              when 'free' then 10
              when 'pro'  then 100
              else 1000
              end
      key(:max_sync_runs).failure("must be <= #{limit} for #{values[:plan]} plan") if
        values[:max_sync_runs] > limit
    end
  end

  # Ruby — dry-validation (Lane A: Rich Rule Contracts)

  ## Lane
  Lane A: runtime validation using a Contract class that wraps a dry-schema.

  ## The dry-rb Ecosystem Design
  4–5 sentences. dry-validation uses dry-schema as its foundation. dry-schema
  handles type coercion and structure: "is this field present, is it the right
  type?" dry-validation adds rule blocks for semantic validation: "is this email
  valid, does this cross-field rule hold?" The separation is intentional —
  use dry-schema alone when you only need structural/type coercion; use
  dry-validation when you also need semantic rules.

  ## Contract Pattern
  The Contract class, params block (schema), and rule blocks. Rules fire after
  the schema passes. If the schema fails, rules are skipped.

  ## Discriminated Union Limitation
  dry-validation does not have native discriminated union support. Manual dispatch
  via rule blocks on the data field. Document this.

  ## Cross-Field Rules
  rule(:field1, :field2) { } — values hash provides access to both fields.

  # Ruby — dry-schema (Lane A/B: Schema Foundation + JSON Schema Output)

  ## Lane
  Primarily Lane A (structural validation + coercion). The JSON Schema export
  adds a Lane B capability.

  ## What dry-schema Does (and Does Not Do)
  4–5 sentences. dry-schema handles: type checking, coercion, presence/absence
  rules, value constraints (min, max, inclusion). It does NOT handle: complex
  cross-field rules, semantic validation rules (is this string a valid email?),
  custom error messages beyond what the predicate system provides. For those,
  wrap the schema in a dry-validation Contract.

  ## The Layering
  dry-schema → dry-validation → application rules.
  Each layer adds power and complexity. Use the lowest layer that meets your needs.

  ## JSON Schema Export
  dry-schema can export a JSON Schema description of its structure. The output
  reflects the schema definition, not the application-level rules in dry-validation.
  A round-trip drift check: does the exported schema accept the same fixtures as
  the runtime schema? Should hold true for structural rules; may diverge for
  application-layer rules.

  defmodule WorkspaceSyncContext.WorkspaceConfig do
    use Ecto.Schema
    import Ecto.Changeset

    @valid_plans ~w[free pro enterprise]

    embedded_schema do
      field :id, :string
      field :name, :string
      field :slug, :string
      field :owner_email, :string
      field :plan, :string
      field :max_sync_runs, :integer
      field :tags, {:array, :string}, default: []
      field :created_at, :utc_datetime_usec
      field :suspended_until, :utc_datetime_usec
      embeds_one :settings, WorkspaceSyncContext.SettingsBlock
    end

    def changeset(config, attrs) do
      config

      max_runs = get_field(changeset, :max_sync_runs)

        _            -> 1000
      end

          "must be <= #{limit} for #{plan} plan")
      else

      end
    end

  # Elixir — Ecto.Changeset (Lane A: Cast-Then-Validate Pipeline)

  ## Lane
  Lane A: runtime validation via a pipeline of cast and validate operations.

  ## What Ecto.Changeset Is
  4–5 sentences. A changeset is a data structure holding: the original data,
  the changes, and any accumulated validation errors. It is NOT a schema in the
  JSON Schema sense — it is a pipeline result. You create one with cast/3, chain
  validation functions on it, and inspect it for validity. This functional
  pipeline style is idiomatic Elixir: pure functions, explicit state, no mutations.

  ## Cast vs Validate: Two Distinct Phases
  3–4 sentences. cast/3 converts raw params (string keys, string values) to the
  struct's typed fields. validate_*/2 then checks constraints. A changeset can
  fail at the cast phase (uncastable type) or the validation phase (constraint
  violated). Understanding this separation is key to debugging Ecto errors.

  ## Cross-Field Rules
  Private functions using get_field and add_error. More verbose than Konform's
  DSL but explicit and readable. The pipeline makes it easy to see the order
  of operations.

  ## Discriminated Union Handling
  Ecto does not have native discriminated union support in the sense of sealed
  classes or tagged unions. Manual dispatch on event_type after cast. Document.

  ## Comparison with Norm
  See elixir/norm/. Norm is declarative and spec-oriented; Ecto.Changeset is
  pipeline-oriented. Choose based on whether you prefer spec-first or pipeline-first.

  # Elixir — Norm (Lane A: Declarative Spec-Oriented Validation)

  ## Lane
  Lane A: declarative specs conforming data. Inspired by Clojure's spec.

  ## What Norm Is
  3–4 sentences. Norm provides schema/1 and spec/1 for defining data shapes
  and constraints. conform/2 returns {:ok, data} or {:error, errors}.
  The mental model is "data conforms to a spec" rather than "run validation
  functions on a changeset."

  ## Comparison with Ecto.Changeset
  4–5 sentences. Ecto.Changeset: stateful pipeline (cast → validate → inspect).
  Norm: stateless spec conformance (define spec → conform data). Ecto integrates
  with Phoenix and database changesets. Norm is more general-purpose and not
  tied to the Ecto ecosystem. Choose Ecto.Changeset for Phoenix web apps;
  choose Norm for libraries, pure-data validation, or when Ecto is not in scope.

  ## Cross-Field Rules
  Norm specs can include arbitrary predicates, so cross-field rules are possible
  but not as ergonomic as Ecto.Changeset's named functions. Document with example.

  defmodule WorkspaceSyncContext.SchemaValidator do
    @moduledoc """
    Lane B: validate data against a JSON Schema document using ex_json_schema.
    This is the contract-first approach: the schema is the source of truth,
    not the application code.
    """

    def load_schema(path) do
      path |> File.read!() |> Jason.decode!() |> ExJsonSchema.Schema.resolve()
    end

    def validate_against_schema(data, schema) do
      case ExJsonSchema.Validator.validate(schema, data) do
        :ok -> {:ok, data}
        {:error, errors} -> {:error, errors}
      end
    end

    def demo do
      schema = load_schema("workspace_config.schema.json")

      # Load valid fixture
      valid_data = "../../domain/fixtures/valid/workspace_config_basic.json"
                   |> File.read!() |> Jason.decode!()
      IO.puts("Valid fixture: #{inspect(validate_against_schema(valid_data, schema))}")

      # Load invalid fixture
      invalid_data = "../../domain/fixtures/invalid/workspace_config_bad_slug.json"
                     |> File.read!() |> Jason.decode!()
      IO.puts("Invalid fixture: #{inspect(validate_against_schema(invalid_data, schema))}")
    end
  end

  # Elixir — ex_json_schema (Lane B: JSON Schema Contract Workflows)

  ## Lane
  Lane B: validate data against a JSON Schema document. Contract-first.

  ## What ex_json_schema Does
  3–4 sentences. ex_json_schema is a JSON Schema validator for Elixir. You
  provide a JSON Schema document and data (as an Elixir map); it returns :ok
  or {:error, errors}. This is the contract-first lane: the JSON Schema is
  the source of truth, not the Elixir code. Use this when you receive a schema
  from an external source or when you want to validate data against a published
  contract.

  ## Use Cases
  - Validating webhook payloads against a published JSON Schema
  - Contract-first API design: schema is published first, implementation conforms
  - Drift detection: validate that the runtime data matches the committed schema
  - Cross-language interoperability: the same JSON Schema works in Elixir,
    Python (jsonschema), Rust (jsonschema crate), TypeScript (Ajv)

  ## Comparison with Ecto.Changeset
  ex_json_schema validates against a JSON Schema document; Ecto.Changeset
  validates against your Elixir schema definition. For most Phoenix app use cases,
  Ecto.Changeset is the right tool. Use ex_json_schema when you need schema-first
  validation or cross-language contract compatibility.
