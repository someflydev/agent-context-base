# PROMPT_META_RUNNER

## Section 1: Purpose
This file is a behavioral contract for one bounded assistant session executing
one `.prompts/` file inside this repository or a generated repo derived from it.
It defines how the assistant should behave inside the session:
- how to boot
- how to stay within prompt scope
- how to group commits
- how to validate work before reporting done
- how to stop cleanly when drift or blockers appear
It is not:
- a multi-session scheduler
- a quota tracker
- a test runner
- a substitute for repo-local runtime state
- an automation layer that decides what prompt comes next
Between sessions, the operator uses the live `scripts/work.py` workflow and
repo-local runtime files to decide sequencing, continuity, and readiness.

## Section 2: Single-Session Invariants
These rules apply in every prompt session.
### 1. One Prompt Per Session
Execute exactly one `.prompts/` file in the session.
- Start with one active prompt.
- End when that prompt reaches its completion condition or a stop condition.
- Do not begin the next prompt in the same session.
### 2. Boot Before Work
Read the repo boot instructions before implementation starts.
Minimum expectation:
- read `AGENT.md` and `README.md` when present
- run `python3 scripts/work.py resume` when the repo exposes `scripts/work.py`
- read `context/TASK.md` and `context/SESSION.md` when present
- read `memory/INDEX.md` and any relevant `memory/summaries/` handoff before
  loading broader task context
- continue into `.acb/` boot files in generated repos when applicable
Do not start by scanning the whole repository.
### 3. Commit Discipline
Every commit created during the session must use the active prompt prefix.
- Commit subjects must begin with `[PROMPT_XX]`.
- Use Tim Pope style multi-line commit messages.
- Use `git add <paths>` when a whole-file grouping is clean.
- Use `git add -p` only when one file contains multiple logical changes that
  must be separated.
- Never batch changes from different prompts in one commit.
### 4. Validation Before Done
A prompt is not complete until its required verification steps are handled.
- Run the verification required by the prompt.
- If tests are not applicable, say so explicitly.
- If tests cannot run, report `blocked` or `incomplete`.
- If tests fail, do not report `done`.
### 5. Explicit Scope
Stay within the active prompt's stated completion condition.
- Touch adjacent files only when required to make the prompt coherent.
- Do not fold opportunistic cleanup into the same session unless the prompt
  requires it.
- Prefer narrow, reviewable changes over "while I am here" expansion.
### 6. No Auto-Sequencing
When the prompt is complete, stop and report.
- Do not open the next prompt.
- Do not infer the next queue item and start it automatically.
- Leave prompt-to-prompt progression to the operator.

## Section 3: Operator Console Integration
This repo uses a two-layer model.
### Operator Layer
The operator manages sequencing and continuity between sessions.
Typical operator responsibilities:
- run the live `work.py` flow to inspect current state
- choose the next prompt
- start, pause, and finish prompt sessions in the local operator console
- review quota or queue health when the repo supports those commands
- decide whether another fresh session should begin
### Assistant Layer
The assistant executes the chosen prompt inside one bounded session.
Typical assistant responsibilities:
- boot from repo instructions
- read runtime state and relevant memory summaries
- implement the prompt
- validate the work
- report `done`, `blocked`, or `incomplete` truthfully
These layers are separate.
- Do not assume `work.py start`, `pause`, `done`, `verify`, `recent-commits`,
  or `log-quota` exist unless the live repo actually provides them.
- Use the commands that exist in the repo you are in.
- Do not invent doctrine around historical prompt assumptions if the code does
  not support them.
Expected operator workflow around a session:
- Before: review runtime state, queue state, and any resume summary
- During: let the assistant execute one prompt; checkpoint when useful
- After: write the prompt completion summary and decide what comes next

## Section 4: Artifact Gates
Before declaring a prompt done, confirm all applicable gates below.
- required files exist
- promised content is present
- placeholders and template stubs are removed
- committed artifacts match the prompt description
- verification steps passed, or are documented as blocked or not applicable
- commit subjects use the correct `[PROMPT_XX]` prefix
- no unintended template overwrite occurred
- the prompt's explicit completion condition is satisfied
If any gate fails, do not report completion.

## Section 5: Commit And File Conventions
- Use real multi-line commit messages, not escaped newline hacks.
- Prefix every commit subject with the active `[PROMPT_XX]`.
- Do not create empty commits.
- Use `git add -p` only when one file must be split into logical groups.
- Do not rename prompt files as part of normal execution.
- Do not overwrite templates or examples accidentally.

## Section 6: Drift Stop Conditions
Stop and report `blocked` or `incomplete` when any of the following occurs:
- an artifact required by the prompt is missing and cannot be resolved safely
- smoke or integration verification fails without a justified reason to defer
- a template or canonical file would be overwritten unintentionally
- the change has drifted beyond the prompt's stated completion condition
- the working session is approaching context limits without a clean finish
When context pressure forces a stop:
- write `memory/summaries/PROMPT_XX_resume.md`
- capture the exact next safe step
- report `incomplete`
- stop instead of improvising a second prompt in the same session

## Section 7: Subagent And Role Conventions
These role names are shorthand for how the assistant is operating.
### AGENT_ARCHITECT
- interprets architecture or doctrine prompts
- declares artifact targets when the prompt is doc-heavy
- clarifies boundaries before implementation
### AGENT_BUILDER
- creates and edits files
- implements scripts, docs, and tests required by the prompt
- keeps the working diff aligned to the completion condition
### AGENT_AUDITOR
- verifies artifacts and commit scope
- runs or reviews required checks
- confirms pass, fail, blocked, or not applicable
### AGENT_REFACTOR
- tightens coherence inside an already-bounded prompt scope
- resolves drift introduced during the current prompt
- does not expand into later prompts
Most sessions use one dominant role. Role changes are allowed, but note them in
the working summary or resume handoff when they matter.

## Section 8: Testing Discipline
Testing requirements depend on what the prompt changes.
When the prompt introduces or changes runtime systems, Docker stacks, APIs,
CLIs, data flows, or significant features:
- run smoke tests before claiming completion
- run minimal integration tests when the prompt explicitly requires them or the
  change obviously crosses real service boundaries
When the prompt is documentation-only, doctrine-only, or workflow-only:
- testing may be limited to validation or consistency checks
- still run any repo-level verification the prompt explicitly requires
Do not claim a prompt is complete from file edits alone when runtime behavior
was part of the requested change.

## Section 9: How This File Is Used
For operators:
- paste this file into a fresh session before the active prompt when you want
  the assistant to follow these conventions explicitly
- use it most often for complex implementation prompts or repo-shaping prompts
- rely on `scripts/work.py` and repo-local runtime state for sequencing between
  sessions

For assistants:

- treat this file as standing operating procedure for the current session
- apply it from the moment it is read until the active prompt ends
- stop at the prompt boundary and hand control back to the operator
