BACKGROUND: WHY A SHARED CORPUS MATTERS
================================================================================

Cross-language validation comparison only means something when every example
is solving the same problem. If Python validates a User object and TypeScript
validates an Order object, you cannot compare ergonomics, error messages, or
library philosophy — you are just looking at different programs.

PROMPT_114 establishes:

  1. A single shared domain (WorkspaceSyncContext) — the same five models
     appear in every language example across all seven languages.

  2. A shared fixture corpus — the same JSON payloads are used in all language
     smoke tests. A valid workspace fixture that Python accepts must be accepted
     by Go, Rust, TypeScript, Kotlin, Ruby, and Elixir (modulo coercion
     differences that are explicitly documented).

  3. A parity matrix — encoding what should behave identically across all
     implementations, what is expected to differ, and why.

  4. A catalog — a navigation index over all planned and implemented examples.

Required content:

  # WorkspaceSyncContext Domain Specification

  ## Overview
  (3–4 sentences) What this domain represents — a workspace-scoped sync system
  where workspaces have configuration, sync jobs run against sources, webhook
  events are emitted, ingestion sources are registered, and review requests can
  be created for content. Designed to surface validation complexity across all
  three lanes.

  ## Design Goals
  A bulleted list of what this domain exercises:
    - Nested objects (WorkspaceConfig → SettingsBlock)
    - Enum / tagged variants (SyncRun.status, WebhookPayload.event_type — discriminated)
    - Optional vs nullable (SyncRun.started_at: null if pending; not absent)
    - Collections with constraints (ReviewRequest.reviewer_emails: 1..5 items)
    - Cross-field validation (SyncRun.finished_at must be after started_at)
    - Coercion / parsing sensitivity (WebhookPayload.timestamp: ISO 8601 string → datetime)
    - Boundary validation (WorkspaceConfig.max_sync_runs: 1..1000)
    - Discriminated union / sum type (WebhookPayload.data varies by event_type)

  ## Model Definitions

  Use a consistent pseudocode format (field: Type | constraints | notes).
  Write every model in full. Do not abbreviate.

  ### SettingsBlock
    retry_max:          integer,   0..10
    timeout_seconds:    integer,   10..3600
    notify_on_failure:  boolean
    webhook_url:        string | null,  valid URL when present, null otherwise

  ### WorkspaceConfig
    id:                 UUID (string, RFC 4122)
    name:               string,    min 3, max 100 chars
    slug:               string,    lowercase letters, digits, hyphens only;
                                   pattern: ^[a-z][a-z0-9-]{1,48}[a-z0-9]$
    owner_email:        string,    valid email address
    plan:               enum(free | pro | enterprise)
    max_sync_runs:      integer,   1..1000
                        cross-field: if plan == "enterprise", max_sync_runs may be 1..1000;
                                     if plan == "free", max_sync_runs must be 1..10;
                                     if plan == "pro", max_sync_runs must be 1..100
    settings:           SettingsBlock (nested, required)
    tags:               list[string], 0..20 items, each item 1..50 chars
    created_at:         datetime (ISO 8601 string), not null, not future-dated
    suspended_until:    datetime | null (ISO 8601 string)

  ### SyncRun
    run_id:             UUID
    workspace_id:       UUID
    status:             enum(pending | running | succeeded | failed | cancelled)
    trigger:            enum(manual | scheduled | webhook)
    started_at:         datetime | null  (null when status == pending)
    finished_at:        datetime | null  (null when status == pending or running)
    duration_ms:        integer | null   (null when not finished; >= 0 when present)
    records_ingested:   integer | null   (null when status != succeeded; 0..10_000_000)
    error_code:         string | null    (present only when status == failed)
    cross-field rules:
      - if finished_at is present, started_at must also be present
      - if finished_at is present, finished_at must be >= started_at
      - if finished_at is present, duration_ms must be present
      - if status == failed, error_code must be present
      - if status != failed, error_code must be null

  ### WebhookPayload
    event_type:         enum(sync.completed | sync.failed | workspace.suspended)
    payload_version:    string, one of: "v1" | "v2" | "v3"
    timestamp:          string, ISO 8601 datetime; parseable from multiple common formats
                        (e.g., "2024-01-15T12:00:00Z", "2024-01-15T12:00:00+00:00")
    signature:          string, lowercase hex, exactly 64 chars (HMAC-SHA256 representation)
    data:               discriminated union — structure depends on event_type:
      when event_type == "sync.completed":
        SyncCompletedData { run_id: UUID, workspace_id: UUID, duration_ms: int >= 0,
                            records_ingested: int 0..10_000_000 }
      when event_type == "sync.failed":
        SyncFailedData { run_id: UUID, workspace_id: UUID, error_code: string,
                         error_message: string max 500 chars }
      when event_type == "workspace.suspended":
        WorkspaceSuspendedData { workspace_id: UUID, suspended_until: datetime,
                                 reason: enum(policy_violation | payment_failure | manual) }

  ### IngestionSource
    source_id:          UUID
    source_type:        enum(http_poll | webhook_push | file_watch | database_cdc)
    config:             discriminated union on source_type:
      http_poll:        HttpPollConfig { url: string (valid URL), method: enum(GET|POST),
                                         headers: map[string, string] 0..20 entries }
      webhook_push:     WebhookPushConfig { path: string starts with "/", secret: string }
      file_watch:       FileWatchConfig { path: string, pattern: string (glob) }
      database_cdc:     DatabaseCdcConfig { dsn: string, table: string, cursor_field: string }
    enabled:            boolean
    poll_interval_seconds: integer | null
      cross-field: if source_type == http_poll, poll_interval_seconds must be present >= 60;
                   if source_type != http_poll, poll_interval_seconds must be null

  ### ReviewRequest
    request_id:         UUID
    workspace_id:       UUID
    reviewer_emails:    list[string], 1..5 items, each a valid email, no duplicates
    content_ids:        list[UUID], 1..50 items, no duplicates
    priority:           enum(low | normal | high | critical)
    due_at:             datetime | null
      cross-field: if priority == critical, due_at must be present
    notes:              string | null, max 2000 chars when present

  ## Library-Specific Considerations

  A section noting where language/library behavior will legitimately diverge:

    - Coercion: Some libraries parse "2024-01-15T12:00:00Z" to a datetime automatically
      (Pydantic, Ecto.Changeset); others require an explicit parse step (Go, Rust validator).
      Document this in each example; do not fail parity rules for it.

    - Nullable vs optional: In some languages, null and absent are the same (Ruby, Elixir).
      In others (TypeScript, Rust Option<T>), they are distinct. Examples must handle this
      explicitly and the PARITY.md rules must note the expected divergence.

    - Error accumulation: Pydantic, Zod, Konform, and dry-validation accumulate multiple
      errors. Go's validator and Rust's validator may return all errors or only the first.
      Both behaviors are valid; the smoke tests must not require a specific error count.

    - Discriminated unions: Rust (enums), TypeScript (discriminated union types), Kotlin
      (sealed classes) have native constructs. Go and Ruby require more manual dispatch.
      Examples must show the idiomatic approach for each language.
