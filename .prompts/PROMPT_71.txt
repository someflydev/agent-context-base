You are working inside the agent-context-base repository.

This prompt sharpens the testing discipline for derived repos. PROMPT_70 established how tools
are invoked. This prompt establishes what tests to write, how to structure them, and what to
avoid — particularly: no mocks at the integration layer, no CI infrastructure until explicitly
asked, and a clear split between unit tests (pure logic) and integration tests (real services
via docker-compose.test.yml).

The failure patterns this addresses:
  - Integration tests that mock the database and therefore never catch real schema drift
  - Pytest fixtures that spin up an in-memory SQLite fake instead of the Dockerized Postgres
  - CI (GitHub Actions) scaffolded early, breaking on every push before the stack is stable
  - Assistants writing "unit tests" for route handlers that require real storage to mean anything
  - Overly broad test suites that are slow, flaky, and never run because docker isn't up

The goal: derived repos should have a small, honest test suite that proves the boundaries work.
Fast unit tests for pure logic. Real-infra integration tests for the storage/queue/search
boundary. Nothing in between that pretends to be integration but isn't.

Apply changes in this order.

3A. Update: context/doctrine/testing-philosophy.md
  This is the authoritative testing doctrine. Expand it to cover the gaps from 2A.
  The updated doc should include all of the following sections:

  "Default Test Layers" (already exists — keep, verify it is still accurate)

  "Unit Tests: When and What"
    - Pure logic that does not require real infra: validators, data transforms, algorithms
    - Parameterized edge cases where you want fast coverage of many inputs
    - Correct: test a filter-query builder function with 10 inputs/outputs
    - Wrong: test a route handler as a "unit test" when it needs a real DB to mean anything

  "Integration Tests: When and What"
    - Any change that crosses a real storage, queue, search, or service boundary
    - Must use Docker-backed infra via docker-compose.test.yml — never mocked storage
    - Must include at least one write + read cycle to confirm the boundary works
    - Required when a change touches: DB writes/reads, cache reads/writes, message pub/sub,
      search index or retrieval, vector index or retrieval, service-to-service calls

  "The Mock Anti-Pattern"
    - Mocking the database in an integration test produces a test that cannot catch:
      schema drift, migration errors, query planner differences, constraint violations
    - Unit tests may use mocks/stubs for isolated logic. Integration tests must not.
    - Specific anti-patterns to name:
        sqlite in-memory as a stand-in for Postgres
        unittest.mock patching the storage adapter in an integration suite
        a fake message broker that doesn't exercise the real pub/sub contract

  "Test File Layout"
    Document the canonical layout: tests/unit/, tests/integration/, tests/e2e/
    State that each directory should have its own README if the suite is non-trivial.

  "Running Integration Tests"
    The docker-compose.test.yml stack must be up before integration tests run.
    Document the standard flow:
      docker compose -f docker-compose.test.yml up -d
      <wait for health checks>
      <run integration tests>
      docker compose -f docker-compose.test.yml down
    The README or a Makefile target in the derived repo should document this flow.

  "CI: Do Not Add Until Asked"
    GitHub Actions (or any other CI platform) should NOT be added to a derived repo
    until the operator explicitly requests it. Reasons:
      - CI adds maintenance surface before the test suite is stable
      - Broken CI on every push discourages good test hygiene early
      - The right moment for CI is when the integration tests reliably pass locally
    This rule applies to: .github/workflows/, Makefile CI targets, any other automation
    that pushes or runs on merge. Local Makefile helpers are fine.

  "Smoke Tests vs Integration Tests"
    Smoke: confirms the app boots and the happy path responds — no docker needed if
    the app can start without real infra (use test env vars to skip infra on boot).
    Integration: proves a real boundary works — always needs docker-compose.test.yml.
    Never conflate the two. A smoke test that requires full infra to be up is not a smoke test.
