Create: docs/schema-validation-drift-detection.md

This guide explains how to detect drift between a library's runtime validation
behavior and its exported JSON Schema. This is doctrine Rule 7 made operational.

Required sections:

  # Schema Drift Detection Guide

  ## What Is Drift?
  (3–4 sentences) When the runtime validation behavior and the exported JSON Schema
  diverge, consumers of the schema receive incorrect expectations. Drift can happen
  when a field is added but the schema is not regenerated, or when a serde attribute
  changes and shifts the wire representation without updating the schema.

  ## When Drift Matters
  Bullet list:
    - When a JSON Schema or OpenAPI spec is published for external consumers
    - When cross-language teams share a contract and one side changes independently
    - When the schema is committed to version control as a baseline artifact
    - When the contract is used to generate client code in another language

  ## Detection Strategies

  ### 1. Round-Trip Fixture Check (simplest, always applicable)
  Procedure:
    a. Serialize a known-valid domain object to JSON
    b. Validate the JSON against the exported schema using a JSON Schema validator
    c. Assert it passes
    d. Take a known-invalid domain object (from fixtures/invalid/)
    e. Validate against the exported schema
    f. Assert it fails

  This catches gross drift (missing fields, wrong types, wrong required fields).
  It does NOT catch cross-field rule drift (the schema cannot express all rules).

  ### 2. Committed Schema Baseline + Regeneration Diff (for CI)
  Procedure:
    a. Commit the exported schema to version control (e.g., workspace_config.schema.json)
    b. Add a CI step that regenerates the schema (e.g., cargo run --bin schema_export)
    c. Diff the freshly generated schema against the committed baseline
    d. Fail CI if they differ

  This catches: serde attribute changes, field additions/removals, type changes.
  Shown in: rust/serde-schemars/ (the committed workspace_config.schema.json)
             go/openapi-generation/ (the committed JSON Schema document)

  ### 3. Parity Matrix Runner (cross-language drift)
  When the same fixture corpus is used across multiple language implementations,
  the parity matrix runner catches behavioral drift between languages. See
  verification/schema-validation/run_parity_check.py (created in PROMPT_118).

  ## Per-Language Notes

  PYTHON (Pydantic):
    model_json_schema() is always in sync with the model — it derives from the
    same class definition. The only drift risk is if you hand-edit the schema after
    generation. Use the committed baseline strategy if publishing externally.

  TYPESCRIPT (TypeBox):
    TypeBox schemas ARE JSON Schema. Drift risk is minimal within a single codebase.
    Risk increases if you hand-modify the schema or if the Ajv version changes
    schema interpretation behavior. Pin your Ajv version.

  GO:
    High drift risk because the contract is typically maintained separately from
    the struct definitions. Commit the schema baseline and add CI regeneration.
    Swaggo annotations are comments — the compiler does not verify them.

  RUST (serde + schemars):
    Moderate drift risk. schemars reads serde attributes — if you change
    #[serde(rename = "...")] or #[serde(skip_serializing_if = "...")], the
    generated schema changes. Commit the schema baseline. Run cargo test
    with a test that regenerates and compares.

  ELIXIR (ex_json_schema):
    The schema is external (a committed JSON file). Drift occurs when the Ecto
    changeset changes but the schema is not updated. Round-trip fixture check is
    the primary strategy.

  ## The Cross-Field Rule Gap
  (3–4 sentences) JSON Schema (draft 7 and later) can express some cross-field
  rules using if/then/else. However, many domain rules cannot be expressed in
  JSON Schema at all (e.g., "if plan == free then max_sync_runs <= 10"). For these
  rules, the schema exports a relaxed constraint, and runtime validation enforces
  the stricter rule. This is expected and should be documented in the schema's
  $comment field.

Create: verification/schema-validation/run_parity_check.py

This is the parity matrix runner. It checks whether the Python implementations
(the ones that run without a separate toolchain) accept and reject the expected
fixtures according to PARITY.md's universally-required rules.

Note: Only Python-based checks run here. Go, Rust, Kotlin, Ruby, and Elixir
implementations are checked via their toolchain-specific test runners. Document
this scope boundary explicitly in the script.

Implementation:

  #!/usr/bin/env python3
  """
  Parity matrix runner for the schema-validation arc.

  Checks universally-required behavior (PARITY.md rules 1-7) against Python
  implementations (Pydantic, marshmallow). These run without a separate toolchain.

  For Go/Rust/Kotlin/Ruby/Elixir parity checks, see the toolchain-specific test
  runners in verification/schema-validation/{language}/.

  Exit code 0: all checks pass.
  Exit code 1: one or more checks failed.
  """

  import sys
  import json
  import pathlib
  import importlib

  FIXTURES_DIR = pathlib.Path(__file__).parent.parent.parent / \
      "examples" / "canonical-schema-validation" / "domain" / "fixtures"

  VALID_FIXTURES = list((FIXTURES_DIR / "valid").glob("*.json"))
  INVALID_FIXTURES = list((FIXTURES_DIR / "invalid").glob("*.json"))

  results = []

  def load_fixture(path):
      with open(path) as f:
          return json.load(f)

  def check_pydantic():
      """Rule 1: all valid fixtures must be accepted. Rule 2: all invalid rejected."""
      try:
          from examples.canonical_schema_validation.python.pydantic.models import (
              WorkspaceConfig, SyncRun, WebhookPayload, IngestionSource, ReviewRequest
          )
      except ImportError:
          results.append(("pydantic", "SKIP", "pydantic not importable"))
          return

      MODEL_MAP = {
          "workspace_config": WorkspaceConfig,
          "sync_run": SyncRun,
          "webhook_payload": WebhookPayload,
          "ingestion_source": IngestionSource,
          "review_request": ReviewRequest,
      }

      for fixture_path in VALID_FIXTURES:
          data = load_fixture(fixture_path)
          model_name = next((k for k in MODEL_MAP if k in fixture_path.stem), None)
          if model_name is None:
              continue
          model_cls = MODEL_MAP[model_name]
          try:
              model_cls.model_validate(data)
              results.append(("pydantic", "PASS", f"valid/{fixture_path.name}"))
          except Exception as e:
              results.append(("pydantic", "FAIL",
                  f"valid/{fixture_path.name} rejected: {e}"))

      for fixture_path in INVALID_FIXTURES:
          data = load_fixture(fixture_path)
          model_name = next((k for k in MODEL_MAP if k in fixture_path.stem), None)
          if model_name is None:
              continue
          model_cls = MODEL_MAP[model_name]
          try:
              model_cls.model_validate(data)
              results.append(("pydantic", "FAIL",
                  f"invalid/{fixture_path.name} was accepted (should be rejected)"))
          except Exception:
              results.append(("pydantic", "PASS", f"invalid/{fixture_path.name} rejected"))

  def check_marshmallow():
      """Same checks for marshmallow schemas."""
      try:
          from examples.canonical_schema_validation.python.marshmallow.schemas import (
              WorkspaceConfigSchema, SyncRunSchema
          )
      except ImportError:
          results.append(("marshmallow", "SKIP", "marshmallow not importable"))
          return

      # Run workspace_config and sync_run fixtures only
      for fixture_path in VALID_FIXTURES:
          if "workspace_config" in fixture_path.stem:
              data = load_fixture(fixture_path)
              errors = WorkspaceConfigSchema().validate(data)
              if errors:
                  results.append(("marshmallow", "FAIL",
                      f"valid/{fixture_path.name}: {errors}"))
              else:
                  results.append(("marshmallow", "PASS", f"valid/{fixture_path.name}"))

      for fixture_path in INVALID_FIXTURES:
          if "workspace_config" in fixture_path.stem:
              data = load_fixture(fixture_path)
              errors = WorkspaceConfigSchema().validate(data)
              if errors:
                  results.append(("marshmallow", "PASS", f"invalid/{fixture_path.name} rejected"))
              else:
                  results.append(("marshmallow", "FAIL",
                      f"invalid/{fixture_path.name} accepted (should be rejected)"))

  if __name__ == "__main__":
      check_pydantic()
      check_marshmallow()

      passes = [r for r in results if r[1] == "PASS"]
      fails  = [r for r in results if r[1] == "FAIL"]
      skips  = [r for r in results if r[1] == "SKIP"]

      print(f"\nParity Check Results: {len(passes)} PASS, {len(fails)} FAIL, {len(skips)} SKIP\n")
      for impl, status, msg in results:
          print(f"  [{status}] {impl}: {msg}")

      if fails:
          print(f"\n{len(fails)} parity check(s) FAILED.")
          sys.exit(1)
      else:
          print("\nAll parity checks PASSED.")
          sys.exit(0)
