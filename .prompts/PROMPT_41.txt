SECTION 2 — What the companion document must cover

The companion `duckdb-polars-example.md` must answer these questions concretely:

**When is this the right example?**

This is the right example when:
- the analytical data lives in a local DuckDB file under `docker/volumes/dev/`
- the output of the query needs to be a Polars DataFrame for downstream transformation
- the workload is development, testing, or local analytical pipelines — not a deployed
  service reading from an object store
- there is no MinIO or S3 bucket involved in the read path

Do not use this example when:
- the data lives in MinIO or S3; use `parquet-minio-example.py read_back()` for that
- the workload is a production ETL pipeline reading from object storage
- the goal is to show Trino federation; use `trino-federated-analytics-example.sql`

**Database path convention**

The `DATABASE_PATH` constant follows the repository's volume layout:
```
docker/volumes/dev/warehouse/analytics.duckdb
```
Test workloads should use a separate path under `docker/volumes/test/` and never share
the dev DuckDB file. The path should be passed as a parameter or resolved from an
environment variable in repos where the distinction matters.

**What `pl.read_database()` returns and why**

Explain that `pl.read_database()` converts the DuckDB result directly to a Polars
DataFrame, making it composable with the Polars transformation layer that typically
follows. Contrast this with `conn.execute().fetchall()`, which returns raw Python
tuples and loses schema information.

**Relationship to `parquet-minio-example.py`**

These two examples cover the two canonical DuckDB read paths in this repository:

| Example | Data lives | DuckDB access mode | Output |
|---|---|---|---|
| `duckdb-polars-example.py` | Local file (`docker/volumes/dev/`) | Embedded, file path | Polars DataFrame |
| `parquet-minio-example.py read_back()` | MinIO/S3 bucket | S3 extension via httpfs | Row count scalar |

When a repo uses the parquet-minio write path AND needs to transform the result into
a Polars DataFrame, compose both: run `read_back()` for the S3 read, then pass the
result to Polars if needed. Do not use `duckdb-polars-example.py` as the S3 read path.
