--------------------------------------------------------------------------------
CANONICAL FILE ROLES
--------------------------------------------------------------------------------

Implement and document the roles of these files carefully:

1. `PLAN.md`
   - milestone-level roadmap
   - larger phases, subtracks, or multi-session plan
   - should be updated when a specific megaprompt from `.prompts` materially reshapes the current plan, introduces a new phase, or changes milestone structure
   - should NOT be updated for every tiny implementation step

2. `context/TASK.md`
   - current active slice of work
   - concrete scope, success criteria, immediate steps, blockers
   - should reflect the current subtask or active implementation unit

3. `context/SESSION.md`
   - compact current working state
   - recent decisions, relevant files, current status, next safe step, what not to reload
   - must stay concise and action-oriented

4. `context/MEMORY.md`
   - durable repo-local truths
   - assistant behavior expectations, stable conventions, recurring guardrails, known pitfalls
   - should NOT accumulate temporary task sludge

Design the workflow and docs so these distinctions are explicit and easy to follow.

--------------------------------------------------------------------------------
EXPECTED `work.py` BEHAVIOR
--------------------------------------------------------------------------------

`resume`
- detect repo root safely
- inspect whether canonical files exist
- determine whether state appears stale relative to repo changes
- summarize the current working situation in a concise, useful way
- guide a fresh assistant session toward the correct next reads/actions
- help humans and assistants quickly answer: “where are we and what should happen next?”

`checkpoint`
- inspect repo state
- determine whether TASK / SESSION / MEMORY / PLAN likely need refresh or not
- update or scaffold runtime files appropriately
- use simple, grounded heuristics to warn when active context appears too large, stale, or drifting
- produce a better continuation surface for the next session

`init` (optional, if appropriate)
- scaffold the expected local runtime files from templates/examples if missing
- be safe and idempotent

`status` (optional)
- summarize current workflow state without mutating files

You do not need to implement every imaginable feature. Favor minimal usefulness over elaborate machinery.

--------------------------------------------------------------------------------
STATE DETECTION + COMPACTION HEURISTICS
--------------------------------------------------------------------------------

This is a central part of the design.

Because subscription coding assistants generally cannot reliably expose precise context-window usage, define “active context too large” operationally using visible proxies.

Use practical heuristics such as:
- line counts
- byte size
- staleness relative to git changes
- number of changed files since last checkpoint
- whether `context/SESSION.md` lacks a clear next safe step
- whether `context/TASK.md` looks empty/stale compared to active work
- whether `PLAN.md` likely needs update after a major `.prompts`-driven phase shift
- whether `context/MEMORY.md` appears polluted with short-lived details
- whether current working state is too broad or too verbose for a clean assistant boot sequence

Do not fake token counts. Do not pretend to know model internals. Use grounded repo-local measurements.
