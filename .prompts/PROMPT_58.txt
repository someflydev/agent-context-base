Content notes:

When gRPC IS the right choice:
- Schema enforcement across language boundaries is required — the .proto file is the
  canonical contract; breaking changes are caught at stub generation time
- Synchronous results are required (caller waits for typed response)
- Performance matters for the call itself: proto binary encoding is more compact and faster
  to serialize/deserialize than JSON, especially for repeated numeric fields
- Streaming is needed: server-streaming, client-streaming, or bidirectional streaming RPCs
  are all expressible in proto and handled by gRPC natively
- The service pair includes Rust (tonic), Go (google.golang.org/grpc), Kotlin (grpc-kotlin),
  or Scala (ScalaPB) — all have excellent gRPC support

When gRPC is NOT the right choice:
- Simple request/response where JSON readability is preferred → REST with FastAPI or Echo
- The caller is a browser — gRPC-Web is possible but adds complexity; use REST or GraphQL
- Both services are the same language and the call is in-process → function call
- The schema will change frequently in early development — proto schema evolution is
  strict (field numbers must be stable); REST with JSON is more forgiving at the cost
  of runtime errors instead of build-time errors
- Team is unfamiliar with protoc and does not want to manage stub generation steps

Proto file organization:
  - All .proto files live in docs/seam-contract/ or a shared proto/ directory at the repo root
  - Package naming: {domain}.{version} — e.g., inference.v1, compute.v1, processing.v1
  - go_package option: required for Go stub generation; use your module path
  - One service per .proto file; keep the file small and focused
  - Add API-level comments to every service, RPC, message, and field
  - Never reuse field numbers — once a field is added, its number is permanent
  - Use reserved to document removed fields: reserved 4; reserved "old_field";
  - Always start field numbers at 1; never use 19000–19999 (reserved by proto)

Show a well-structured example proto with comments:
```protobuf
// docs/seam-contract/inference.proto
syntax = "proto3";
package inference.v1;

// go_package required for Go codegen
option go_package = "github.com/example/myapp/gen/inference/v1;inferencev1";

// InferenceService handles ML model prediction requests.
service InferenceService {
  // Predict runs a single inference over the provided features.
  rpc Predict(PredictRequest) returns (PredictResponse);

  // PredictStream accepts a stream of requests and returns a stream of responses.
  // Use for batched inference where client sends multiple requests.
  rpc PredictStream(stream PredictRequest) returns (stream PredictResponse);
}

// PredictRequest carries the feature vector for a single inference.
message PredictRequest {
  // model_id identifies which model to use. Required.
  string model_id = 1;

  // features is the numeric feature vector. Required; must be non-empty.
  repeated float features = 2;

  // correlation_id is propagated for distributed tracing. Optional.
  string correlation_id = 3;
}

// PredictResponse carries the inference result.
message PredictResponse {
  // label is the predicted class name.
  string label = 1;

  // confidence is the probability of the predicted class (0.0–1.0).
  float confidence = 2;

  // duration_ns is the server-side inference duration in nanoseconds.
  int64 duration_ns = 3;
}
```

RPC method types:
  Unary: rpc Predict(Request) returns (Response)
    - One request, one response. Use for synchronous query/response patterns.
  Server-streaming: rpc Scan(ScanRequest) returns (stream ScanResult)
    - One request, stream of responses. Use for paginated results or progress updates.
  Client-streaming: rpc Upload(stream Chunk) returns (UploadResult)
    - Stream of requests, one response. Use for file upload or batched writes.
  Bidirectional streaming: rpc Chat(stream Msg) returns (stream Msg)
    - Full-duplex stream. Use for real-time protocols (voice, game state).
  For most seam patterns in this repo: use unary. Only add streaming when the use case
  requires it — streaming adds complexity to both sides.

Error handling:
  gRPC status codes to know:
  - OK (0): success
  - INVALID_ARGUMENT (3): caller sent bad input — the server should validate and return this
  - NOT_FOUND (5): entity not found
  - ALREADY_EXISTS (6): conflicts with existing state
  - PERMISSION_DENIED (7): caller authenticated but not authorized
  - UNAUTHENTICATED (16): caller not authenticated
  - RESOURCE_EXHAUSTED (8): rate limit or quota exceeded
  - INTERNAL (13): unexpected server error — do not expose internal details
  - UNAVAILABLE (14): server is temporarily unavailable (safe to retry with backoff)
  - DEADLINE_EXCEEDED (4): the call deadline elapsed before completion

  In Rust (tonic): return Err(Status::invalid_argument("features must not be empty"))
  In Go: return nil, status.Errorf(codes.InvalidArgument, "features must not be empty")
  In Python: raise grpc.RpcError (context.abort(grpc.StatusCode.INVALID_ARGUMENT, "..."))

Deadlines and timeouts:
  Always set a deadline on the caller side. A missing deadline means a hung downstream
  can block the caller indefinitely.
  Go: ctx, cancel := context.WithTimeout(ctx, 5*time.Second); defer cancel()
  Python: stub.Predict(request, timeout=5.0)
  Kotlin: stub.withDeadlineAfter(5, TimeUnit.SECONDS).predict(request)

Health checking standard:
  Use grpc.health.v1.Health/Check — the standard health checking protocol. Both sides
  should implement it.

  Proto import: google/health/v1/health.proto (available in grpc-health-probe repo)

  Alternatively: expose an HTTP /healthz alongside the gRPC server. This is simpler and
  avoids requiring grpc_health_probe in the container. Rust (tonic) can run an Axum
  HTTP server on a separate port alongside the gRPC server using tokio::spawn.

  docker-compose health check without grpc_health_probe:
  ```yaml
  healthcheck:
    test: ["CMD", "curl", "-f", "http://localhost:8082/healthz"]
    interval: 10s
    timeout: 5s
    retries: 3
    start_period: 20s
  ```

Reflection and grpc-gateway:
  gRPC server reflection (grpc.reflection.v1alpha.ServerReflection): allows tools like
  grpcurl and grpc-ui to introspect the server without the .proto file.
  - Rust (tonic): add tonic-reflection crate; register the service at startup
  - Go: add google.golang.org/grpc/reflection; register at startup
  - Enable in dev; consider disabling in production for security

  grpc-gateway: generates an HTTP/JSON reverse proxy from the proto file, allowing REST
  clients to call gRPC services. Useful when a browser or REST-only client must reach
  a gRPC service. Add google.api.http annotations to the proto methods.

Testing expectations:
- Test stub generation: run protoc and verify generated files compile in each target language
- Prove one unary RPC round-trip with a real server (no mocks)
- Prove error handling: call with invalid input, verify the correct gRPC status code is returned
- Prove deadline handling: make a request to a slow server, verify DEADLINE_EXCEEDED is returned
- Prove health check: call the /healthz endpoint or Health/Check RPC, verify 200/OK
- Do NOT mock the gRPC server in integration tests — the stub generation and wire behavior is what's tested
- Keep test protos separate from production protos if testing error injection

Common assistant mistakes:
- Not setting go_package option in the proto file — Go codegen fails without it
- Reusing or skipping field numbers — this breaks backward compatibility silently for binary proto
- Not setting a deadline on the caller — any hung server causes the caller to block indefinitely
- Using INTERNAL status code for all errors — use the most specific status code available
- Forgetting to run protoc after changing the proto — stubs become stale
- Not exposing an HTTP health endpoint alongside gRPC — docker-compose health checks are simpler
  with curl than with grpc_health_probe (which requires installing the tool in the container)
- Running Rust tonic server on a single async runtime thread — use tokio::main with multi-thread
