BACKGROUND
================================================================================

Python goes first in this arc for two reasons:
  1. The domain/generation_patterns.py module (PROMPT_120) is already a Python
     reference implementation. The canonical example extends it into a full
     deliverable with CLI, three library demonstrations, and output profiles.
  2. Python is the only language in this arc that can be smoke-tested without
     a separate toolchain install. All arc-level parity checks run against
     Python output (see PROMPT_126).

The three Python library surfaces cover meaningfully different positions:
  - Faker: imperative provider calls; the standard "call a method" approach
  - Mimesis: schema-driven, typed, faster; shows a different generation idiom
  - factory_boy: factory declarations + lazy evaluation; ORM-oriented, but
    useful for showing how a declarative object graph layer sits on top of faker

The three examples must generate the same TenantCore domain entities.
Their output JSONL files must be structurally identical (same keys, same schema).

PART 4: mimesis_pipeline/generators.py
================================================================================

Create: examples/canonical-faker/python/mimesis_pipeline/generators.py

This module implements the same TenantCore generation pipeline using Mimesis.

Key Mimesis APIs to demonstrate:
  - from mimesis import Person, Address, Internet, Datetime, Text, Finance
  - from mimesis import Generic (multi-provider entry point)
  - from mimesis.locales import Locale
  - from mimesis.schema import Schema, Field — show schema-based generation

Mimesis-specific demonstrations:
  1. Schema-based generation: define the organization entity using
     mimesis.Schema and mimesis.Field, showing the schema-as-code pattern
     that Mimesis specializes in.
     Example:
       field = Field(locale=Locale.EN, seed=profile.seed)
       schema = Schema(schema=lambda: {
           "id": field("uuid"),
           "name": field("company"),
           ...
       }, iterations=profile.num_orgs)
       orgs = schema.create()

  2. Custom field: add a custom provider that generates TenantCore-specific
     enums (plan, region, role) using mimesis.providers.BaseProvider.

  3. Batched generation: demonstrate mimesis's built-in iteration support.
     Mimesis Schema.create(iterations=N) is faster than a Python for loop
     calling faker methods N times. Document the performance note in the README.

The module must expose: run_mimesis_pipeline(profile: Profile, output_dir: Path)

The relational graph layer (memberships, projects, audit_events) must follow
the same 7-stage ordering as generation_patterns.py, even if atomic field
generation uses Mimesis instead of Faker.

IMPORTANT: Mimesis alone does not handle relational integrity. The graph layer
(parent-first ordering, ID pools, cross-field rules) must be implemented
explicitly on top of Mimesis, just as it is in the Faker pipeline.
Add a comment at the top of the file: "Mimesis generates realistic atomic
values. This module adds the relational graph layer on top."

PART 5: factory_graph/factories.py
================================================================================

Create: examples/canonical-faker/python/factory_graph/factories.py

This module shows factory_boy as a declarative object-graph layer.
factory_boy does not replace the generation pipeline — it provides a
declarative syntax for defining entity construction patterns.

Key demonstration points:
  1. Define factory classes for Organization, User, Membership, Project,
     AuditEvent using factory.Factory (not DjangoModelFactory — no ORM).

  2. Show lazy attributes: joined_at must be after org.created_at.
     Use factory.LazyAttribute.

  3. Show SubFactory: membership.user = factory.SubFactory(UserFactory).

  4. Show Sequence: unique email via factory.Sequence(lambda n: f"user_{n}@example.com")

  5. Show Trait: an "enterprise_org" trait that sets plan="enterprise" and
     generates a larger initial member count.

  6. Show batch creation: UserFactory.create_batch(10, seed=42) — note that
     factory_boy's "seed" concept requires use_seed() on the Faker instance
     inside the factory.

IMPORTANT scope note: factory_boy is useful for object graph declaration.
It is NOT ideal for large-volume generation (it creates Python objects, not
raw dicts, and the overhead is meaningful at 50,000+ rows). Add a note in
the module docstring: "Use factory_boy for smoke/small profiles. For medium/
large profiles, use the faker_pipeline or mimesis_pipeline directly."

factory_graph/runner.py must:
  - Run the factory graph for the smoke profile
  - Collect output as dicts (not ORM objects)
  - Write to output_dir using write_jsonl from domain/generation_patterns.py
  - Run validate_dataset
  - Print the validation report
