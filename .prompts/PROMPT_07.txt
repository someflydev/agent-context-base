You are designing a deterministic “Context Boot Sequence” for coding assistants working inside a repository derived from the agent-context-base architecture.

The purpose of the Context Boot Sequence is to make assistants behave like a predictable runtime environment when they begin operating in the repository.

Instead of randomly scanning files or loading large amounts of context, the assistant should follow a well-defined initialization procedure.

This boot sequence should function similarly to how an operating system initializes a process environment.

The boot sequence must ensure that assistants:

- understand the architecture of the repository
- identify the correct context layers
- infer the correct stack and archetype
- load only the smallest relevant context bundle
- consult canonical examples before implementing code
- respect doctrine constraints
- avoid introducing new patterns unnecessarily
- preserve repo-derived Compose `name:` values, non-default ports, and test-data isolation when Docker-backed local infrastructure is relevant

The boot sequence must be deterministic, minimal, and reliable.

The architecture of the repository includes the following layers:

Context Layers
- doctrine
- workflows
- stacks
- archetypes
- routers
- manifests
- examples
- templates

Operational Scripts
- context lint
- manifest validator
- context bundle preview
- pattern diff analyzer
- prompt analyzer

Routing Intelligence
- task router
- stack router
- archetype router
- alias catalog
- repo signal detection

Knowledge Artifacts
- canonical examples
- prompt-first examples
- fixture datasets
- observability examples

Deployment Philosophy
- Dokku-oriented deployment
- simple deployable services
- smoke-test verification
- repo-derived Compose `name:` values and isolated primary/dev versus test topologies

Your task is to design the Context Boot Sequence specification that assistants must follow when entering the repository.

The output should function as both:

1. a conceptual explanation of the boot sequence
2. a practical implementation document that can live inside the repository
