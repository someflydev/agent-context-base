Content notes:

When RabbitMQ IS the right choice:
- Task distribution: a message must be delivered to exactly one worker in a pool (work queue semantics)
- Routed messaging: messages need content-based routing by routing key to different queues
  (topic exchange with pattern bindings, e.g., *.orders.us goes to one queue, *.orders.eu to another)
- Priority queues: urgent tasks must preempt normal tasks in a queue
- Request/reply patterns: caller publishes to a queue and waits for a response on a reply-to queue
  (RPC-over-AMQP pattern; use sparingly — prefer gRPC for sync calls)
- Teams with existing AMQP infrastructure or where AMQP client libraries are already in use
- The workflow already uses AMQP conventions (dead-letter-exchange, ttl, priority) that map directly
- Elixir is a producer or consumer — Broadway has first-class AMQP support via broadway_rabbitmq

When RabbitMQ is NOT the right choice:
- High-throughput log-style event streams where multiple independent consumer groups need replay
  → use Kafka (log append semantics, offset-based replay)
- Lightweight subject-routed pub/sub without queue semantics → use NATS JetStream
- The repo already runs Redis and volume is modest → use Redis Streams
- Fan-out to many consumers where each consumer gets a copy (fanout exchange works, but
  NATS subject wildcards are simpler for most fan-out use cases)
- Cross-team schema contracts requiring Avro or Protobuf with a registry → use Kafka

Core concepts:

Exchange types:
  - direct: routes messages whose routing key exactly matches the queue binding key
    Example: routing key "order.created" → queue "order-processor"
  - topic: routes messages by routing key pattern matching (* matches one word, # matches zero or more)
    Example: routing key "us.orders.*" → queue "us-order-handler"
                         "*.orders.#" → queue "all-orders-monitor"
  - fanout: broadcasts to all bound queues regardless of routing key
    Example: "notifications" fanout exchange → email-queue, sms-queue, push-queue
  - headers: routes by message header attributes (rarely needed; use topic instead)

Message schema and payload convention:
  Every RabbitMQ message in this repo should include the following in the JSON body:
  {
    "payload_version": 1,
    "correlation_id": "req-abc123",
    "published_at": "2026-03-16T10:30:00Z",
    "routing_key": "domain.orders.created",
    "entity_id": "ord-001",
    "tenant_id": "acme",
    "data": { ... }
  }
  Include payload_version and correlation_id on every message — same convention as NATS/Kafka.
  The routing_key field in the body should mirror the AMQP routing key used for publishing.

Publisher patterns — Python (aio-pika):
```python
# pip install aio-pika
import aio_pika
import json
import asyncio

async def publish_event(amqp_url: str, exchange_name: str, routing_key: str, event: dict) -> None:
    connection = await aio_pika.connect_robust(amqp_url)
    async with connection:
        channel = await connection.channel()
        exchange = await channel.declare_exchange(
            exchange_name,
            aio_pika.ExchangeType.TOPIC,
            durable=True,
        )
        message = aio_pika.Message(
            body=json.dumps({
                "payload_version": 1,
                "correlation_id": event.get("correlation_id", ""),
                "published_at": event.get("published_at", ""),
                "routing_key": routing_key,
                **event,
            }).encode(),
            delivery_mode=aio_pika.DeliveryMode.PERSISTENT,  # survive broker restart
            content_type="application/json",
        )
        await exchange.publish(message, routing_key=routing_key)
```

```elixir
# mix.exs deps:

defmodule MyApp.OrderConsumer do
  use Broadway

  def start_link(_opts) do
    Broadway.start_link(__MODULE__,
      name: __MODULE__,
      producer: [
        module: {BroadwayRabbitMQ.Producer,
          queue: "order-processor",
          connection: [
            host: System.get_env("RABBITMQ_HOST", "rabbitmq"),
            port: 5672,
            username: System.get_env("RABBITMQ_USER", "guest"),
            password: System.get_env("RABBITMQ_PASS", "guest"),
          ],
          on_failure: :reject_and_requeue_once,  # dead-letter after one retry
          qos: [prefetch_count: 10],
        },
        concurrency: 1,
      ],
      processors: [
        default: [concurrency: 4]
      ]
    )
  end

  @impl true
  def handle_message(_processor, message, _context) do
    event = Jason.decode!(message.data)
    case event["payload_version"] do
      1 -> process_v1(event)
      v -> Broadway.Message.failed(message, "unsupported payload_version: #{v}")
    end
  end

  defp process_v1(event) do
    # business logic
    :ok
  end
end
```

Also show Python consumer (aio-pika with manual ack):
```python
async def consume(amqp_url: str, queue_name: str) -> None:
    connection = await aio_pika.connect_robust(amqp_url)
    async with connection:
        channel = await connection.channel()
        await channel.set_qos(prefetch_count=10)
        queue = await channel.declare_queue(queue_name, durable=True)
        async with queue.iterator() as queue_iter:
            async for message in queue_iter:
                async with message.process(requeue=False):  # ack on context exit
                    try:
                        event = json.loads(message.body)
                        process_event(event)
                    except Exception as e:
                        # reject without requeue — message goes to DLX
                        raise
```

Dead letter exchanges and queues:
- Declare a dead-letter exchange (DLX) on the main queue declaration:
  arguments={"x-dead-letter-exchange": "dlx", "x-dead-letter-routing-key": "queue-name.failed"}
- Messages that are rejected (basic.reject with requeue=False), exceed TTL, or exceed max-length
  are routed to the DLX and into the DLQ automatically
- Pattern: for each main queue "foo", declare a DLQ "foo.dlq" bound to the DLX
- Include in DLQ payload (added by RabbitMQ automatically): x-death header with original queue,
  exchange, routing-key, reason, and count

Testing expectations:
- Run integration tests against a real RabbitMQ broker (Docker-backed rabbitmq:3.13-management-alpine)
- Prove one publish + consume round-trip with ack
- Prove dead-letter delivery: reject a message and verify it appears in the DLQ
- Prove topic exchange routing: publish with different routing keys and verify correct queue delivery
- Prove prefetch_count: publish N messages and verify at most prefetch_count are in-flight simultaneously
- Do NOT mock the AMQP client in integration tests — the broker behavior is what's tested
- Keep test queues isolated per test run using a unique queue suffix or vhost

Common assistant mistakes:
- Using auto-ack (no_ack=True) in consumer code — auto-ack means ack is sent before the
  message is processed, so crashes cause message loss. Always use manual ack.
- Declaring exchanges/queues without durable=True — they disappear on broker restart
- Not declaring the DLX on the main queue at startup — dead letters silently disappear
- Using a direct exchange when a topic exchange is needed for routing flexibility
- Hardcoding the connection URL in code — use RABBITMQ_URL env var
- Not setting prefetch_count — without it, one slow consumer starves all others by holding all messages
