BACKGROUND: THE FAKER AND SYNTHETIC DATA GENERATION ARC
================================================================================

agent-context-base has canonical examples for API services, CLI tools, pipelines,
observability, schema validation, and terminal tooling. It does not yet have a
canonical area focused on a concern that underlies all of those: generating
realistic synthetic data at scale for development, testing, benchmarking, and
demo environments.

This arc adds canonical-faker examples in 10 languages, each demonstrating:
  - Realistic atomic field generation (names, emails, addresses, phones, UUIDs, dates)
  - Relational graph generation with parent-first ordering and foreign key integrity
  - Deterministic replay via seeded random generators
  - Weighted distributions, edge-case injection, and temporal realism
  - Batch/streaming output (JSONL, CSV, SQL seed export)
  - Profile-based scale control (smoke / small / medium / large)
  - Validation of output realism and relational consistency
  - Docs explaining library tradeoffs and generation design decisions

The canonical domain is a SaaS multi-tenant platform called TenantCore
(defined fully in PROMPT_120). All language examples generate the same domain.

THE CORE INSIGHT THIS ARC MUST PRESERVE:
  Faker libraries generate realistic atomic values. They do NOT generate
  relational systems. The arc must explicitly teach how to build the graph
  layer on top of faker — parent-first ordering, ID pools, lookup tables,
  cardinality rules, temporal consistency, and cross-field correlation.
  This design principle is the primary doctrine this arc enforces.

PART 1: CREATE context/doctrine/synthetic-data-realism.md
================================================================================

Create: context/doctrine/synthetic-data-realism.md

This doctrine governs how canonical faker examples in this repo must be designed.
It must be explicit enough that an assistant following only this doctrine and a
stack file can implement a correct example without prior context.

Required sections:

  # Synthetic Data Realism Doctrine

  ## The Core Separation

  Explain in 3–4 sentences: faker libraries are atomic value generators, not
  relational system builders. An assistant must never conflate "call Faker.name()"
  with "generate a realistic user table." Producing a realistic dataset requires
  a deliberate generation pipeline on top of faker, not just more faker calls.

  ## Seven Mandatory Generation Properties

  Number each rule clearly. Every canonical faker example in this repo must satisfy
  all seven:

  Rule 1 — DETERMINISTIC SEED
    Every generator must accept an explicit seed and produce identical output
    for the same seed on every run on every machine. Document the seed. Test it.
    Do not use random seeds in smoke tests or CI.

  Rule 2 — PARENT-FIRST ORDERING
    Entities must be generated in dependency order: parents before children.
    No child record may reference an ID that does not yet exist in a generated
    pool. Document the generation order in each example's README.

  Rule 3 — STABLE ID POOLS
    Before generating children, build and hold the full parent ID pool in memory
    (or in a lookup file for large datasets). Children sample from this pool.
    Do not generate IDs on demand during child generation — this breaks referential
    integrity under sampling and partial runs.

  Rule 4 — WEIGHTED DISTRIBUTIONS, NOT UNIFORM RANDOMNESS
    Real datasets are not uniformly distributed. Organizations have skewed user
    counts (some have 1, a few have hundreds). Events cluster around active users.
    Every canonical example must demonstrate at least one non-uniform distribution.
    Document the distribution choices and why they are realistic.

  Rule 5 — TEMPORAL REALISM
    Events happen after the entities they reference were created. Memberships
    join after org creation. Audit events happen during a member's tenure.
    Every example must validate that no event timestamp precedes its parent's
    creation timestamp.

  Rule 6 — CROSS-FIELD CONSISTENCY
    Fields that constrain each other must be generated together. An audit event's
    user_id must be a member of the audit event's org_id. A project's created_by
    must be a member of the project's org. Generate these relationships explicitly —
    do not rely on faker's value generators to produce consistent cross-field values.

  Rule 7 — VALIDATION BEFORE OUTPUT
    Every example must run a validation pass before writing output. Validation
    must check: FK integrity (every foreign key references a generated parent),
    uniqueness (emails, slugs, API keys are unique), row count sanity,
    and seed reproducibility (two runs with the same seed produce identical output).
    The validation pass must produce a human-readable summary report.

  ## What Faker Libraries Do Well

  A short list (4–6 items): realistic names, emails, addresses, phone numbers,
  lorem text, dates within a range, UUIDs, product names, company names, URLs,
  IP addresses, credit card numbers (test only), locale-aware text.

  ## What Faker Libraries Do NOT Do

  A short list (4–6 items): relational integrity, cardinality constraints,
  temporal ordering, weighted distributions across entity graphs, cross-field
  consistency, domain-specific business logic validation.

  ## Output Profiles

  Define four standard profiles. Every canonical example must implement these or
  document why a profile is not applicable:

  | Profile | Organizations | Users  | Projects | Events   | Purpose                          |
  |---------|---------------|--------|----------|----------|----------------------------------|
  | smoke   | 3             | 10     | 9        | ~50      | CI gate, runs in <5s, deterministic |
  | small   | 20            | 200    | 100      | ~2,000   | local dev data load              |
  | medium  | 200           | 5,000  | 2,000    | ~50,000  | realistic demo environment       |
  | large   | 2,000         | 50,000 | 20,000   | ~500,000 | benchmark or showcase run        |

  Profiles are additive approximations. The actual row counts depend on
  cardinality rules. Document the actual counts in the validation report.

  ## Commit Rule

  Do not commit a faker canonical example until its smoke profile produces
  a passing validation report and smoke tests pass in the language's toolchain.

Target length: 80–100 lines. Voice: direct, imperative, matches existing doctrine tone.

  # Synthetic Data Generator

  ## When to Use This Archetype

  2–3 sentences: when the primary work is generating realistic seed data for
  development, testing, benchmarking, or demo environments. Not for generating
  one-off test fixtures — this archetype is for structured, relational, scalable,
  reproducible dataset generation.

  ## Required Context
  (The files an assistant must load before implementing in this archetype)
  - context/doctrine/synthetic-data-realism.md
  - context/stacks/faker-{language}.yaml for the target language
  - examples/canonical-faker/domain/schema.md
  - examples/canonical-faker/domain/generation-order.md

  ## Common Goals
  - Parent-first relational generation with stable ID pools
  - Deterministic seeded replay
  - Profile-based scale control (smoke / small / medium / large)
  - JSONL/CSV/SQL output with validation reports
  - Smoke tests that are fast and CI-safe

  ## Anti-Patterns
  - Calling faker methods inline without a generation pipeline
  - Generating children before parents
  - Using language-native random without a fixed seed
  - Treating faker output as validated output without a verification pass
  - Generating everything in memory for large profiles (use chunked writes)
  - Conflating "realistic-looking values" with "relational integrity"

  ## Typical Verification Gate
  - smoke profile runs in < 5 seconds
  - validation report shows 0 FK violations
  - two identical-seed runs produce byte-identical JSONL output
  - row counts match profile spec within ±10%

  ## Related Artifacts

  - context/skills/faker-library-selection.md
  - context/skills/synthetic-dataset-design.md

Create: context/skills/synthetic-dataset-design.md

This skill guides the design of a relational synthetic dataset generation pipeline.
It teaches the seven-step pattern that every canonical example in this arc follows.

Required sections:

  # Synthetic Dataset Design Skill

  ## When to Use
  When asked to generate seed data, fixtures, synthetic datasets, or realistic
  test data that spans multiple related entities.

  ## The Seven-Step Generation Pipeline

  Step 1: Define the entity graph and generation order
    Draw the dependency graph. Write it down in the example README before coding.
    Identify which entities are roots (no parents) and which are leaves (deepest children).

  Step 2: Choose a seed and make it explicit
    Pick a seed (e.g., 42) and use it consistently throughout. Document it.
    Every random call in the pipeline must flow from the same seeded RNG.

  Step 3: Generate roots first, build ID pools
    Generate all root entities. Store their IDs in a lookup structure.
    Do not proceed until roots are complete and ID pool is built.

  Step 4: Generate each dependent layer in order
    For each layer: sample parent IDs from the parent pool, generate children,
    apply cardinality rules, apply weighted distributions.

  Step 5: Apply cross-field consistency rules
    After all entities are generated, run consistency rules:
    - membership.user_id must be in users pool
    - event.user_id must be a member of event.org_id
    - event.project_id must belong to event.org_id

  Step 6: Validate
    Run the validation pass. Check FK integrity, uniqueness, row counts,
    temporal ordering, distribution sanity. Produce a validation report.
    Do not write output until validation passes.

  Step 7: Write output
    Write to JSONL/CSV/SQL in the requested profile size.
    Use chunked writes for medium/large profiles.
    Report progress and final row counts.

  ## Common Design Mistakes

  List 5 common mistakes with one-line explanations each.

Target length for each skill: 40–60 lines.
