================================================================================
BACKGROUND: WHY SKILLS WERE DEFERRED
================================================================================

PROMPT_100 intentionally scoped the foundation to: doctrine, archetypes, stacks,
routing, manifests, and workflows. Skills were explicitly left out to keep that
prompt focused on the structural layer.

Skills are a different kind of artifact — they encode "how to decide" rather than
"what to build." They are short procedural guides (30–90 lines each) that the
assistant applies during task inference, routing, and verification selection.

Without terminal-aware skills, two decision gaps remain:

  Gap 1 — Example selection:
    canonical-example-selection.md does not distinguish CLI, TUI, and dual-mode
    surfaces. An assistant handling a Go TUI request might select a Python CLI
    example, or load examples/canonical-cli/ instead of examples/canonical-terminal/.

  Gap 2 — Validation path selection:
    verification-path-selection.md has no entry for PTY-based or marker-based
    terminal assertions. Its decision tree covers storage writes, UI state (HTMX),
    pure logic, and smoke tests — nothing terminal-specific. An assistant could
    apply a plain smoke-test verdict to a TUI change that requires PTY validation.

Both gaps are closed by this prompt.

This skill guides selection of the right terminal canonical example when adding,
extending, or referencing one. Read the existing skill files first to calibrate

Required sections and content direction (write actual content, not placeholders):

  # Terminal Example Selection

  (one-sentence description: what this skill is for)

  ## Procedure

  Numbered steps covering:
    1. Identify the terminal tier from task signals:
         - Tier 1 CLI: subcommands, flags, --output json, scriptable, headless
         - Tier 2 TUI: full-screen keyboard-driven UI, rich rendering, interactive
         - Tier 3 dual-mode: shared core, both CLI and TUI surfaces via --interactive
    2. Identify language and flagship/secondary split:
         Flagship (prefer for new tasks):
           python-typer-textual, rust-clap-ratatui, go-cobra-bubbletea,
           typescript-commander-ink, java-picocli-lanterna, ruby-thor-tty,
           elixir-optimus-ratatouille
         Secondary (use only when the stack explicitly matches the secondary library):
           python-click-blessed, rust-argh-tui-realm, go-urfave-tview,
           typescript-yargs-blessed, java-jcommander-jline, ruby-clamp-tty,
           elixir-optionparser-owl
    3. Load DECISION_GUIDE.md for language-vs-tier guidance
    4. Confirm the example exists in CATALOG.md before recommending it
    5. If no exact-language match exists for the requested tier:
         state the gap explicitly; do not silently substitute a different language
    6. Prefer implemented examples over stubs; if using a stub, flag its status

  ## Priority

  Ranking: exact language + exact tier (flagship first), then exact language +
  adjacent tier (note the difference), then same language family with gap noted.

  ## Good Triggers

  Phrases that indicate this skill applies — e.g., "add a terminal example for Go",
  "which Rust TUI example should I follow", "what is the canonical reference for
  Elixir CLIs", "how do I validate a dual-mode operator console".

  ## Avoid

  Specific anti-patterns: selecting canonical-cli/ for a terminal tooling task,
  mixing tier levels silently, using a web/API example for a terminal task,
  presenting stubs as canonical references.

  ## Reference Files

  - examples/canonical-terminal/CATALOG.md
  - examples/canonical-terminal/DECISION_GUIDE.md
  - context/archetypes/terminal-tui.md
  - context/archetypes/terminal-dual-mode.md
  - context/archetypes/cli-tool.md

Target length: 60–80 lines. Match the voice of canonical-example-selection.md.

PART 2: CREATE context/skills/terminal-validation-path-selection.md
================================================================================

Create: context/skills/terminal-validation-path-selection.md

This skill guides selection of the right validation approach for a terminal tooling
change. Read verification-path-selection.md first to match its format precisely.

Required sections and content direction:

  # Terminal Validation Path Selection

  (one-sentence description)

  ## Procedure

  Numbered steps:
    1. Identify the change surface:
         - CLI output, flags, subcommands  → non-interactive smoke test + marker assertion
         - TUI rendering, keyboard flow    → PTY harness
         - JSON output structure           → smoke test + --output json assertion
         - cross-language consistency      → comparison harness
         - fixture corpus schema or count  → fixture validation tests
    2. Apply the decision tree (one entry per surface type):
         - CLI smoke test: run in non-interactive mode, fixture-backed, assert at least
           one ## BEGIN_JOBS ## / ## END_JOBS ## marker or --output json field; must
           pass without a TTY
         - marker assertion: assert markers are present and entry count matches expectation
         - JSON output assertion: parse with json.loads(); assert top-level fields and count;
           do not diff raw text
         - PTY harness: use verification/terminal/pty_harness.py; script keystrokes;
           assert on normalized (ANSI-stripped) transcript; run after smoke test, not instead
         - comparison harness: run scripts/run_terminal_comparison.py; assert count
           and field matches across all implemented (non-stub) examples
    3. Escalation rule: a TUI change requires BOTH a non-interactive smoke test AND
       PTY validation; one is not a substitute for the other
    4. A fixture corpus change also requires fixture validation tests
    5. Verification is not complete until the test was actually run

  ## Good Triggers

  Phrases indicating this skill applies — e.g., "what test for this terminal change",
  "do I need a PTY test or a smoke test", "how do I validate TUI rendering",
  "is a smoke test enough for a new --output json flag".

  ## Avoid

  Anti-patterns: skipping smoke test for TUI changes, asserting raw ANSI output,
  treating JSON generation as equivalent to asserting JSON schema, running comparison
  harness on stubs and reporting results as failures, using PTY for CLI-only changes.

  ## Reference Files

  - docs/terminal-validation-contract.md
  - verification/terminal/pty_harness.py
  - verification/terminal/harness.py
  - scripts/run_terminal_comparison.py
  - context/doctrine/terminal-ux-first-class.md  (Rules 4, 5, 6)

Target length: 65–90 lines. Match the voice of verification-path-selection.md.
