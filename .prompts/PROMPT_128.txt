BACKGROUND: DOGFOODING THE FAKER ARC
================================================================================

The faker arc (PROMPT_119–126) created a canonical domain called TenantCore with
7 entities and a Python reference implementation for generating relational
synthetic datasets (examples/canonical-faker/domain/generation_patterns.py).

This arc's canonical analytics examples will represent an "Ops & Product Analytics
Workbench" — a server-rendered analytics tool for exploring operational metrics
derived from that same domain. The data shapes naturally arise from TenantCore:

  - Events (counts, rates, volume over time)
  - Sessions (durations, funnel stages, conversions)
  - Services/teams (category comparisons, error rates)
  - Deployments/releases (time-series, rollout patterns)
  - Incidents (severity distributions, MTTR analysis)
  - Latency samples (histogram, P50/P95/P99 distributions)
  - Funnel stages (drop-off rates, conversion analysis)

Instead of inventing a new seed system, the analytics examples will import or
call the faker arc's generation pipeline to produce seeded test data. Every
implementation (Python, Go, Rust, Elixir) reads from the same JSON fixture
files that the faker arc's Python reference implementation produces.

This is the dogfooding principle from AGENT.md in action.

  # Ops & Product Analytics Workbench — Domain Spec

  ## Purpose
  3–4 sentences. What this canonical example teaches. Why the domain was chosen
  (it arises directly from the TenantCore faker arc, enabling dogfooding).
  What a developer should understand after working through all four implementations.

  ## Data Domain

  The analytics domain is derived from TenantCore (examples/canonical-faker/).
  The following entities drive the analytics:

  | Entity          | Source TenantCore Entity | Key Analytics Shape          |
  |-----------------|--------------------------|------------------------------|
  | events          | Event                    | time series, count rates     |
  | sessions        | Session                  | funnel, duration histogram   |
  | services        | Service / Team           | category comparison          |
  | deployments     | Deployment               | time series, rollout pattern |
  | incidents       | Incident                 | severity distribution, MTTR  |
  | latency_samples | Latency / Metric         | histogram, P95/P99 box plot  |
  | funnel_stages   | FunnelStage              | drop-off funnel              |

  For exact field definitions, see:
    - examples/canonical-faker/domain/schema.md
    - examples/canonical-analytics/domain/fixtures/smoke.json (generated)

  ## Seed Data Contract
  3–4 sentences. Fixtures are generated by generate_fixtures.py, which
  dogfoods the faker arc. All four implementations read from the same fixture
  files. Seed=42 guarantees deterministic output across all runs and machines.
  Reference examples/canonical-analytics/domain/seed-data-contract.md for rules.

  ## The Workbench Layout

  The workbench is a single web application with these top-level views:

    /                    — Overview: event volume trend + service error rate bar
    /trends              — Time Series view: event count line chart with date filter
    /services            — Service Comparison: horizontal sorted bar, drilldown panel
    /distributions       — Distribution view: latency histogram + box plot pair
    /incidents           — Incident Analysis: severity bar + MTTR summary card
    /funnel              — Funnel view: session funnel chart + conversion summary
    /heatmap             — Activity Heatmap: hour-of-day × day-of-week event density

  HTMX fragment endpoints (all views):
    /fragments/chart     — returns only the chart panel for the current view
    /fragments/summary   — returns only the summary card
    /fragments/details   — returns a drilldown detail panel (where applicable)

  Filter state for all views:
    date_from, date_to   — ISO date strings
    services             — multi-value (one or more service names)
    severity             — multi-value (incident severity levels)
    environment          — multi-value (production/staging/development)

  ## Required Chart Families

  Each implementation must include all six chart families:

    Family 1 — Time Series (line chart)
      View: /trends
      Chart: event count over time, one line per environment
      HTMX trigger: date_from/date_to filter form submission
      Summary card: total events, date range, active filters

    Family 2 — Category Comparison (horizontal sorted bar)
      View: /services
      Chart: error rate per service, sorted descending
      HTMX trigger: environment multi-select
      Detail panel: top-N incidents for the selected service (drilldown)

    Family 3 — Distribution (histogram + box plot pair)
      View: /distributions
      Chart A: latency histogram (P50 annotated)
      Chart B: box plot by service (P95, P99 whiskers)
      HTMX trigger: service multi-select
      Summary card: P50, P95, P99 values

    Family 4 — Heatmap
      View: /heatmap
      Chart: event density by hour-of-day × day-of-week
      HTMX trigger: date range + environment
      Summary card: peak hour, peak day

    Family 5 — Funnel
      View: /funnel
      Chart: session funnel stages with drop-off percentages
      HTMX trigger: environment multi-select
      Summary card: overall conversion rate, biggest drop-off stage

    Family 6 — Incident Distribution (bar + summary)
      View: /incidents
      Chart: incident count by severity level (bar)
      HTMX trigger: date range + service + environment
      Summary card: MTTR (mean time to resolve) by severity

  ## Parity Contract

  All four implementations (Python, Go, Rust, Elixir) must:
    - implement all six chart families
    - read from the same fixture files (smoke.json and small.json minimum)
    - apply the same filter state semantics
    - expose the same fragment endpoints (/fragments/chart, /fragments/summary)
    - pass the verification contract (see verification-contract.md)
    - expose a /health endpoint that returns 200 and {"status": "ok"}

  ## What Each Implementation Does NOT Need to Match
    - UI pixel-level layout (Tailwind class choices may differ)
    - exact hover text wording (hovertemplate content may vary)
    - database backend (in-memory or file-backed storage is acceptable)
    - port number (each implementation uses its own default port)

  Each implementation must pass all items in this contract before it is marked
  [x] in the parity matrix and CATALOG.md.

  Every route listed in spec.md must return HTTP 200 with no server error.

  Minimum routes to smoke-test:
    GET /                      → 200, HTML response
    GET /trends                → 200, HTML response
    GET /services              → 200, HTML response
    GET /distributions         → 200, HTML response
    GET /heatmap               → 200, HTML response
    GET /funnel                → 200, HTML response
    GET /incidents             → 200, HTML response
    GET /health                → 200, {"status": "ok"} JSON
    GET /fragments/chart?view=trends → 200, HTML fragment containing plotly JSON
    GET /fragments/summary?view=trends → 200, HTML fragment with text content

  ## Figure Builder Unit Tests (required; no server needed)

  For each of the six chart families, there must be a unit test covering:
    1. valid aggregate input → assert trace count ≥ 1 and axis titles non-empty
    2. empty aggregate input → assert a valid figure is returned (not an error)
    3. single-item aggregate input → assert valid output shape

  ## Aggregation Unit Tests (required; no server or DB needed)

  For each chart family's aggregation function:
    1. smoke fixture input → assert output matches expected shape and row count
    2. filtered input (e.g., one service, one environment) → assert filtered output

  ## Filter State Alignment Tests (required)

  For each chart family:
    - Apply a filter (e.g., services=["service-A"]) and assert that both the
      visible results and the chart data reflect the same filtered set.
    - Assert: chart trace labels or x-axis values do not include filtered-out values.

  ## Fixture Determinism Test (required)

  Run generate_fixtures.py twice. Assert the output files are byte-identical.

  ## Empty State Test (required)

  Pass an empty or zero-result filter state to each chart endpoint.
  Assert the response is HTTP 200 and the figure contains an appropriate empty-state
  trace or annotation (not a 500 error).

  ## Parity Alignment Check (required before marking complete)

  Run the arc's parity check runner (PROMPT_133 will create it):
    python3 verification/analytics/run_parity_check.py --stack <python|go|rust|elixir>

  # Page Layout Specification

  ## Overview
  Each view is a server-rendered HTML page. The layout is consistent across
  all four implementations. Tailwind provides the utility classes.

  ## Chrome (consistent across all views)
  - Top navigation bar: links to all 7 views, active view highlighted
  - Page title: matches view name from spec.md
  - Filter panel: left sidebar or collapsible top strip (implementation choice)
    - Multi-select inputs for: services, environment, severity (where applicable)
    - Date range inputs (date_from, date_to)
    - Submit button with hx-get and hx-target pointing at the main content panel
  - Main content panel: chart panel + summary card (HTMX target)
  - Footer: seed info (fixture profile name, seed value used, record counts)

  ## Filter Panel Requirements
  - Multi-select filter controls must use native <select multiple> OR a
    Tailwind-styled checkbox list. Either is acceptable.
  - Follow context/doctrine/filter-panel-rendering-rules.md if implementing
    include/exclude semantics. For the analytics examples, include-only is
    sufficient (full include/exclude is optional enhancement).
  - Active filter state must be visually indicated (Tailwind: ring, bg, or badge).
  - Submit must use hx-get (not POST) so filter state is URL-addressable.

  ## Chart Panel + Summary Card (HTMX fragment target)
  The main panel contains:
    - Chart div: a <div id="chart-panel"> populated by Plotly.js from JSON
    - Summary card: key metrics (total count, date range, active filters)
    - For views with drilldown: a <div id="detail-panel"> populated by a
      separate HTMX request when user selects a chart element or table row

  ## Detail Panel (drilldown, optional per view)
  - Rendered as a side panel or modal overlay
  - Fragment endpoint: /fragments/details?view=<view>&item=<item>
  - Returns: an HTML partial with detail metrics for the selected item

  ## Accessibility Minimum
  - All filter inputs must have a visible or sr-only <label>
  - Chart divs must have an aria-label describing the chart purpose
  - Summary card text must be readable without interpreting the chart
