SECTION 2 ‚Äî What each companion type must cover

There are three variant types. Each requires different content.

--- api-endpoint companions ---

Each `*-api-endpoint-example.md` must cover:

1. **Route registration shape** ‚Äî what the route registration looks like in this
   framework and why (is it a handler struct, a router builder, a macro, a function?).

2. **Request parsing approach** ‚Äî how path params and query params are extracted
   (typed deserialization, manual extraction, binding). Note whether the framework
   validates or coerces automatically, or whether that responsibility falls to the handler.

3. **Service/transport separation** ‚Äî describe what the example shows for service
   separation. Is there a service struct/class/protocol? Is the DB query inline?
   Is this a deliberate simplification for the example or is this how the framework
   is typically structured? Be explicit.

4. **Error handling posture** ‚Äî what does the example show? If the example has no
   error handling, say "error handling is not shown; add `404`/`502` returns when
   deriving." If there is a placeholder comment (as in Rust), quote it and explain
   what to replace it with. Do not pretend error handling is present if it is not.

5. **Response shape** ‚Äî what is returned and how is it encoded? Typed JSON, a lens,
   a view struct, manual serialization?

--- html-fragment companions ---

Each `*-html-fragment-example.md` must cover:

1. **HTMX fragment contract** ‚Äî the `hx-swap-oob="true"` attribute, the stable
   element `id`, and why the `id` must match the target in the surrounding page.
   Every language should show this; if it is missing in a language's example, say so.

2. **Content-type discipline** ‚Äî how `text/html; charset=utf-8` is set in this
   framework. Some set it on the response object, some via a helper, some
   implicitly through the view layer.

3. **Server-side rendering approach** ‚Äî what mechanism is used to generate the
   HTML? Multiline string interpolation? A dedicated HTML builder (Hiccup,
   TyXML)? A template? Name the mechanism and note whether it escapes by default.

4. **Escaping and XSS posture** ‚Äî is the value interpolated safely (escaped by
   the HTML library) or raw? If raw string interpolation is used, flag it.

--- data-endpoint companions ---

Each `*-data-endpoint-example.md` must cover:

1. **Chart payload contract** ‚Äî the `{metric, series: [{name, points: [{x, y}]}]}`
   shape. Note whether it is enforced by a typed struct, derived from a builder
   function, or assembled manually.

2. **Typed response encoding** ‚Äî how is the JSON produced? Derive codecs, JSON
   lenses, manual struct-to-JSON, `%*{}` macro, `std.fmt.bufPrint`? Note the
   tradeoff each approach creates for schema evolution.

3. **Storage query integration** ‚Äî if the example queries a database, describe
   the query boundary (Exposed DSL, Avram query class, Caqti request, raw SQL,
   Sequel). If the example uses stub data, say "the DB query is stubbed; replace
   with a real query boundary for production."

4. **Metric parameter safety** ‚Äî is the `metric` path param validated before use?
   If it is passed directly to a SQL query or embedded in a response without
   sanitization, call that out explicitly.


SECTION 3 ‚Äî Per-language specific content requirements

Each language has specific design choices that the companion must name. This section
gives the key points for each language. You must read the actual code to confirm
these before writing the companion; if the code has changed from what is listed
here, trust the code.

**Go (Echo)**
- `api-endpoint`: Handler struct with `Register()` and `ListRecent()` methods.
  Echo error handling via `echo.NewHTTPError`. `strconv.Atoi` for limit with
  integer fallback. Delegates to `services.ReportService`. Returns an HTML view
  via `views.ReportSummaryList`.
- `html-fragment`: Returning HTML content from Go ‚Äî note the `echo.MIMETextHTMLCharsetUTF8`
  constant usage.
- `data-endpoint`: If the Go data endpoint is not in the canonical-api directory,
  note the absence and reference the closest available example.

**Rust (Axum)**
- `api-endpoint`: Extractor-based handler ‚Äî `State`, `Path`, `Query` as function
  arguments. `#[derive(Deserialize, Serialize)]` for request/response types.
  `.unwrap_or(20).min(100)` for limit clamping. The `expect()` on the service call
  is a deliberate placeholder ‚Äî quote it and explain what the real error return type
  should be.
- `html-fragment`: Note how Axum handles HTML responses (IntoResponse impl or
  Html wrapper).
- `data-endpoint`: Note the `ReportService` stub and `Json(reports)` return pattern.

**TypeScript (Hono)**
- `api-endpoint`: Hono router, inline Drizzle query directly in handler (no service
  layer). Note this is the TypeScript pattern ‚Äî Hono handlers with ORM access inline
  is idiomatic for small Bun services. `Math.min(limit, 50)` for clamping.
- `html-fragment`: `context.html()` with JSX. Note that Hono's JSX is compile-time,
  not runtime.
- `data-endpoint`: Chart data with inline construction. No storage shown.

**Elixir (Phoenix)**
- `api-endpoint`: Two files ‚Äî controller and router. The controller delegates to an
  `Example.Reporting` context module. This is the canonical Phoenix pattern:
  controllers do not contain business logic; they call context functions. Note the
  `String.to_integer() |> min(100)` limit clamping.
- `html-fragment`: Phoenix renders via templates. Note the `render(conn, :index, ...)`
  pattern and that HTML escaping is automatic via HEEx.
- `data-endpoint`: Note if a data endpoint exists or if Phoenix only has the two
  source files. If the data endpoint is absent, say so directly.

**Scala (Tapir + http4s + ZIO)**
- `api-endpoint`: Tapir endpoint definition is separate from the logic. Types
  flow through the endpoint description. `serverLogicSuccess` is used ‚Äî this means
  the handler is expected to always succeed (no error path in the example).
  Note that real endpoints should use `serverLogic` for error cases.
- `html-fragment`: `Http4sDsl` pattern matching for routing, manual content-type
  setting. String interpolation for HTML.
- `data-endpoint`: `JsonCodec` derived via ZIO JSON macros. The endpoint contract
  (`chartEndpoint`) is defined separately from the server logic.

**Kotlin (http4k + Exposed)**
- `api-endpoint`: `Body.auto<ReportEnvelope>().toLens()` for typed JSON response.
  Exposed `transaction { }` block inline in the route handler. The fallback stub
  when no row is found means this returns 200 with stub data instead of 404 ‚Äî
  call this out explicitly as a simplification.
- `html-fragment`: Template string with `trimIndent()`. Note that Kotlin string
  templates do not escape HTML ‚Äî XSS risk if `tenantId` came from user input.
- `data-endpoint`: Describe the lens pattern and JSON response.

**Clojure (Kit + next.jdbc + Hiccup)**
- `api-endpoint`: Route defined as a data structure (Ring router map). Handler is
  a plain function. `jdbc/execute!` inline in the handler. Explain that in Kit
  the inline query is appropriate for small routes but larger services extract a
  repository namespace.
- `html-fragment`: Hiccup is a Clojure DSL for HTML ‚Äî data structures become HTML.
  `h/html` produces escaped HTML by default. Note this.
- `data-endpoint`: JSON assembled with `clojure.data.json/write-str`. The data
  structure mirrors the chart payload contract.

**OCaml (Dream + Caqti + TyXML)**
- `api-endpoint`: Caqti's typed SQL layer ‚Äî `Caqti_request.Infix.(->*)` for a
  query that returns a list. `Db.collect_list` runs the query and `>>=
  Caqti_lwt.or_fail` raises on DB error ‚Äî note this means a DB failure is a
  fatal unhandled exception in the example, not a 500 response.
- `html-fragment`: TyXML for type-safe HTML generation. Note that TyXML validates
  the HTML structure at compile time.
- `data-endpoint`: Manual `Yojson` assembly. Note the verbosity compared to
  languages with derive macros ‚Äî this is intentional in OCaml's value-layer JSON
  approach.

**Crystal (Kemal + Avram)**
- `api-endpoint`: Avram model + query class separation (model describes the table,
  query class provides the query DSL). This is Crystal's ORM idiom ‚Äî name it.
- `html-fragment`: ECR template or string interpolation. Note whether HTML is
  escaped.
- `data-endpoint`: Struct with `JSON::Serializable` ‚Äî Crystal's compile-time JSON
  codec via mixin. The fallback to hardcoded data when the query is empty is a
  deliberate in-example simplification.

**Nim (Jester)**
- `api-endpoint`: Route macro `routes:` / `get "/path": resp`. Handler is
  effectively inline. No service separation in Nim Jester examples.
- `html-fragment`: Karax or string interpolation ‚Äî note what the example actually uses.
- `data-endpoint`: `%*{}` JSON literal macro for typed JSON construction.
  `proc buildSeriesPayload` separates the payload logic from the route.

**Dart (Dart Frog)**
- `api-endpoint`: Dart Frog route is a file-per-route pattern ‚Äî `onRequest()` is
  the handler function. No router configuration shown; routing is by file system.
- `html-fragment`: Note what HTML generation Dart uses (string interpolation or
  package).
- `data-endpoint`: `Map<String, Object>` for JSON payload. `Response.json()` for
  the response. Note that Dart Frog auto-serializes Maps to JSON.

**Ruby (Hanami)**
- `api-endpoint`: `Hanami::Action` for request handling, `Hanami::Router` for
  routing. `MetricsRepo` provides the storage boundary. Sequel for DB access.
- `html-fragment`: Haml, ERB, or Phlex ‚Äî note what the example uses.
- `data-endpoint`: `JSON.generate()` with a plain Ruby hash. `MetricsRepo.series_for`
  hides the Sequel query.
