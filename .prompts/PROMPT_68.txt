PHASE 3 — REPAIR RULES

Apply fixes following these rules precisely.

RULE: One canonical location per concept.
  If a doc should be reached via multiple phrases, those phrases all go into task-router.md
  pointing to the same target. Do not create a separate alias-catalog entry that duplicates
  a task-router entry — only add aliases for concepts that are NOT already well-covered
  by task-router entries or that benefit from short named references.

RULE: Alias strings should be short and human-natural.
  Aliases are what an operator types when they know what they want but don't know the internal
  path. Examples: "broker selection", "polars vs pandas", "dokku deployment", "nats jetstream".
  They are not full task descriptions.

RULE: Stack-router.md entries are for repo-signal-based inference.
  stack-router.md maps file signals (lockfiles, source patterns, imports) to stack docs.
  It is not a synonym for task-router.md. Only add stack-router entries when a new file
  signal or repo artifact is not yet covered.

RULE: Archetype-router.md maps project shape descriptions to archetypes.
  Each entry should describe the kind of project (in a few words) and the doc to load.
  If new archetypes exist without entries, add them. Do not change existing archetype descriptions.

RULE: repo-signal-hints.json maps file/directory patterns to context hints.
  Check whether any new stacks, patterns, or infra packs added since the last update
  (e.g., duckdb-parquet, python-ml-library-selection, backend-driven-ui, multi-backend trios)
  have corresponding signal hints. Add minimal signal hints for any that don't.

RULE: Minimal diff. Do not reformat, reorder, or restructure router sections that are still
  accurate. Insert new entries near semantically related existing entries.

RULE: Do not add coverage for content that is genuinely internal-only or assistant-boot-only.
  Some docs (e.g., context/anchors/, context/memory/) are not meant to be reached via
  task routing — they load automatically or via specific boot steps. Do not add task-router
  entries for these unless they are genuinely missing and would benefit from explicit routing.
