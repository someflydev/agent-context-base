# Plotly + HTMX + Tailwind Server-Rendered Visualization Arc

## Purpose
This arc solves the challenge of building complex, testable analytics applications without resorting to heavy single-page application (SPA) architectures. By treating Plotly figures as declarative models built on the server, we decouple data aggregation, visualization logic, and HTTP handling. The combination of HTMX for seamless fragment swaps, Tailwind for strict styling contracts, and server-rendered templates provides a highly maintainable, testable, and robust architecture. Developers working through these examples will understand how to construct declarative visualization pipelines and interactive endpoints across multiple backends.

**Status:** Complete (PROMPT_127–133). All four stacks are implemented. See `examples/canonical-analytics/CATALOG.md` for per-stack status.

## The Core Insight
Plotly figures are structured data objects, not merely visual rendering commands. Server-side construction decouples aggregation logic, figure-builder logic, and HTTP handling — each layer becomes independently testable without involving a browser. HTMX enables fragment-based interactivity without a frontend framework, letting the server own the source of truth. Tailwind provides the strict filter UI state contract. These four choices are mutually reinforcing, forming a cohesive architecture rather than an incidental collection of tools.

## The Three-Layer Model
The architecture implements the three-layer model defined in `context/skills/plotly-figure-builder-design.md`. The layers separate Aggregation (fetching and grouping data), Figure Construction (building the Plotly schema object), and Serialization/Routing. This ensures that the charting logic remains pure and easily testable.

The domain relies on TenantCore, a fictional SaaS multi-tenant platform used exclusively for canonical data generation. Dogfooding the faker arc guarantees robust, relational test data. The domain models 7 key entities:

- **sessions**: Funnels, duration histograms.
- **services**: Category comparisons.
- **deployments**: Time series, rollout patterns.
- **incidents**: Severity distributions, MTTR.

## Key Distinctions to Preserve

- Figure builders return structured data objects, not JSON strings. The handler serializes. This is the most important architectural invariant.
- Aggregation and figure construction are separate layers. A route handler that contains aggregation SQL or filter logic is a violation.
- HTMX fragment boundaries are at the panel, not the page. Over-wide swaps mask filter-state bugs.
- Tailwind class contracts for filter UI are defined in `filter-panel-rendering-rules.md` — they are part of the backend-to-frontend contract, not cosmetic choices.
- The Go implementation uses plain Go structs + `encoding/json` for figures. No external Plotly Go library is needed or used.
- The Rust implementation uses `plotly.rs` and returns `serde_json::Value` from figure builders (documented deviation from the three-layer model). The strict three-layer model requires figure builders to return a language-native struct that the handler serializes to JSON; the `plotly.rs` API returns `plotly::Plot` objects that serialize internally, so the figure builder layer returns `serde_json::Value` directly rather than a typed intermediate struct. The separation of aggregation from figure construction is preserved; only the serialization boundary differs.
- The Elixir implementation uses Phoenix controllers, NOT LiveView. HTMX + HEEx is the pattern here. LiveView is a valid but different approach.
- Fixture files are generated by dogfooding the faker arc's Python reference implementation. Do not build a parallel seed system.
